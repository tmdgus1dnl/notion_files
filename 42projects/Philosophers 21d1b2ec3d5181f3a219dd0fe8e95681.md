# Philosophers

설명: 멀티 스레드, 프로세스
상태: 완료

철학자들이 굶어 죽지 않도록 동기화 시켜 살리는 과제이다.

# 과제 소개

원탁에 1명 이상의 철학자들이 앉아 있다.

중앙엔 스파게티가 담긴 큰 그릇이 있다.

철학자들의 행동은 “잠자기”, “먹기”, “생각하기” 이렇게 3 가지가 있다.

한 가지의 행동을 할때 다른 행동은 할 수 없다.

예를 들어 먹으면서 생각할 수 없고,

생각하면서 잘 수 없다.

원탁에는 포크가 놓여져 있는데, 철학자들의 수 만큼 놓여져 있다.

철학자가 스파게티를 먹으려면 한 손으로만 포크를 집고 먹기엔 불편하기에,

양 손에 포크를 집고 먹어야 한다.

철학자가 먹는 행동을 끝내면,

양 손에 집었던 포크를 내려놓고 잠을 자기 시작한다.

잠에서 깨면 생각하기 시작한다.

이 모든 시뮬레이션은 한명의 철학자라도 굶어 죽을 시 끝이 난다.

따라서 모든 철학자는 굶어 죽지 않기 위해 먹어야 한다.

철학자들은 서로 말하지 않고, 누가 죽을 위기에 처했는지 모른다.

# 사전 지식 모음

프로세스 (Process) : 작업이 실행되는 실행 단위이다.

프로세스는 프로그램이 실행될 때 만들어 지고,

운영 체제가 프로세스를 메모리에 할당하여 프로세스가 실행된다.

따라서 독립된 주소 공간을 가지고 있으며, 다른 프로세스와는 별도의 메모리 공간을 사용한다.

따라서 프로세스 간의 직접 접근은 불가하며, IPC라는 매커니즘을 통해 통신이 가능하다.

프로세스 간의 문맥 전환이 발생하면 비용이 상대적으로 높다.

쓰레드 (Thread) : 프로세스 내에서 실행되는 작은 실행 단위.

같은 프로세스 내의 쓰레드들은 해당 프로세스의 자원(메모리, 파일 등)을 공유한다.

쓰레드는 실행 될 시에 프로세스 내의 코드, 데이터, 열린 파일 등의 자원을 공유하며 실행된다.

같은 프로세스(메모리 공간)을 공유하기 때문에 쓰레드 간의 직접 접근이 가능하지만,

동기화 기법을 통해 접근 순서나 방식이 꼬이지 않게 설정 해줘야 한다.

쓰레드 간의 문맥 전환이 발생하면 상대적으로 비용이 낮다.

프로세스, 쓰레드의 생성, 소멸등의 관리는 운영체제의 **커널**에서 이루어진다.

- 💡 문맥 전환 (Context Switching) 이란?
    
    CPU가 한 프로세스나 쓰레드에서 다른 프로세스나 쓰레드로 제어를 전환하는 과정이다.
    
    문맥 전환이 발생하는 몇 가지 예시 상황이 있다.
    
    1. 다중 작업 환경 : 프로세스와 쓰레드가 동시에 여러개 실행 되는 상황에서
    
     작업간 전환을 해야 하는 상황이 생긴다.
    
    1. 인터럽트 : 각종 인터럽트(interrupt) 이벤트가 발생하면 현재 실행중인 작업을 중단하고
    
     작업을 전환하여 해당 이벤트를 처리한다.
    
    문맥 전환을 하기 위해선 현재 작업의 상태를 저장하고,
    
    다음 작업을 스케쥴링 알고리즘에 따라 선택한다.
    
    다음 작업이 끝나면 상태를 저장했던 작업를 다시 로드한다.
    
    이러한 과정은 **추가적인 시간(오버헤드)**가 발생하기 때문에, 
    
    시스템 성능이 저하되는 원인이 될 수 있다.
    
    따라서 문맥 전환 관리는 중요하다.
    
- 💡 동기화(Synchronization) 란?
    
    다중의 실행 단위가 공유 자원에 안전하게 접근 하고 상호 작용 하는 것을 보장하는 프로그래밍 기법.
    
    쉽게 말해 여러 프로세스나 쓰레드가 구조 상 순서에 맞게 
    
    공유 자원에 접근하여 문제 없이 동작하게 만드는 것을 말한다.
    
    동기화를 하지 않으면 데이터 레이스(Data race)가 발생할 수 있다.
    
    <aside>
    💡 데이터 레이스 (Data race) 란?
    여러 실행 단위가 공유 자원에 동시에 접근하려고 할때 발생하는 문제이다.
    쉽게 예를 들자면, 
    어떠한 공유 변수에 프로세스 A가 값을 수정하려고 접근하기 전에,
    프로세스 B가 먼저 접근하여 값을 수정해버리면
    프로세스 A는 수정할 값이 본래 값과의 불일치 상황에서 수정을 하게 되어
    예기치 않은 결과값이 나올 수 있다.
    
    </aside>
    

임계 영역 (Critical section) : 여러 실행 단위가 동시에 접근하면 안되는 공유 자원 혹은 데이터 영역

실행 단위는 프로세스가 될 수도 있고 쓰레드가 될 수도 있다.

임계 영역을 보호하는 동기화 기법은 **뮤텍스**가 있다.

<aside>
💡 실제 코드에서 임계 영역은 코드 블럭으로 구분된다.
mutex_lock 함수 바로 밑 줄부터
mutex_unlock 바로 윗 줄까지가 임계 영역이다.

</aside>

뮤텍스(Mutex) : 상호 배제(Mutual Exclusion)방식을 제공하는 동기화 기법.

0 또는 1인 이진 상태의 변수로 표현되며 두가지 연산인 Lock(획득) 과 Unlock(해제)가 존재한다.

임계 영역에 들어가려면 반드시 뮤텍스를 획득해야 하며, 하나의 뮤텍스만이 존재한다.

따라서 하나의 실행 단위가 임계 영역에 들어가면 다른 실행 단위는 들어 갈 수 없기 때문에 대기한다.

Lock 연산은 임계 영역에 들어 가기 전에 뮤텍스를 획득하고,

Unlock 연산은 임계 영역에서 나온 후에 뮤텍스를 반환한다.

뮤텍스는 여러개 선언될 수 있고 각각 다른 임계 영역을 관리 할 수 있다.

세마포어(Semaphore) : 정수 단위의 카운팅 기반인 동기화 기법.

정수형 변수값(세마포어)을 기준으로 실행 단위가 공유 자원에 접근 할 수 있는지를 결정한다.

두 가지 연산인 P(획득) 연산과 V(해제) 연산이 있는데,

획득(P)과 해제(V) 연산은 뮤텍스의 Lock 과 Unlock 의 연산 방식과 비슷하지만,

뮤텍스는 하나이고 세마포어는 하나 또는 그 이상을 가질 수 있는 것이 큰 차이이다.

따라서 다중의 실행 단위가 공유 자원에 접근이 가능하다.

세마포어도 여러개 선언이 가능하다.

접근을 시도하는 실행 단위는 wait 연산을 통해 접근을 시도할 수 있다.

wait 연산은 세마포어 값을 감소시킨다.

세마포어 값이 0보다 크면 값을 감소시키고 공유 자원에 접근하며 연산이 끝난다.

세마포어 값이 0인 상태라면 wait 연산을 시도한 실행 단위는 접근이 차단되며 그 상태로 대기한다.

post 연산은 세마포어 값을 증가시킨다.

post 연산을 통해 접근 중인 실행 단위가 작업을 끝내고 접근을 해제한다.

해제 한 후의 세마포어 값은 +1이 되기 때문에 대기중인 상태의 실행 단위는 이제 접근 할 수 있다.

이러한 작동 방식으로 인해 실행 단위는 세마포어의 값만 큼 공유 자원에 접근 할 수 있다.

세마포어는 FIFO(First in First out) 방식이다. 이 말은 먼저 wait 연산을 수행한

실행 단위가 대기열에서 가장 먼저이며, 다른 실행 단위가 post 연산을 할 시에

대기열중 가장 먼저인 실행 단위가 임계 영역으로 들어간다.

뮤텍스, 세마포어 두 개의 동기화 기법 모두

들어 가기 전에 증거 값을 가지고 들어가고, 나온 후에 증거 값을 반환하는 게 특징이다.

그러나 뮤텍스 기법에서 특유의 문제가 발생할 수 있는데, 바로 데드락(Deadlock)이다.

데드락 (Deadlock) : 두개 이상의 실행 단위가 서로에게 있는 자원을 서로 기다리면서 무한히 대기하는 상태

이러한 상태를 “교착 상태” 라고도 부른다.

이 상태가 발생하려면 몇 가지 조건이 필요하다.

1. 상호 배제(Mutual exception) :  공유 자원에 한번에 하나의 실행 단위만 접근 할 수 있어야 한다.
2. 점유 대기(Hold and wait) :  실행 단위가 적어도 하나 이상의 자원을 점유 한 상태로 
다른 실행 단위 점유의 자원을 기다린다.
3. 비선점(No preemption) :  다른 실행 단위의 점유 자원을 강제로 뺏지 못하며
스스로 반납할 때 까지 기다려야 한다.
4. 순환 대기(Circulat wait) : 순환적 구조로 실행 단위가 점유 자원을 기다린다.
예를 들어, 프로세스 A가 프로세스 B의 자원을 기다리고
프로세스 B는 프로세스 C의 자원을 기다리고
프로세스 C는 프로세스 A의 자원을 기다린다.
원을 그리며 기다린다고 생각하면 편하다.

데드락이 발생하면 실행 단위끼리 무한하게 점유 자원을 기다리는 상황이 발생한다.

프로세스나 쓰레드는 CPU를 통해 실행되는데,

이러한 실행 단위가 무한 대기 상태에 들어가면 프로그램이 끝나지 않아 

해당 CPU를 무한 점유하게 된다.

따라서 다른 실행 단위가 해당 CPU를 사용하지 못하여 CPU 성능이 저하되는 결과를 초래한다.

또한 각 실행 단위가 이미 점유하고 있는 자원들도 무한히 점유되는 상황이다.

그럼 시스템은 이 자원들을 더 이상 사용할 수 없어 자원이 낭비된다.

이러한 이유들 때문에 데드락이 생기지 않게 코드를 잘 구성해야 한다.

# 전역 룰

프로그램의 매개변수는 선택적 매개변수(optional argument)까지 **총 5개**이다.

number_of_philosophers : 철학자들의 인원 수 임과 동시에 포크의 갯수.

즉 철학자 한명당 포크가 하나씩 놓아진다.

time_to_die (밀리세컨드 단위) : 철학자가 안먹고 버틸 수 있는 시간이다.

이 시간동안 안먹고 시간이 지나버리면 죽는다.

time_to_eat (밀리세컨드 단위) : 철학자가 먹는데 걸리는 시간이다.

이 시간동안 철학자는 2개의 포크를 쥐고 있어야 한다.

time_to_sleep (밀리세컨드 단위) : 철학자가 자는데 걸리는 시간.

number_of_times_each_philosopher_must_eat (선택적 매개변수): 철학자들이 먹을 총 횟수 이다.

모든 철학자들이 이 횟수 만큼 식사를 끝마치면 시뮬레이션은 멈춘다.

만일 이 매개변수가 없을 시에는 철학자가 한명이라도 죽을때까지 계속된다.

각 철학자들은 인덱스 번호가 1번부터 number_of_philosophers 까지 있다.

1번 철학자의 옆 한편에는 number_of_philosophers 번호의 철학자가 앉아있다.

나머지 철학자들은 N 양 옆에 N - 1 과 N + 1 이 앉아있다.

원탁에 특정 방향으로 인덱스 번호 순으로 앉은 것이다.

# Mandatory 파트

철학자들을 **쓰레드(Thread)**로 구현해야 한다.

철학자들 사이에 포크가 하나씩 놓여있다.

따라서 각 철학자의 왼쪽과 오른쪽에 포크가 하나씩 놓여있다.

만약 철학자가 한명이라면, 포크도 원탁에 하나만 놓여있다.

철학자들이 포크를 복제하는 것을 막기위해, 포크는 뮤텍스 사용하여 동기화 시켜야 한다.

철학자의 대기상태가 생각하는 상태인가..? → 맞는듯. 생각하는 시간은 따로 제한이 없다.

음식을 최대로 동시에 먹을 수 있는 철학자의 수는 **포크 갯 수(철학자 수) / 2** 이다.

각 철학자에게 포크가 하나씩 주어지고, 해당 철학자의 인덱스에 해당하는 포크는

해당 철학자의 오른쪽에 놓여있다. 그렇다면 해당 철학자가 밥을 먹으려면, 자기 인덱스에

해당하는 오른쪽 포크를 들고, 이전 인덱스의 철학자의 포크인 인덱스 - 1 포크도 집어야 한다.

원탁이기 때문에 인덱스가 0 에서 - 1 이 되면 인덱스 최대값으로 바뀌어야 하고,

최대값에서 + 1 이 된다면 0으로 바뀌어야 한다.

상황에 따라 들어온 매개변수에 usleep(arg * 1000)을 하면 될 것 같다.

예를 들어 먹는시간이 200 으로 들어오면

철학자가 양손에 포크를 집으면 뮤텍스로 양 손 포크를 잠근 상태일거고 그 상태에서

mutex_unlock을 하기 전에 usleep(time_to_eat * 1000)을 한 다음

잠금을 해제하여 양손에 있는 포크를 놓아주고 철학자를 재워준다.

철학자가 죽기까지 남은 시간을 어떻게 줄까?

철학자 상태 노드의 멤버변수에 time_left를 주었는데 

철학자 스레드 배열을 생성하고 나서,

예를 들어 반복문에서 가장 위에 gettimeofday를 해주고,

이전 반복 와 현재 반복간의 시간 차를 구하기 위하여 변수를 2개 만들어주고

현재 반복 - 이전 반복 = 이런 식으로 연산하여 0 이상이면

→ 아니 그냥 매개변수로 얼만큼 시간이 걸리는지 아니깐 그걸 time_left에 뺸 다음

상황 판단하면 되는거 아냐? 0 이하 수가 나오면 죽는거니깐

애초에 특정 행동을 하면 그 시간동안 해당되는 쓰레드(철학자)는 usleep모드로 들어갈텐데

그 상태에서 변수값을 수정 할 수가 없잖아.

→ 근데 첫번째 가정에 0 이하의 수가 큰 음수값으로 나온다면,

     그 이전에 다른 철학자가 행동을 끝내면 그걸 알아야 하잖아.

     → 그럼 이건 어떨까? 현재 time_left - 현재 행동에 걸릴 시간 = 이 시간이 

0 이하가 된다면 time_left만큼 usleep을 주고 죽이는거지.

바로 이 철학자를 죽이지 않으니까

나머지 철학자들은 이 철학자가 죽기 전까지 다른 행동을 하겠지.

time_left는 이전에 실행한 usleep의 매개 변수만큼 빼주자.

→ 이 문제를 해결하려면 2가지 방법이 있다.

1. usleep을 최대한 쪼개어 반복문에서 gettimeofday를 통해 실시간으로 time_left를 구할건지.
2. usleep모드로 들어가기 전에 매개 변수인 걸릴 시간을 time_left에 빼서 미리 계산을 할건지.

→ 일단 조사를 해보니 두 방법 다 가능한 줄 알았는데, 2번 방법은 문제가 하나 있다.

1번은 반복하면서 usleep(200); 정도를 주면 밀리 세컨드 +1 차이는 다 잡아낸다.

현재 시간, 지나온 시간 차이 계산해서 매개 변수만큼 도착하면 행동을 멈추면 될듯.

생각은 무한반복이고.

2번은 말그대로 미리 계산 하면 되는데 이 방법이 생각하는걸 커버 못한다.

먹는거랑 자는거는 시간이 따로 정해져있어서 미리 계산하고 시간 설정이 가능한데

생각하는건 따로 시간 제한이 없어서 얼마나 걸릴지 미리 알 수 가 없다.

미리 계산을 할 수 가 없기 때문에, 1번으로 하기로 했다.

철학자가 다 먹고 나서는 time_left를 처음 매개 변수(time_to_die)로 초기화해준다.

철학자들이 스레드인데 인덱싱을 어떻게 줄지 생각해보자.. 스레드라 생각보다 복잡하다.

스레드 함수의 매개변수로 스레드의 인덱스가 공유 변수인 상태라 이 인덱스 변수에

data_race가 일어난다.

공유 변수가 아닐 수 있는 방법이 뭐가 있을까..

→ 해결 방법을 알아냈는데, 각 스레드 마다 독립된 메모리 공간을 주어서

스레드간 공유하면 안되는 데이터를 그 공간에 넣어야 한다.

스레드 마다 같은 구조체를 넘겨주면 각 스레드가 모든 변수를 공유하며

인덱스 같은 변수들은 모든 스레드에서 값이 동일해져 사용이 불가능하다.

공유해야 하는 변수들 같은 경우는 따로 구조체 하나를 만들어 각 스레드 메모리에

포인터로 연결 시켜줄까 생각중. 구조체를 만들지 아님 단순 멤버 변수 형식으로 줄지 고민.

→ 공유 데이터는 각 스레드에게 포인터로 연결시킨 구조체로 주었다.

스레드(철학자)간 연결 고리를 배열로 줄까 리스트로 줄까 고민중이다.

둘다 가능한데, 배열이든 리스트든 일단 스레드 생성하는 함수 호출 전에 

메모리 할당부터 완료시켜 완성된 메모리 자료구조를 넘긴다.  

배열은 노드 배열로서 2차원 배열로 선언하되,

멤버 변수에 2차원 배열의 시작 주소를 담아야 배열 전체의 참조가 가능하다.

이렇게 하지 않으면 스레드 함수 매개변수로 [i]번째 요소를 넘길때

해당 요소 외에 다른 배열 요소 참조가 불가능하다.

멤버 변수로 2차원 배열 시작주소가 있다면

스레드 함수 내에서도 인덱스로 접근하여 다른 요소도 참조 가능.

이렇게 귀찮게 하는 이유는 2차원 배열 시작주소 자체를 스레드 함수 매개 변수로 넘기면

몇 번째 인덱스 요소를 참조할지 스레드 함수에서 알 길이 없다.

리스트로 하면 해당하는 인덱스만큼 리스트 주소를 옮겨주고

해당하는 리스트 노드 포인터를 넘긴다.

멤버 변수로 인덱스도 있고

next, prev 포인터가 있기 때문에 주변 노드들도 전부 참조 할 수 있다.

이 짧은 시간차 안에서 집어야할 포크가 다른 철학자가 집는 경우가 생길 것이다.

그럴 경우엔 포크 경쟁상태이기 때문에, 집으려는 철학자와 집은 철학자 중에

time_left가 적은 철학자에게 양보해 주는 방식으로 한번 짜보자.

→ 이 방식은 잘못되었다.

과제 pdf에 기재되어 있는 조건 중에 철학자들은 서로 대화하거나

서로 언제 죽을지 알 수 없는 상태로 구현하라는 조건이 있다.

철학자가 죽기까지 서로의 time_left를 알 수 없는 새로운 로직으로 갈아 엎어야 한다.

새로운 로직으로 생각한게, 일단 짝수 번호부터 처음에 먹인다.

나머지 인원들은 먼저 먹은 인원 들이 포크를 내려놓기를 기다린다.

이렇게 하면 데드락도 안생기고 인원수가 짝수면 번갈아 가며 잘 먹는다. 

그러나 인원수가 홀수면 죽는다.

예를 들어 5명이면 포크가 5개니깐 한번에 먹을 수 있는 인원 숫자는 2이다.

2명 먹고 2명 먹으면 나머지 한명이 무조건 낙오되기 때문에

낙오된 한명이 대기열에서 우선 순위를 가져야한다.

우선 순위를 가진 다는건 결국 포크를 먼저 집는 것이기 때문에,

포크를 놓는 코드를 아예 빼버렸다.

이 코드를 빼면 낙오된 한명이 한 손만 포크를 잡는데,

잡은 쪽의 철학자가 자는 상태이기 때문에 잡을 수 있었던거다. 그 말은,

자는 그 철학자가 일어날 때 포크를 집으려 할거고, 그 때 포크를 놓고 있었다면

자고 일어난 철학자가 포크를 먼저 집어버릴 수도 있다.

따라서 낙오된 철학자가 잡고 있어야 방금 깬 철학자에게 포크를 뺏기지 않고 식사를 할 수 있다.

포크를 못 잡고 있던 다른 손은 괜찮은게, 그 손 쪽의 철학자가 먹고 있는 상태라서 못 잡는 거라

다 먹고나서 놓아주면 알아서 바로 잡는다.

또한 포크를 놓는 코드가 없으면 데드락이 생길까 걱정했는데,

처음에 짝수 번째 철학자들부터 포크를 잡고 시작하니 데드락은 안 생긴다.

포크를 잡으려고 시도하는 행동에 시간차가 생겼기 때문.

이 로직으로 짜니 실제로 죽지 않고 잘 돌아간다.

포크를 놓아야 할지 체크하는 함수와 놓는 함수도 제거하니 성능도 좀 더 나아졌다.

아무리 고민하고 찾아봐도 허용 함수 내에서 sem_wait과 동시에 지나가는 시간을

계산 하는 방법은 스레드를 따로 빼는 방법밖에 없다.

각 프로세스마다 스레드가 하나 씩 생기는건 크게 신경 한 써도 되는듯.

부모(모니터링)프로세스는 fork를 뜰때 자식의 pid값을 전부 받을 수 있으니

그걸 배열로 저장해서 끝날때 waitpid함수로 전부 기다려 줄 수 있다.

그러나 이걸 가지고 자식 프로세스에서 뭔가를 기대하기는 어려울듯.

자식 프로세스에서 fork 반환값은 0이고 getpid()함수 사용도 허용하지 않는다.

이 방법으로 얻은 pid들에게 kill(sigterm)도 해준다.

실행 단위 구조 :

메인 → 모니터링 스레드(ate_cnt)

   |

자식(철학자들) → 포크 집는 스레드(sem_wait)

자 그럼 결정 됬다.

모니터링 스레드 만들고, 자식 프로세스들로 철학자들 만든다.

모니터링 스레드는 sem_wait으로 ate_cnt를 체크한다.

죽는 모니터링은 따로 하지 않아도 되는게

메인 프로세스에서 sem_wait(dead)를 하고,

자식들 중 죽으면 죽은 자식이 sem_post(dead)하면 된다.

메인 프로세스와 모니터링 스레드에서 시뮬레이션을 종료해야하는 상황이 포착되면

모든 철학자 프로세스에게 kill(sigterm)을 한다.

각 프로세스(철학자)들도 sem_wait 연산을 하는 스레드를 따로 빼서

생각 하는 중에는 따로 뺀 sem_wait을 하는 스레드가 동시에 실행 중이면서

프로세스는 지나가는 시간을 계산하는 방식으로.

여기서 뺀 스레드가 포크 집는 스레드이다.

포크를 집는 스레드는 생각하는 행동 시작때에 생성된다.

스레드가 sem_wait을 하는 상태에서 프로세스가 kill(sigterm)을 받아서

프로세스 자체가 종료하게 되어 종속된 스레드가 join 이나 detach를

하지 않아도 프로세스가 종료될때 자동으로 스레드 자체 리소스도 정리 된다고 한다.

다만 포크를 집는 스레드 같은 경우는 집은 후에 정상적으로 종료되야 하기 때문에

join을 해주었다.

세마포어가 열린 상태의 프로세스가 종료되면 해당 프로세스의 세마포어는 알아서 닫힌다.

다만 unlink가 되지 않은 상태로 종료되면

세마포어는 재부팅 하지 않는 이상 삭제 되지 않고 남아있다.

무한으로 돌아가는 프로그램이라 ctrl + c로 멈춰야 하는 경우가 많은데

이렇게 종료시키면 unlink가 되지않아 다음 실행때 삭제 되지 않은

오염된 세마포어로 실행되어 시뮬레이션이 정상적으로 돌아가지 않았다.

따라서 시뮬레이션 시작 전 세마포어를 생성 단계에서 생성 전에 먼저 unlink하는

코드를 추가하니 해결되었다.

# 로직 설명

보너스 로직은 생각보다 간단했다.

포크가 양 옆에 있는게 아니라 중앙에 모여있기 때문에

멘데토리처럼 한명씩 띄어서 먹을 필요가 없다.

그러나 우선 순위 없이 무작정 먹이면 굶어 죽는 철학자가 분명히 발생한다.

그래서 멘데토리 처럼 시간차를 이용하기로 했다.

먼저 처음에 정해진 절반을 먹이고, 그들이 다 먹고 잘때

나머지 절반이 먹는 형식으로.

이렇게 하면 시간차가 생겨 계속해서 알아서 잘 먹고 잘 잔다.

다만 처음에 절반을 먹일때, 0번 인덱스 철학자 부터 max / 2번째 철학자 까지 먹인다.

fork를 한 순서대로 인덱싱이 이루어지기 때문에, fork로 생성 된 순서대로 포크를 집게 만들었다.

fork한 순서대로 시뮬레이션이 시작하기 때문에 효율성 측면에서도 이 방법이 좋다.

다만 프로세스라 데이터를 공유 하는게 어렵기 때문에

처음 절반이 포크를 집을 수 있게 세마포어로 동기화를 시켜주었다.

한 명이 2개의 포크를 집을때 까진 다른 철학자는 집을 수 없게.

동기화를 시켜주지 않으면 처음에 집을때

마지막에 집는 max / 2번째 철학자와 max / 2 + 1 번째 철학자가

포크를 하나씩 쥐는 경합 상태가 가끔 발생한다.

# Data Race 발생과 해결 과정

-fsanitize=thread 옵션을 주면 data_race를 잡아주는데 이런 옵션이 있는 줄도 모르고

나중에 돌려보니 생각 보다 많이 잡혀서 놀랐다.

data_race가 발생하면 내가 원하는 값으로 읽히거나 쓰이지 않을 거라 예상하고

딱히 옵션을 주지 않아도 알아서 결과값이 틀릴거라고 생각했는데

맞는 결과 값이 나와도 race가 발생한다는게 의외였다.

일단 data_race가 발생하는 이유는 2개 이상의 다른 스레드(메인 스레드도 포함)가

특정 변수에 동시에 접근할때, 적어도 한 스레드가 쓰기 작업을 하려고 하면 발생한다.

따라서 이 접근되는 특정 변수들에 한번에 하나씩만 접근 할 수 있게 동기화 시켜야 한다.

해결 방법은 공유되는 특정 변수를 최대한 줄이거나, 공유 해야 하는 정보가 있다면

해당 정보 자체를 공유 변수로 두지않고 그 정보를 담은 플래그 형식의 변수를 공유 변수로 두었다.

이렇게 하면 좋은 점이 해당 정보는 다양한 곳에서 많이 쓰일 가능성이 높고,

쓰이는 곳마다 동기화 처리를 해놓으면 성능은 분명히 느려진다.

따라서 해당 정보가 공유될 순간에만 동기화 처리를 할 수 있게 공유 변수를 따로 선언하여,

해당 정보의 데이터를 공유 변수에 넣어주고 공유 변수의 값을 쓰거나 읽을 때만

동기화 처리를 하면 훨씬 나은 성능을 보여주었다.

동기화가 적을 수록 문맥 전환도 적고, 그로 인한 오버헤드도 줄기 때문이다.

또한 공유 데이터마다 뮤텍스를 만들어서 최대한 락 분할을 시켜주고,

락을 할때도 락 언락 사이에 코드 한 두줄만 유지하도록 함수로 따로 빼서 최적화를 시켜주었다.

# 쓰레드 실행의 불확실성

쓰레드는 병렬로 실행 되기 때문에 어떠한 작업을 순차적으로 진행 하는게 아닌

각자 생성된 후에 운영 체제의 스케쥴링 알고리즘에 따라 실행 순서가 결정 된다.

따라서 코드를 짠 사람은 특정 스레드가 언제 실행되는지 알 수가 없다.

멀티스레드 프로그래밍에서는 일반적으로 이러한 특징 때문에

특정 스레드의 실행 순서나 완료 시점을 가정하지 않고 설계하는 것이 좋다.

대신, 스레드 간의 동기화 메커니즘(예: 뮤텍스, 세마포어 등)을 사용하여 

필요한 순서나 동기를 명시적으로 관리해야 한다.

그렇지 않으면 data race가 발생하거나 데드락같은 상황이 발생 할 수 있다.

# 세마포어의 광범위한 사용 가능 범위

세마포어는 어떤 프로그램의 어떤 프로세스에서 선언하던

sem_open 함수를 통한 이름이 부여된 세마포어를 생성한다면

시스템 전역에서 접근 가능하다.

이 말은 한 프로그램의 특정 프로세스에서 이름이 있는 세마포어를 생성한다면

해당 프로그램의 다른 프로세스에서 접근 할 수 있는 것은 물론,

다른 프로그램에서도 접근이 가능하다.

이게 가능한 이유는 이름 있는 세마포어는 파일 시스템에 생성되기 때문이다.

파일 시스템에 있는 특별한 공유 메모리 영역(보통 /dev/shm)에 생성되어

시스템의 모든 프로세스들이 접근 가능하다.

이름이 없는 세마포어는 단순 메모리(힙, 스택 등) 영역에 생성되어

따로 통신 수단을 구현하거나 공유 메모리 영역을 사용하지 않는 한

다른 프로세스에서 사용할 수 없다.

# 문맥 전환

특정 스레드 혹은 프로세스가 cpu를 점유했는데

lock 상태에 걸리면 그 순간부터 아무것도 못하고 다른 객체로 전환되며

전환 과정에서 생기는 문맥 전환 코스트 때문에

시간차가 생기면서 철학자가 죽는 경우가 생긴다.

최대한 lock상태를 안 만들어야 문맥 전환이 적게 일어난다.

성능 측면으로 보더라도 병렬 프로그래밍에서 문맥 전환 최적화는 필수 인 것 같다.

# 스핀락(Spin Lock)

내가 짠 로직에서 생각하는 행동 중에는

무한 루프를 돌며 지나가는 시간을 재고 있음과 동시에 포크를 집으려고 계속 시도한다.

이렇게 락을 얻을 때까지 루프를 돌며 지속적으로 확인하는 방법이 스핀락 이라고 한다.

스핀락의 특성은 다음과 같다.

1. **활성 대기 (Busy-waiting)**:
    
    스레드가 락을 얻을 때까지 CPU를 계속 사용하여 반복적으로 확인 한다.
    
    이 과정에서 스레드는 대기하는 동안 실행을 계속하며, 이로 인해 "스핀"이라는 이름이 붙었다.
    
2. **짧은 대기 시간에 적합**:
    
    스핀락은 짧은 시간 동안 자원을 보호해야 할 때 유용하다.
    
    대기 시간이 길어지면 CPU 자원을 낭비하게 되므로 비효율적일 수 있다.
    
3. **컨텍스트 스위칭 오버헤드 없음**:
    
    스핀락은 대기 중에 스레드를 블록(block)시키지 않기 때문에
    
    문맥 전환으로 생기는 오버헤드가 없다.
    

# 동시에 시작하기

위에 기재한 것 처럼,

병렬 프로그래밍에서는 어떤 실행 객체가 언제 시작될지 내가 정할 수 없다.

실행 객체들이 서로 상관 관계가 있고 이러한 실행 객체들로

어떠한 시뮬레이션을 돌리는 프로그램 이라면 동시에 시작하는게

정확한 결과가 나올 것이다.

내 코드에서는 스레드나 프로세스가 생성될때 모든 실행 객체가 생성되기를

기다리지 않고 생성되면 바로 시뮬레이션을 시작 시켰는데

이렇게 할 경우 타임스탬프 기준으로 철학자가 죽어야 하는데

죽지 않고 행동하는 경우가 생긴다.

물론 철학자 자체는 자기 남은 시간을 스스로 감소 시키기 때문에

자기 자신은 죽지 않는게 맞지만 타임스탬프는 철학자가 재는게 아닌

시간 자체를 재기 때문에 각자의 생성 시간이 달라 타임 스탬프 기준으론

누군가 죽어야 할 타이밍에 죽지 않는것 처럼 보일 수 있다.

이러한 경우를 방지하고자 스레드는 시작 시간을 현재 밀리 세컨드 + 100으로

정해주고 모든 스레드가 만들어 지길 기다린 후에 +100 밀리  세컨드에서 동시에

모두가 시작한다.

보너스 같은 경우는 100 밀리세컨드 안에 프로세스들이 전부 생성 될 거라는 보장이

없기 때문에 세마포어로 동기화를 시켰다.

생성된 프로세스가 새마포어를 post하고 메인은 wait으로 그걸 받아서

계수가 철학자 수 만큼 도달하면 start 세마포어를 post 함으로써

모든 프로세스를 시작시킨다.

무엇보다 운영체제에 관한 공부를 더 해야겠다는 생각이 들었다.

평가를 진행하면서 내 배경지식이 부족해서 애매하게 대답한 것들이

생각보다 많았다.

면접 볼때 이런 상황이 생기면 안되니깐 시간이 나면 공부해보자.

성능이 좋은 것도 좋지만 프로그램의 안정성이 더 중요하다는 것도.
