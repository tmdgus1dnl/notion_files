# push_swap

설명: 스택 기반 정렬 알고리즘
상태: 완료

스택으로 들어온 숫자들을 최소한의 명령어 수행으로 정렬하는 과제이다.

# Mandatory 파트

stack 을 리스트 형식으로 구현하여 stack a 와 b 를 만든다.

stack a 에는 매개변수로 들어오는 int범위 내 정수형 숫자들이 들어온다.

들어오는 매개변수 숫자들 중 첫번째로 들어오는 매개변수가 stack a 의 top 에 있어야 한다. (순서 조심)

<aside>
💡 stack의 top이 리스트의 마지막 이어야 유리하다. 그 이유는 리스트의 마지막 노드를 추가하거나 삭제하는 것이 리스트 구조상 편리하기 때문이다. stack의 주요 명령어(push, pop) 등이 top에서 이루어 지는걸 생각하면 top을 리스트의 마지막으로 정하고 구현 하면 편리하다.

</aside>

stack b 는 처음엔 무조건 비어있어야 한다.

stack a로 들어온 숫자들을 주어진 명령어를 통해 정렬을 시작한다.

sa = stack a 에 있는 가장 상위의 2개 숫자를 스왑한다. (숫자가 하나 있거나 없을 시 명령 수행 x)

sb = stack b 에 있는 가장 상위의 2개 숫자를 스왑한다. (숫자가 하나 있거나 없을 시 명령 수행 x)

ss = sa 와 sb 동시에 수행.

pa = stack b 에 있는 가장 상위의 숫자 1개를 stack a 로 push 한다. (stack b 가 비어있을시 명령 수행 x)

pb = stack a 에 있는 가장 상위의 숫자 1개를 stack b 로 push 한다. (stack a 가 비어있을시 명령 수행 x)

ra = stack a 에 있는 모든 숫자들을 1칸씩 위로 올린다. (stack a 의 최상위 숫자는 최하단으로 내려간다.)

rb = stack b 에 있는 모든 숫자들을 1칸씩 위로 올린다. (stack b 의 최상위 숫자는 최하단으로 내려간다.)

rr = ra 와 rb 동시 수행.

rra = stack a 에 있는 모든 숫자들을 1칸씩 아래로 내린다. (stack a 의 최하위 숫자는 최상단으로 올라간다.)

rrb = stack b 에 있는 모든 숫자들을 1칸씩 아래로 내린다. (stack b 의 최하위 숫자는 최상단으로 올라간다.)

rrr = rra 와 rrb 동시 수행.

<aside>
💡 pa 와 pb 는 동시에 수행할 필요가 없다. push a 를 수행하고 push b 를 수행하면 push a를 하기 전의 원래 상태로 돌아오기 때문에 의미가 없는 명령어.

</aside>

</aside>

# 구현과 최적화 기록

rr이나 ss같은 2개 동시 명령을 수행하는 명령어는 둘중 하나가 터지면 어떻게 작동하지?

작동은 한다. 다만 터지는 스택만 수행 안하고 다른 스택은 수행. 근데 애초에 이런 일이 있을지..

그리디 알고리즘을 사용할 예정인데.. 푸시 스왑에 어떤식으로 적용해볼까?

일단 아이디어 하나가 떠올랐다.

명령어 하나를 수행하면 스택 a 는 오름차순, 스택 b는 내림차순에 가까울때 

해당 명령어 점수를 가장 높게 주는거야.

여기서 문제는 언제 스택a 에서 b 로 가는 명령어를 수행할지 정해주는건데..

아마도? b로 빼버리면 a가 오름차순에 가까운데, 그 경우가 b도 내림차순에 가까운 경우만 수행되게?

근데 여기서 또하나, b의 바닥은 정삼각이면 유리하다. 왜냐하면 rrb를 한번 수행하면 바로 가장 위로 가서 역삼각을 만들 수 있기 때문에.

다른 여러가지 정렬들이 있으나 전부 부분 삼각형으로 만들고 그걸 통합하고 큰 삼각형으로 만든다.

전부 모래시계랑 비슷하다.. 재미없어보인다.

일단 다 떠나서.. 생각난게 하나 있다.

stack a의 맨 위 원소 2개를 비교해서 최상위 원소가 밑 원소보다 더 작으면 b로 푸시.

만약 더 크면 아래랑 스왑한다. 동시에 스택 가장 아래의 원소랑 비교도 해서 

가장 아래 원소가 더 작으면 로테이트한다.

더 크면 사실상 정렬이 되있는거라 무시 해도 된다고 생각한다..(일단 지금은)

이 과정을 반복하고 stack b는 반대로 반복한다.

이러한 과정중에 stack a, b가 동시에 스왑이나 로테이트가 가능한지 체크도 해보면서 최적화도 가능할듯?

왜 push_b만 실행이 되는지 봤는데, 내 반복문 순서가 스왑이 먼저라서 일단 위에 2개를 체크를 한 다음 스왑을

진행한 후에 push 함수로 들어가니 push_b가 항상 참 인거였다.

리스트 노드 멤버변수에 gap을 추가했다. 여기에 stack_a 소팅이 끝난 후 남은 숫자중 최상위 숫자와 stack_b 리스트에 있는 숫자 전부를 비교해서 그 차이 값을 gap에 넣어준다. 

그런 후 비교문에서 gap의 값이 가장 작은 수를 stack_b를 소팅하면서 push_a 할 예정. 이 과정을 반복한다.

푸시는 항상 맨 나중에 하는게 좋을듯.

소팅이 될 때가 있고 안될 때가 있다. 

난수 개수는 크게 상관없고 안되는 이유는 아마도 stack_b를  소팅하는 반복문에서 swap과 rotate의 조건이 끝나버려서 그런듯. 

모든 조건이 끝났을 때 어떻게 해야할지 생각해보자. 강제로 다른 명령어들을 수행시켜 버릴지 아니면 push_a를 시켜버릴지 고민이다.

추가로 gap을 노드에 기록할때 내가 짠 코드를 보니 stack_a의 첫번째에 있는 제일 큰수랑 한번만 비교하고 기록해면 gap을 기록하는 반복문은 더 이상 필요하지 않다. 이 부분 나중에 수정 가능할듯.

가중치를 줘보자. gap + index 차이 값을 가중치로 한번..

가중치도 명령어 개수가 좀 넘어간다.

명령어에 점수를 준다고 생각해보면.. 특정 명령어를 수행했을때 stack_a의 가장 상위 수의 가장 가까운 수가 가장 적은 명령어로 위에 쌓일때 가장 높은 점수를 준다.

더 작은수 들어오면 로테이트..?

가장 weight이 작은 b의 요소 위치를 구하고, push_a 하기 위해 rb 혹은 rrb를 하여 b의 top으로 보낸다.

push_a는 b의 top 크기가 a의 top 크기보다 작은 경우에 수행, 더 크면 push 하지않고 a를 ra 혹은 rra한다.

weight을 잴때 b의 top 과 index 거리 차이 값도 계산하여 gap과 합산 한다. 그리하여 가장 최적의 값이 push 되리란 보장이 없다.

b의 gap 을 계산할때 a의 top보다 큰 수도 gap값을 계산하여 weight에 합산하기 때문에, a의 top 보다 더 큰 수가 min_weight로 설정되는 경우가 있어 a에서 정렬이 필요한 경우가 있다.

a도 마찬가지로 b의 top 과 큰수 혹은 작은수의 gap 을 계산하여 대입한다.

따라서 a안에서 push_a 하기 전에 먼저 값 정렬이 될 수 있게, 

혹은 b의 min_weight보다 더 큰수를 a의 top으로 올릴 수 있게

2가지의 반복문으로 a의 min_gap을 계산하여 정렬을 먼저 시도한다.

첫번째 반복문은 일단 min_weight가 b의 top 에 있는 상태에서, stack_a 의 bottom이 top보다

작아야하며, b의 top보다는 커야한다. 그래야 먼저 정렬을 시키고 최적의 값을 push_a 할 수 있다.

두번째 반복문은 똑같이 min_weight가 b의 top 에 있는 상태에서, 똑같이 b의 top보다는 커야하며,

동시에 제일 gap이 적은수를 a의 top으로 올린다.

쉽게 말해 더 큰 수의 gap중에서 가장 gap이 작은 수를 올린다.
