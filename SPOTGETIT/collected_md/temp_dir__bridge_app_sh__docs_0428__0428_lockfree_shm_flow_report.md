# 락프리 기반 공유메모리 데이터 파이프라인 분석 보고서

## 1. 개요
브릿지 앱(Bridge Daemon)과 Qt 애플리케이션 간의 데이터 교환에서 시스템 병목을 없애기 위해 **락프리(Lock-Free) 트리플 버퍼링 구조**가 적용되어 있습니다. 
특히 네트워크로 들어온 이미지 조각(Fragment)들이 `reassemble` 과정을 거쳐 조립되고 공유메모리로 안전하게 전달되는 전체 흐름을 3가지 실제 구동 상황(Scenario)으로 나누어 분석했습니다.

---

## 2. 파이프라인 동작 흐름 (3가지 상황별 분석)

### 상황 1: 정상적인 이미지 수신 및 전달 흐름 (The Happy Path)
가장 이상적이고 일반적인 데이터 파이프라인의 처리 순서입니다.

1. **슬롯 재조립 (`reassembly_shm_thread`)**
   네트워크 큐에서 UDP 프래그먼트가 하나씩 팝(pop) 됩니다. 브릿지 앱은 `find_slot()`을 통해 동일한 `frame_id`를 가진 조립용 슬롯을 찾거나 `alloc_slot()`으로 새로 할당하여, 수신된 조각을 순차적으로 안전하게 메모리(`rs->buf`)에 복사합니다.
2. **이미지 완성 (`commit_image`)**
   마지막 조각까지 도착하여 `frags_recv == frags_total` 조건이 충족되면 조립이 완성됩니다.
3. **공유메모리에 이미지 넣기**
   현재 공유메모리에 있는 `img_ready_idx`를 확인하고, `(ready + 1) % IMG_SLOTS` 수식을 통해 3개의 버퍼 중 사용 중이 아닌 **다음 빈 슬롯**에 완성된 이미지 데이터를 단숨에 복사(`memcpy`)합니다.
4. **원자 연산 (Atomic Operation)**
   복사가 끝난 직후, `atomic_store(&shm->img_ready_idx, slot)` 함수를 호출합니다. 뮤텍스(Mutex) 락을 걸지 않고, 오직 CPU의 원자적 명령어 하나로 "최신 이미지가 있는 슬롯 번호"를 갱신합니다.
5. **공유메모리에서 빼가기 (Qt Consumer)**
   브릿지가 `sem_post()`로 신호를 주면, Qt 스레드가 깨어납니다. Qt는 `atomic_load()`로 가장 최신 인덱스를 락 대기 없이 가져간 후, 브릿지 쓰기 스레드와 겹치지 않는 슬롯에서 안전하게 이미지를 꺼내어 화면에 렌더링합니다.

> [!TIP]
> **왜 락프리인가?** 전통적인 `pthread_mutex_lock`을 사용했다면 브릿지가 복사하는 동안 Qt 화면이 멈추거나, 반대로 화면을 그리는 동안 네트워크 수신이 막히는 핑퐁 현상이 발생합니다.

---

### 상황 2: Qt 앱 렌더링 지연 발생 (트리플 버퍼의 진가)
Qt 애플리케이션 측의 그래픽 처리(UI 갱신, 3D 렌더링 등)에 과부하가 걸려 이미지 처리가 늦어지는 상황입니다.

1. **상황 발생**
   Qt 앱이 `img_slots[0]`번 버퍼에서 이미지를 꺼내와 렌더링을 하느라 오랜 시간(예: 30ms) 해당 버퍼를 점유하고 있습니다.
2. **브릿지 앱의 독립적 쓰기 유지**
   이 와중에도 브릿지 앱은 네트워크에서 새로운 프레임을 재조립 완료했습니다. 브릿지 앱은 락(Lock)을 획득하려고 기다리지 않고, 단순히 수식에 의해 다음 번호인 `img_slots[1]`번에 새로운 이미지를 덮어씁니다.
3. **원자적 인덱스 교체 (Atomic Index Swap)**
   복사가 끝나면 `atomic_store`를 이용해 가리키는 포인터(Index)만 0에서 1로 휙 넘겨버립니다.
5. **결과 및 효과**
   Qt 앱은 0번 버퍼를 읽고 있으므로 데이터가 도중에 찢어지거나 오염되는 현상(Tearing)을 겪지 않습니다. 브릿지 앱 역시 Qt 앱의 느린 처리 속도에 발목 잡히지 않고 프레임 드랍 없이 최고 속도로 메모리에 기록합니다.

> [!NOTE]
> 이처럼 4개의 버퍼(Triple Buffer)와 Atomic 연산의 조합은 생산자(Bridge)와 소비자(Qt)가 서로의 작업 속도에 영향을 주지 않도록 완전히 격리(Decoupling) 시킵니다.

---

### 상황 3: 네트워크 불안정으로 인한 프래그먼트(조각) 중간 유실
로컬 무선망(Wi-Fi) 간섭으로 인해 총 100조각으로 이루어진 프레임 중 일부 조각이 도착하지 않는 상황입니다.

1. **상황 발생**
   프레임의 49번째 조각이 네트워크에서 소실되었습니다. 조립 버퍼에는 99개의 조각만 들어있습니다.
2. **조립 무한대기 현상**
   `frags_recv`가 `frags_total`에 도달하지 못하므로, 해당 슬롯은 공유메모리로 넘어가지(commit) 못한 채 계속 자리를 차지하게 됩니다.
3. **가비지 컬렉션(GC) 가동**
   이를 막기 위해 패킷이 들어올 때마다 `expire_stale()` 함수가 호출됩니다. 각 조립 슬롯이 첫 조각을 받은 시간(`first_recv_us`)을 현재 시간과 비교합니다.
4. **안전한 메모리 회수**
   지정된 타임아웃 100ms (`REASM_TIMEOUT_US`)가 경과하면, 수신을 포기하고 해당 슬롯을 강제 해제(`in_use = 0`)합니다. 
   - 미완성(깨진) 프레임이 공유메모리로 전달되는 것을 원천 차단하여 Qt 측의 디코딩 에러를 방지합니다.
   - 쓰레기 데이터로 인해 조립 버퍼 슬롯(최대 6개)이 모두 고갈되어 새로운 정상 프레임을 받지 못하는 치명적 오류를 막습니다.

---

## 3. 요약
현재 브릿지 메인 소스코드(`reassembly_shm.c`)에 구현된 파이프라인은 단순히 메모리를 공유하는 것을 넘어, **원자 연산(stdatomic)**과 **타임아웃 기반 자동 메모리 관리(GC)**를 통해 예측 불가능한 네트워크 지연과 애플리케이션 과부하 상황에서도 무중단/저지연 성능을 보장하는 매우 견고한 구조로 설계되어 있습니다.
