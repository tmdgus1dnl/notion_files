제공해주신 다이어그램을 바탕으로 Jetson과 Raspberry Pi 5 간의 통신 및 데이터 처리 파이프라인에 대한 아키텍처 보고서를 작성했습니다.

시스템 아키텍처 보고서: 로봇 데이터 통신 및 처리 파이프라인
1. 시스템 개요
본 구조는 Jetson(송신부)에서 생성된 대용량 센서 데이터(ROS App)를 네트워크를 통해 Raspberry Pi 5(수신부)로 전송하고, 수신된 데이터를 재조립하여 공유 메모리를 통해 사용자 애플리케이션(Qt)에 전달하는 고성능 단방향 통신 파이프라인입니다. CPU 코어 고정(Pinning), DMA, 환형/트리플 버퍼를 적용하여 지연 시간을 최소화하고 처리량을 극대화하도록 설계되었습니다.

2. 송신부 (Jetson 환경)
데이터 소스 (ROS App): 세 가지 타입의 데이터 스트림을 생성합니다.

이미지 (Image): 2MB/s

라이다 (LiDAR): 10MB/s

오도메트리 (Odom): N B/s

패킷 단편화 (Fragmentation): 네트워크 전송을 위해 MTU 1400 byte 기준으로 데이터를 분할합니다.

패킷 헤더 구조 (PktHeader):

type: 데이터 타입 (0x01: Image, 0x02: LiDAR, 0x03: Odom)

frame_id: 메시지 고유 ID (특정 시간에 찍힌 하나의 이미지 ID 등)

frag_idx: 현재 패킷의 조각 순서

frag_total: 메시지가 쪼개진 전체 조각 수

offset: 전체 프레임 중 현재 조각이 위치할 메모리 오프셋 시작점

payload_len: 현재 패킷에 담긴 순수 데이터 크기

timestamp: 생성 시간

전송 경로: 커널 버퍼를 거쳐 NIC로 전달되며, 네트워크 대역폭은 660Mbps(80MB/s) 환경을 기준으로 합니다.

3. 수신부 (Raspberry Pi 5 환경)
수신부는 성능 최적화를 위해 두 개의 독립된 프로세스/스레드로 분리되어 특정 CPU 코어를 독점 사용합니다.

3.1. 네트워크 수신 (jetson_rx)
하드웨어 처리: 수신 NIC는 DMA(Direct Memory Access)를 사용하여 CPU 자원 개입 없이 4MB 크기의 커널 버퍼에 데이터를 직접 기록합니다.

CPU 할당: CPU Core 2번을 독점(고정)하여 컨텍스트 스위칭 오버헤드를 방지합니다.

수신 버퍼: 수신된 패킷은 frag_queue라는 1024개 패킷 용량의 환형 버퍼(Circular Buffer)에 저장됩니다.

3.2. 데이터 재조립 (reassembly_shm)
CPU 할당: CPU Core 3번을 독점(고정)합니다.

데이터 접근: frag_queue에 저장된 실제 데이터에 포인터로 접근하여 불필요한 메모리 복사(Memory Copy)를 방지합니다.

조립 슬롯 (reasm_slots): 4개의 슬롯(reasm_slots1 ~ reasm_slots4)을 병렬로 운용하여 단편화된 패킷을 원래의 프레임으로 병합합니다.

예외 처리: 각 슬롯은 데이터 조립 완성까지 100ms의 Timeout을 가집니다. 기한 내 조립되지 않으면 해당 프레임은 폐기 처리될 것으로 예상됩니다.

4. 데이터 공유 및 소비 계층 (IPC)
공유 메모리 (dev/shm/robot_bridge_N): 재조립이 완료된 데이터는 메모리 복사 없이 패킷 포인터 형태로 공유 메모리 영역에 전달됩니다.

동기화 구조 (트리플 버퍼): 데이터 Read/Write 충돌과 Tearing 현상을 방지하기 위해 트리플 버퍼링을 적용했습니다.

이미지 write buffer: reassembly_shm이 조립된 데이터를 기록하는 영역

최신 이미지 buffer (ready): 기록이 완료되어 대기 중인 최신 프레임 영역

이미지 read buffer: Qt 애플리케이션이 현재 읽어가고 있는 영역

애플리케이션 (Qt): 공유 메모리의 Read 버퍼를 통해 최종적으로 데이터를 소비하여 화면에 렌더링하거나 후처리합니다.
