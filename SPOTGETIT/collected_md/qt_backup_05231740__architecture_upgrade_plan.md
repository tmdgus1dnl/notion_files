# 로봇 관제 시스템 아키텍처 논의 및 차세대 업그레이드 계획서

이 문서는 Qt 기반 로봇 관제 애플리케이션(`disaster_control_qt`)과 미들웨어(`bridge_app`)의 아키텍처를 분석하며 논의한 핵심 소프트웨어 공학적 인사이트와, 이를 바탕으로 한 차세대 성능 개선 계획을 정리한 문서입니다.

---

## 1. 아키텍처 및 시스템 공학 논의 요약 (대화 기록)

### 1.1 추상화(Abstraction)와 은닉의 진짜 이유
* **문제 제기**: "헤더 파일에서 `SharedData`를 숨기고 `void *`로 감추는 게 보안 때문인가? 어차피 개발자가 소스 코드를 다 가지고 있는데 왜 감추지?"
* **결론**: 현업에서의 추상화는 보안이 아니라 **'관심사의 분리(Separation of Concerns)'와 '결합도(Coupling) 낮추기'**를 위함. 
* **효과**: UI 개발자는 복잡한 공유 메모리(POSIX) 로직을 몰라도 함수만 호출하면 되며, 추후 통신 방식이 웹소켓 등으로 바뀌더라도 UI 코드는 단 한 줄도 수정할 필요가 없게 됨. 또한 빌드(컴파일) 시간을 획기적으로 단축시킴.

### 1.2 진정한 미들웨어와 SDK의 구조
* 현 구조는 논리적으로만 분리되어 있을 뿐 한 프로젝트 내에 공존함.
* 진정한 미들웨어 구조가 되려면 Bridge 팀이 통신 모듈을 **공유 라이브러리(`librobot_comm.so`)**와 **껍데기 헤더 파일(`robot_connection.h`)** 형태로만 제공해야 함. (Android SDK나 VSCode API처럼 동작)

### 1.3 바이너리와 링커의 마법 (기계어가 실행되는 원리)
* 소스 코드를 숨기고 기계어 덩어리(`.so`)만 주어도 실행이 가능한 이유는 **'심볼 테이블(Symbol Table)'** 덕분.
* 링커(Linker)가 헤더 파일(메뉴판)을 보고 바이너리 맨 앞의 심볼 테이블(목차)을 뒤져 정확한 메모리 주소를 찾아 연결해 줌.

### 1.4 리버스 엔지니어링(역공학)과 방어 기법
* 기계어(0과 1)는 CPU 명령어와 1:1 매칭되므로 어셈블리어로는 100% 해독 가능함.
* 하지만 변수명, 클래스 구조 등 사람이 읽을 수 있는 정보는 컴파일 단계에서 **영구적으로 증발**하며, 컴파일러의 최적화(Optimization)로 인해 원본 C++ 코드로 완벽하게 복원하는 것은 불가능함. (소고기를 갈아 만든 햄버거 패티 비유)
* **방어 기법**: 심볼 제거(Stripping), 난독화(Obfuscation), 패킹(Packing), 안티 디버깅 등.

### 1.5 TCMS (Train Control and Monitoring System)와의 유사성
* 현재 설계 중인 로봇 관제 아키텍처는 현대 철도의 거대 산업 시스템인 **TCMS**와 완벽하게 동일한 사상을 공유함.
* **Qt 앱 = TCMS HMI**: 기관사가 보는 터치스크린 모니터 (모니터링 및 상위 명령)
* **Bridge 앱 = TCMS VCU**: 하드웨어와 직접 통신하고 데이터를 정제하는 중앙 제어 미들웨어. (로우 레벨 PID/PWM 모터 제어 코드는 TCMS 밖인 말단 하드웨어 MCU에 위치함)

### 1.6 극한의 방어적 프로그래밍 (Fallback 로직)
* 상태(State) 자물쇠 획득 실패 시 위치(Odom) 자물쇠로 Fallback 하는 이유:
* 브릿지 앱의 일부 스레드가 죽거나 자물쇠가 꼬이는 부분 장애(Partial Failure)가 발생하더라도, **모니터링 화면(Qt 앱)이 같이 강제 종료되는 것을 막고 최후의 순간까지 로봇의 생존 데이터(위치)를 화면에 뿌려주기 위한 생명줄** 역할.

### 1.7 큐(Queue)의 VIP 새치기 정책 (Drop Policy)
* 네트워크가 막혀 큐가 가득 찬 비정상 상태일지라도, 긴급 정지(ESTOP) 같은 **생명과 직결된 `CRITICAL` 명령이 들어오면 기존의 잉여 명령을 찾아 쓰레기통에 버리고 빈자리에 새치기해서 꽂아 넣음.** 무시무시하고도 완벽한 안전장치(Fail-Safe).

## 2. 성능 최적화: Polling vs Semaphore 장단점 비교

현재 Qt 앱은 100ms마다 알람을 맞추고 상태를 수거하는 **폴링(Polling)** 방식으로 동작하여 10 FPS의 제한이 존재함. 이를 **세마포어(Semaphore) 기반 이벤트 구동 방식**으로 바꿀 때의 트레이드오프.

| 항목 | 현재: 100ms Polling | 개선: Semaphore + QThread |
|---|---|---|
| **동작 원리** | 0.1초마다 직접 가서 찔러보기 | 새 데이터가 올 때까지 스레드를 재우고 대기 |
| **장점** | 구조가 단순함. 데이터 폭주 시에도 10FPS로 자체 스로틀링되어 화면(UI) 마비 절대 방어. | 지연 시간(Latency) 0초. 30FPS 이상 극한의 부드러움. CPU 헛돌기 방지. |
| **단점** | 미세한 반응 지연 존재. 라이다 등 빠른 센서의 부드러움을 표현하기 어려움. | 렌더링 횟수 제한(Throttle) 코드가 없으면 라이다 폭주 시 그래픽카드 부하로 PC 먹통 위험. 멀티스레드 동기화 복잡도 증가. |

## 3. 다음 단계: Qt 공유메모리 아키텍처 개선 상세 계획

라이다와 영상의 버벅거림을 없애기 위해, 기존 100ms 폴링의 굴레를 벗어던지고 **QThread와 `sem_timedwait`을 결합한 하이브리드 아키텍처**로 코드를 수정합니다.

### 💡 핵심 설계 철학
1. **Zero-Latency 수집**: 브릿지 앱이 `img_sem` 종을 치는 즉시 스레드가 깨어나 데이터를 긁어옴.
2. **UI 렌더링 방파제(Throttle)**: 스레드가 데이터를 1초에 1,000번 가져오더라도, 메인 UI 스레드는 딱 30FPS(33ms) 속도로만 화면을 갱신하도록 분리하여 PC 먹통을 방지함.
3. **Deadlock 방어**: 세마포어만 믿고 영원히 자는 것이 아니라 `sem_timedwait`으로 최대 100ms만 기다리도록 안전장치 설정.

### 📝 수정 대상 파일 및 변경 사항

#### 1. `robotshmconnection.h` & `.cpp`
- **추가**: `bool waitForImage(int timeoutMs);` 함수. 리눅스 `sem_timedwait`을 사용하여 지정된 시간 동안 `&shm->img_sem` 세마포어가 울리기를 기다림.

#### 2. `robotworkerthread.h` & `.cpp` [신규 생성]
- `QThread`를 상속받은 전담 백그라운드 작업자 클래스.
- 로봇 1대당 1개의 `RobotShmConnection`을 소유함.
- **무한 루프 로직**: `waitForImage(100ms)` 대기 ➡️ `connection->poll()`로 데이터 추출 ➡️ `snapshotReady(snapshot)` 시그널로 UI 스레드에 비동기 송출.

#### 3. `shmmonitor.h` & `.cpp`
- **구조 변경**: `RobotShmConnection` 관리자에서 `RobotWorkerThread` 관리자로 승격.
- **방파제 로직 추가**: 백그라운드 스레드에서 날아온 데이터는 `m_snapshots` 배열에 조용히 덮어쓰기만 함. 메인 스레드에 `m_uiTimer(33ms)`를 새로 만들어 33ms마다 변경 사항이 있을 때만 딱 한 번 `snapshotsUpdated` 시그널을 방송함.

#### 4. `CMakeLists.txt`
- 신규 파일 빌드 타겟 등록.

방향성은 100% 맞습니다! 하지만 제가 현재 작성된 코드를 꼼꼼히 검증(Verify) 해보니, 단순히 헤더의 글자만 바꾸는 것으로 끝나지는 않고 내부 구현부(robotshmconnection.cpp)에서 딱 2가지 핵심적인 부분을 손봐야 완벽한 순수 C++ 범용 SDK가 될 수 있습니다.

🔍 검증 결과: SDK 분리를 위해 고쳐야 할 2가지
1. 이미지 디코딩의 '책임' 넘기기 (QImage 분리) 현재 SDK 역할을 하는 robotshmconnection.cpp 내부를 보면 다음과 같이 짜여 있습니다.

cpp
// 현재: SDK 내부에서 친절하게 JPEG를 풀어서 QImage 객체로 만들어서 줌
QImage image = QImage::fromData(slot.data, slot.size, "JPEG");
snapshot.image = image;
만약 순수 범용 SDK로 만든다면 SDK 내부에서는 Qt 라이브러리를 쓸 수 없습니다. 따라서 SDK는 "나는 공유 메모리에서 압축된 JPEG 날것의 바이트(Raw Byte)만 퍼서 줄게. 이걸 풀어서 화면에 그리는 건 가져다 쓰는 앱(Qt든 Unity든 Python이든)이 알아서 해라!" 라는 구조로 책임을 넘겨야 합니다.

변경 후: std::vector<uint8_t> jpeg_data; 에 원본 바이트만 담아서 리턴.
2. 문자열 처리 유틸리티 함수 교체 내부 C++ 코드에서 문자열 처리를 위해 Qt의 편리한 기능들을 쓰고 있는데, 이를 순수 표준 C++로 낮춰야 합니다.

QString("/robot_bridge_%1").arg(m_robotId) ➡️ std::to_string()이나 snprintf로 변경
QString::fromUtf8(state.fault_text) ➡️ std::string 변환으로 변경
💡 아키텍처 결론 (ABI 호환성 개념)
위 2가지만 수정하면(QImage ➡️ 원본 Byte, QString ➡️ std::string) 당장 C++를 지원하는 어떤 환경에서든 쓸 수 있는 훌륭한 범용 SDK가 됩니다.

(참고로 현업 구글/아마존 같은 곳에서 전 세계 배포용 '끝판왕 범용 SDK'를 만들 때는, 아예 std::string 같은 C++ 전용 객체조차 쓰지 않고 '순수 C 언어' 자료형(char*, uint8_t*)으로만 최종 껍데기를 만듭니다. 컴파일러 버전이 다르면 std::string 메모리 크기가 달라져서 앱이 뻗어버리는 'C++ ABI 호환성 문제'를 피하기 위해서입니다.)

결론적으로, **"헤더의 자료형을 표준으로 바꾸고, 내부에서 QImage로 친절하게 변환해주던 로직만 제거하면 완벽한 범용 SDK로 독립시킬 수 있다"**는 질문자님의 통찰은 기술적으로 완벽하게 검증되었습니다!
