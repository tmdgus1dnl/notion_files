# Bridge App 데이터 흐름 및 메모리 처리 상세 분석 보고서
**(Kernel ➔ Application ➔ IPC Shared Memory)**

본 보고서는 Jetson으로부터 전송된 센서 데이터(UDP 패킷)가 Raspberry Pi 5의 네트워크 인터페이스(NIC) 커널 버퍼에 도착한 순간부터, 사용자 프로세스(Qt App)가 읽어갈 수 있도록 공유 메모리(SHM)에 최종 안착하기까지의 **모든 동작과 메모리 복사(Copy) 과정**을 현재 구현된 소스 코드(`bridge_app`)를 기준으로 상세히 분석합니다.

---

## 1. 커널 공간 ➔ 수신 스레드 (`jetson_rx.c`)

네트워크 인터페이스(NIC)를 통해 들어온 데이터는 Linux 커널의 UDP 수신 버퍼에 쌓입니다. `jetson_rx` 스레드는 이 커널 버퍼에서 데이터를 사용자 공간(Application Space)으로 가져옵니다.

1. **소켓 수신 대기**: `jetson_rx` 스레드는 `recvfrom()` 시스템 콜을 호출하여 커널 버퍼에 패킷이 도착하기를 기다립니다.
2. **커널 ➔ 로컬 버퍼 복사 (Copy 1)**: 패킷이 도착하면 커널 공간의 버퍼에서 `jetson_rx` 스레드 내부에 선언된 정적 배열인 `static uint8_t pkt[PROTO_PKT_MAX]`로 데이터가 복사됩니다.
3. **헤더 분석**: 복사된 패킷의 헤더(`PktHeader`)를 분석하여 대상 로봇의 ID(`robot_id`)와 데이터 타입(Image, LiDAR, Odom)을 판별합니다.
4. **Odom 처리**: 데이터가 크기가 작은 Odom인 경우, 큐를 거치지 않고 해당 로봇의 공유 메모리(`SHM`) 영역에 `pthread_rwlock`을 걸고 **직접 기록**합니다.

---

## 2. 수신 스레드 ➔ 패킷 큐 (`frag_queue.h`)

데이터가 여러 조각으로 나뉘어 들어오는 대용량 데이터(Image, LiDAR)인 경우, 조립을 위해 큐(`FragQueue`)로 넘겨집니다. 여기서 스레드 간의 안전한 데이터 교환을 위해 복사가 발생합니다.

1. **큐 진입 시도**: `jetson_rx`는 `frag_queue_push()`를 호출합니다. 이때 다중 스레드 충돌을 막기 위해 큐의 `pthread_mutex_lock`을 획득합니다.
2. **로컬 버퍼 ➔ 큐 엔트리 복사 (Copy 2)**: 큐에 빈 공간이 있다면(또는 가장 오래된 것을 버리고), 방금 커널에서 받은 `pkt` 배열의 내용(헤더+페이로드)을 큐 내부의 `FragEntry.buf`로 `__builtin_memcpy`를 사용하여 복사합니다.
3. **신호 전달**: 복사가 끝나면 큐의 포인터(tail)를 증가시키고, 조립 스레드를 깨우기 위해 `pthread_cond_signal()`을 호출한 뒤 Mutex를 해제합니다.
   * **[설계 의도]**: 여기서 포인터만 넘기지 않고 복사를 수행함으로써, `jetson_rx` 스레드는 즉시 자신의 `pkt` 버퍼를 재사용하여 다음 커널 패킷을 지연 없이 받을 수 있게 됩니다. (스레드 간 생명주기 분리)

---

## 3. 패킷 큐 ➔ 재조립 스레드 (`reassembly_shm.c`)

수신 스레드에 의해 큐에 조각(Fragment)이 쌓이면, 이를 꺼내어 완전한 프레임으로 합치는 과정이 진행됩니다.

1. **큐 대기 및 추출**: `reassembly_shm_thread`는 큐가 빌 경우 `pthread_cond_wait()`으로 잠들어 있다가 신호를 받고 깨어납니다.
2. **큐 엔트리 ➔ 로컬 버퍼 복사 (Copy 3)**: `frag_queue_pop()`을 호출하여 Mutex를 잡은 뒤, 큐 내부의 데이터를 조립 스레드 전용 버퍼인 `static uint8_t pkt[PROTO_PKT_MAX]`로 `__builtin_memcpy`를 통해 복사해옵니다. 복사 후 큐의 공간은 즉시 해제(Head 이동)됩니다.
3. **조립 슬롯 탐색**: 패킷 헤더의 `frame_id`를 확인하여 8개의 병렬 조립 슬롯(`ReasmSlot`) 중 작업 중인 슬롯을 찾거나 새로 할당받습니다. 이때 타임아웃(300ms)이 지난 오래된 미완성 슬롯은 폐기(가비지 컬렉션)됩니다.
4. **로컬 버퍼 ➔ 조립 슬롯 복사 (Copy 4)**: 패킷이 중복 도착한 것이 아님을 확인한 후, 헤더에 적힌 `payload_offset` 위치를 계산합니다. 버퍼 오버플로우 검사를 통과하면, `pkt`에 있는 페이로드(순수 데이터)를 `ReasmSlot->buf`의 정확한 오프셋 위치로 `memcpy`를 이용해 이어 붙입니다.

---

## 4. 조립 완료 ➔ IPC 공유 메모리 (`reassembly_shm.c` ➔ `shm_def.h`)

모든 조각이 모여 하나의 완전한 프레임(이미지 또는 3D 포인트 클라우드)이 완성되면, 이를 Qt App이 읽을 수 있도록 공유 메모리(SHM)로 옮깁니다.

1. **완성 감지**: 현재 프레임에 대해 수신된 조각 수(`frags_recv`)가 전체 조각 수(`frags_total`)와 같아지면 조립 완료로 판단합니다.
2. **Triple Buffer 슬롯 계산**: 락(Lock) 없이 동기화하기 위해 `atomic_load(&shm->img_ready_idx)`를 통해 현재 Qt App이 가장 최근에 읽도록 지정된 슬롯 인덱스를 가져오고, 그 **다음 슬롯**을 계산하여 빈 슬롯을 지정합니다.
3. **조립 슬롯 ➔ 공유 메모리 슬롯 복사 (Copy 5)**: `ReasmSlot->buf`에 완성되어 있는 통짜 데이터 블록(수십~수백 KB)을 방금 계산한 SHM의 빈 슬롯(`ImgSlot` 또는 `LidarSlot`) 내부로 한 번에 `memcpy` 합니다.
4. **상태 업데이트 및 알림 (Zero-copy IPC 시작점)**: 
   * 복사가 완료되면 `atomic_store(&shm->img_ready_idx, slot)`을 호출하여 "이 슬롯에 최신 데이터가 준비됨"을 원자적으로 기록합니다.
   * `sem_post(&shm->img_sem)`을 호출하여 Qt App 프로세스에 데이터 도착 알림을 보냅니다.
   * **[결과]**: Qt App은 이 SHM 슬롯에 접근할 때 **자신의 메모리 공간으로 추가 복사할 필요 없이(Zero-copy Read)** SHM 포인터를 그대로 OpenCV나 렌더러에 넘겨 화면에 그리게 됩니다.

---

## 종합 요약 (메모리 접근 패러다임)

현재 아키텍처는 커널에서 데이터를 꺼내어 SHM에 안착시키기까지 **총 5회의 메모리 복사(Copy)**를 수행합니다.

`Kernel Buffer` ➔ (1) ➔ `rx_thread_buf` ➔ (2) ➔ `FragQueue` ➔ (3) ➔ `reasm_thread_buf` ➔ (4) ➔ `ReasmSlot` ➔ (5) ➔ `Shared Memory`

*   **비효율성의 존재**: 동일한 데이터를 여러 버퍼를 거치며 반복 복사하므로 극한의 Zero-copy 아키텍처에 비하면 CPU 메모리 대역폭을 더 소모하는 것은 명백한 사실입니다.
*   **설계의 당위성**: 하지만 위와 같은 단계적 격리(복사)를 통해 **① 커널 소켓 읽기 차단 방지(Non-blocking RX), ② 스레드 간 메모리 소유권(Lifecycle)의 명확한 분리, ③ 패킷 유실 및 순서 뒤바뀜 환경에서의 안전한 메모리 병합, ④ 버퍼 오버플로우 등 보안 위협 차단**이라는 막대한 안정성 이점을 얻었습니다.
*   **성능적 타협**: RPi 5 수준의 최신 하드웨어에서는 10MB/s 급의 데이터를 몇 번 복사하는 오버헤드가 마이크로초 단위에 불과하여 전체 지연 시간(Latency)에 거의 영향을 주지 않으므로, 성능보다 **멀티 로봇 환경에서의 안정성과 크래시(Crash) 방지를 우선시한 합리적 구조**로 판단됩니다.
