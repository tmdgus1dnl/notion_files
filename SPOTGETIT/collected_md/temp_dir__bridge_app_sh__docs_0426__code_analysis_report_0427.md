## 2. 주요 취약점 상세 분석 및 조치 내역

### 2.1 시그널 처리 및 데드락 취약점 (수정됨)

**문제점 심층 분석:**
프로그램 종료를 위해 `SIGINT(Ctrl+C)` 수신 시 `sig_handler`가 동작합니다. 이전 코드에서는 시그널 핸들러 내부에서 `frag_queue_stop()`을 호출했습니다. 문제는 `frag_queue_stop`이 내부적으로 `pthread_mutex_lock`을 잠근다는 것입니다. 만약 메인 루프나 워커 스레드가 큐에 삽입/추출을 위해 이미 `mutex`를 잡은 상황에서 인터럽트가 걸렸다면 시스템은 영원한 데드락(Deadlock)에 빠지게 됩니다. 
또한 시그널 핸들러 내에서 `fprintf`와 같은 버퍼링 기반 표준 I/O 함수는 비동기 시그널에 안전(Async-Signal-Safe)하지 않습니다.

**해결 방안 및 조치:**
* `bridge_main.c`의 `sig_handler` 루틴 내부에서는 글로벌 플래그(`g_ctx.stop = 1`)만을 세팅하도록 간소화했습니다.
* `fprintf` 대신 Async-Signal-Safe한 POSIX 시스템 콜 `write(STDERR_FILENO, ...)`로 교체했습니다.
* 큐 정지를 브로드캐스트하는 `frag_queue_stop` 호출부는 시그널 환경을 완전히 탈출한 직후, 메인 스레드의 `watchdog_loop` 종료 직후 안전한 동기(Synchronous) 블록으로 위치를 이동시켰습니다.

### 2.2 Reassembly_SHM: Triple Buffer 및 잔류 데이터 누수 방어 (수정됨)

**문제점 심층 분석:**
1. **Triple Buffer 버그:** `shm_ready_idx`를 확인하고 다음 슬롯을 분배하는 과정에서 인덱스가 로테이션하지 않고 0과 1번으로만 수렴하는 버그가 있었습니다.
2. **잔류 데이터 (Ghost Data) 현상:** 10,000 바이트로 전송된 JPEG 데이터가 UDP 유실로 인해 9,000 바이트만 도착하여 버퍼에 군데 군데 빈 공간(Hole)을 만들게 되는 상황에서, 빈 공간에는 직전에 사용했던 오래된 프레임이 남아있게 됩니다.

**해결 방안 및 조치:**
* 인덱스 할당 시 `(ready + 1) % IMG_SLOTS` 방식을 적용하여 0->1->2 순서로 완벽하게 순환하도록 개선했습니다.
* 조립 버퍼가 신규 할당될 때 `memset(rs->buf, 0, REASM_BUF_MAX);`를 수행하여 이전 프레임의 모든 흔적을 깨끗하게 소거하도록 방어 코드를 직접 구현했습니다.

### 2.3 메모리 및 자원 초기화/누수 구조체 재검증 (오류 없음 확정)

**점검 내역:**
이전의 1차 분석 때 지적되었던 스레드 초기화 에러 발생 시 자원 누수 부분의 맥락을 재확인했습니다. 
과정상, `create_thread`로 워커들을 가동하기 **전**에 이미 `for` 루프에서 `shm_init_one`으로 로봇 `num_robots` 수만큼의 모든 IPC 메모리 맵과 `sema`들을 무결하게 확정지어 둔 사실 확인 완료되었습니다. 에러 발생 시 진행되는 `cleanup_all(num_robots)`는 정확하게 모든 맵을 순회파괴하도록 구성되어 있어서 **메모리 누수가 전혀 존재하지 않는 깔끔한 설계**임이 재차 입증되었습니다.

## 3. 총평 (Architecture Conclusion)

전체적인 BridgeDaemon 솔루션 코드 오디팅이 성공적으로 끝났습니다.
- **아키텍처 수준:** `FragQueue` 조건 변수를 통한 블로킹-웨이트의 우수성, Lidar 오버플로우를 막는 bounds check 로직 등 기존 소스의 네트워크 방어 기반 설계가 튼튼합니다.
- **안정성:** 개발자가 놓치기 쉬운 C 언어 고유의 치명적 `Concurrent UB` 및 시스템 `Signal` 데드락 문제들을 말끔히 걸러낸 상태입니다. 
- **결론:** 현재 소스파일들은 임베디드 프로덕션 등급에 맞춰 배포(Build) 및 실가동되어도 손색이 없는 동시성(Concurrency)과 성능을 자랑합니다.
