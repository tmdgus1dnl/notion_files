# 수동 자동 cmd_vel

지금 구조는 현재 상태값을 검사하는 방식이 아니라, 들어온 cmd_type에 따라 바로 publish하는 방식입니다.

현재 동작은 이렇습니다.

노드 시작
-> /control/drive_mode = AUTO publish

cmd_type 0x03 MOVE 수신
-> /control/drive_mode = AUTO publish
-> /control/behavior/mode = CLASSIC publish
-> cmd_vel publish 안 함

cmd_type 0x0A MANUAL_MOVE 수신
-> /control/drive_mode = MANUAL publish
-> /control/behavior/mode = CLASSIC publish
-> /control/cmd_vel/spot_01 publish

cmd_type 0x0B SET_AUTO 수신
-> /control/drive_mode = AUTO publish
-> cmd_vel publish 안 함

cmd_type 0x02 STOP 수신
-> zero cmd_vel publish
-> drive_mode 변경 없음

그래서 만약 현재 /control/drive_mode가 AUTO인 상태여도, 라파에서 실수로 MANUAL_MOVE(0x0A) 패킷을 보내면 cmd_receiver는:

/control/drive_mode = MANUAL
/control/behavior/mode = CLASSIC
/control/cmd_vel/spot_01 = Twist(...)

를 publish합니다.

즉 구조상 “AUTO라서 수동 publish가 불가능”한 게 아니라, MANUAL_MOVE 패킷을 받는 순간 MANUAL로 전환하고 publish하는 구조입니다.

결론: 현재 안전성은 라파가 AUTO 상태에서 MANUAL_MOVE를 보내지 않는다는 프로토콜 약속에 의존합니다. Jetson 쪽에서 엄격히 막으려면
cmd_receiver 내부에 현재 drive_mode 상태를 저장하고, 별도 enable 조건이 있을 때만 MANUAL_MOVE를 허용하는 guard가 필요합니다.
