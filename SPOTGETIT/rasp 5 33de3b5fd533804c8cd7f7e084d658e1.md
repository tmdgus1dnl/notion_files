# rasp 5

### 1. 역할

- **관제 단 (Raspberry Pi):** 무거운 ROS2 환경 대신 가벼운 
Qt GUI 애플리케이션만 구동하여 시스템 부하를 최소화합니다.

### 2. 맵 데이터 전송 최적화 (탐험 맵 대응)

맵이 계속 확장되므로 `OccupancyGrid` 전체를 매번 보내는 것은 틀린 접근입니다. 대신 다음 두 가지 중 하나의 방식을 채택하여 **변경점만 전송**해야 합니다.

- **방식 A (차분 업데이트):** 이전 프레임과 비교해 값이 변경된 픽셀의 `[X, Y, 상태값]`만 추출하여 전송합니다. Qt에서는 `QImage`에 해당 픽셀만 점진적으로 업데이트합니다.
- **방식 B (타일 분할):** 전체 맵을 일정 크기의 격자(Tile)로 쪼개고, 갱신이 일어난 타일 블록만 전송합니다. Qt에서는 `QGraphicsScene`을 이용해 타일을 이어 붙입니다.

### 3. 무선 네트워크 통신 프로토콜

탐험 환경의 불안정한 Wi-Fi 대역폭에서 ROS2 기본 DDS를 사용하는 것은 비효율적입니다.

- **구현 방향:** 젯슨 쪽에 ROS2 토픽을 구독하는 C++ 브릿지 노드를 두고, 맵의 변경 데이터만 가벼운 바이너리 패킷으로 직렬화(Serialization)합니다.
- **통신 구조:** 성능 병목을 없애기 위해 epoll 기반의 I/O Multiplexing을 적용한 Non-blocking 소켓 서버(TCP 또는 UDP)를 직접 구현하여 라즈베리파이의 Qt Socket으로 데이터를 Push하는 방식이 가장 확실합니다.

- qt앱 - 통신모듈 분리
    
    ### 1. GUI 이벤트 루프 블로킹 (화면 멈춤)
    
    Qt는 메인 스레드에서 이벤트 루프(Event Loop)를 돌며 UI를 업데이트합니다. 통신 모듈에서 패킷을 수신 대기하거나 데이터를 파싱하는 데 시간이 소요되면, Qt 메인 스레드가 블로킹되어 화면 전체가 멈춥니다(프리즈 현상).
    예를 들어 탐색 로봇이 이동하며 실시간으로 맵을 동적으로 그려내야 하는 상황에서, 네트워크 지연이나 대용량 데이터 수신이 발생하면 지도가 업데이트되지 않고 렌더링 프레임이 심하게 끊기게 됩니다. 이를 피하기 위해 Qt 내에서 `QThread`로 통신부를 분리할 수도 있지만, 코드가 복잡하게 꼬이기 쉽습니다.
    
    ### 2. 크래시(Crash) 전파 및 안정성 저하
    
    네트워크 통신 중에는 패킷 손실, 예외 처리 누락, 버퍼 오버플로우 등 예기치 못한 에러로 프로세스가 죽는(Segmentation Fault) 상황이 발생하기 쉽습니다.
    통신 코드가 Qt 앱 안에 종속되어 있으면,
    통신 오류 하나 때문에 열심히 지도를 그리던 GUI 프로그램 전체가 뻗어버립니다.
    반면 통신 모듈을 도커(Docker)로 완전히 격리해 두면, 통신 모듈이 죽더라도 컨테이너만 재시작하면 되므로 GUI 프로세스는 타격을 받지 않고 생존합니다.
    
    ### 3. CPU 자원 격리 및 고성능 I/O 최적화 불가
    
    `epoll`을 활용한 I/O 멀티플렉싱이나 스레드 풀링 등 고성능 네트워크 처리를 구현할 때, 단일 프로세스 안에서 GUI 렌더링과 섞이게 되면 컨텍스트 스위칭(Context Switching) 오버헤드가 커집니다.
    프로세스를 분리하면 통신 전담 백엔드 컨테이너에는 `sched_setaffinity` 등을 이용해 특정 CPU 코어를 전담 할당하고,
    남은 코어는 Qt GUI 렌더링에만 집중하도록 시스템 리소스를 완벽하게 통제하고 최적화할 수 있습니다.
