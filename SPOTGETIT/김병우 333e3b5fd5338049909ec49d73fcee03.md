# 김병우

## 무인 플랫폼 하위 안전 제어기 ECU 플랫폼 + 관제 시스템으로 시각화

# 1. 지금 우리가 잡은 큰 방향

너희 팀 프로젝트의 핵심은

**“RTOS를 사용한 임베디드 SW 취업용 프로젝트”**이고,

도메인은 **방산 + 자동차**로 좁혀졌어.

그래서 프로젝트도 단순히 센서 붙여서 동작시키는 게 아니라,

아래처럼 읽혀야 한다고 정리됐어.

## 목표 방향

- MCU 중심 임베디드 SW
- RTOS의 존재 이유가 분명한 시스템
- 센서 → 판단 → 제어 → 안전 전이 → 진단/복구 흐름 포함
- 자동차/방산 직무로 번역 가능
- 면접에서 구조적으로 설명 가능

즉,

**“무인체 전체를 만드는 프로젝트”가 아니라**

**“무인 플랫폼의 하위 RTOS 안전 제어기 ECU를 만드는 프로젝트”** 쪽으로 방향이 잡혔어.

---

# 2. 왜 이런 방향으로 잡았는가

처음에는 RTOS를 축으로 주제를 잡으려고 했는데,

그 자체로는 아이디어가 잘 안 떠오른다는 문제가 있었지.

그래서 발상을 바꿨다.

## 잘못된 출발

- RTOS를 쓰고 싶은데 무슨 주제를 하지?

## 바꾼 출발

- RTOS가 자연스럽게 필요한 시스템이 뭐지?

그 결과, RTOS가 잘 맞는 시스템은 이런 특징이 있어야 한다고 정리했어.

- 여러 태스크가 동시에 동작
- 제어 주기가 중요
- 센서/통신/진단/로깅이 병렬로 돌아감
- 비동기 이벤트가 존재
- fault 발생 시 안전 상태로 전이해야 함

이 기준으로 봤을 때,

**무인 플랫폼 하위 제어기 / 안전 제어 ECU**가 가장 적합하다고 본 거야.

---

# 3. 최종적으로 가장 유력해진 주제

## 주제명

**RTOS 기반 무인 플랫폼 하위 안전 제어기 ECU 플랫폼**

또는 조금 더 풀면,

**상위 계층으로부터 명령을 받아, 다중 센서 기반 상태를 주기적으로 수집·판단하고, 제어 출력 및 안전 전이를 수행하는 RTOS 기반 무인 플랫폼 하위 제어기**

이게 지금까지 가장 설득력 있는 방향으로 정리됐어.

---

# 4. 이 주제가 뜻하는 바

여기서 중요한 건

이 프로젝트가 **UGV 전체**, **UAV 전체**를 만드는 게 아니라는 거야.

## 정확한 의미

이 프로젝트는

**무인체 내부에서 공통적으로 필요한 하위 제어·안전·진단 기능을 담당하는 실시간 제어기 플랫폼**을 만드는 거야.

즉, 핵심은 아래다.

- 상위에서 명령을 받음
- 센서를 읽음
- 상태를 판단함
- 액추에이터를 제어함
- 위험하면 안전 상태로 감
- fault를 감지하고 복구를 시도함
- 로그와 상태를 관리함

그래서 이건 “기체 제작”보다는

**“무인체 안에 들어가는 공통 ECU 플랫폼 제작”**에 가깝다.

---

# 5. 그럼 UGV/UAV에 다 범용 가능한가?

이 부분은 아주 중요한데, 답은 이렇게 정리됐어.

## 정확한 답

**완전히 100% 그대로 꽂아 쓰는 수준은 아니지만, 공통 프레임워크 수준에서는 범용 가능하다.**

즉, 두 부분으로 나뉜다.

## 공통부

- RTOS task 구조
- command manager
- sensor manager framework
- fault manager
- watchdog
- logger
- communication timeout handling
- safety FSM

## 특화부

### UGV 특화

- wheel speed control
- steering control
- obstacle stop logic

### UAV 특화

- motor thrust output
- attitude stabilization input
- arming/disarming
- hover/landing fail-safe

즉,

**공통 플랫폼은 재사용 가능하고**,

**센서/액추에이터/제어 로직은 adapter 형태로 바뀐다**

이게 가장 정확한 정리야.

# 6. planning까지 직접 만들어야 하냐?

이 부분도 정리했지.

## 결론

**아니. planning까지 만들 필요 없다.**

왜냐하면 이 프로젝트의 초점은

**상위 알고리즘**이 아니라

**하위 RTOS 안전 제어기 ECU**이기 때문이야.

그래서 상위 계층은 이렇게 추상화하면 된다고 정리했어.

## 상위 계층 대체 방식

- PC command generator
- Python 스크립트
- 조이스틱/RC 입력
- 간단한 supervisor 앱
- 상위 노드에서 speed/steer/mode command 송신

즉,

**상위 planning은 명령 생성기/감독기로 대체**하고,

하위 제어기 ECU가 그 명령을 안전하게 실행하는 구조로 가는 게 맞다고 했어.

이게 오히려 더 현업스럽고,

RTOS 제어기 자체의 가치가 더 선명하게 드러난다고 정리됐어.

# 7. RTOS 내부 설계 요소가 이 주제에 담기느냐?

이게 네가 제일 중요하게 본 질문이었지.

결론은 이렇게 정리됐어.

## 결론

**응, 충분히 담을 수 있다.**

하지만 **주제만 잡는다고 자동으로 드러나는 건 아니고**,

프로젝트를 **RTOS 설계·검증 프로젝트로 정의해야 한다.**

즉, 단순히

- sensor task
- control task
- comm task
- logger task

이렇게만 나누면

그냥 “기능 태스크 분리 프로젝트”처럼 보일 수 있어.

그래서 반드시

**RTOS 관점의 설계 포인트를 의식적으로 넣어야 한다**고 정리했어.

---

# 8. 이 주제에 녹일 수 있는 RTOS 핵심 요소들

## 1) 선점형 / 비선점형 스케줄링

- 주 설계는 **선점형**
- control task, e-stop 대응처럼 deadline 있는 작업에 적합
- 비선점형은 비교 대상으로 활용 가능

## 2) 우선순위 기반 스케줄링

- control > sensor > comm > diagnostics > logger
- 안전성, deadline, blocking 가능성 기준으로 우선순위 결정

## 3) 태스크 상태 전이

- Ready / Running / Blocked / Suspended
- event-driven task와 periodic task 혼합
- 시스템 모드에 따라 task 동작 변경 가능

## 4) 세마포어 / 뮤텍스

- semaphore: ISR-task 동기화, 이벤트 wake-up
- mutex: shared buffer, sensor snapshot, log buffer 보호
- priority inversion / inheritance까지 설명 가능

## 5) 인터럽트 관리

- E-stop ISR
- UART RX ISR
- encoder ISR
- timer ISR
- ADC complete ISR
- ISR은 최소 처리, 실제 일은 task에서 수행

## 6) 메모리 할당 및 최적화

- static task allocation
- static queue/buffer
- stack high-water mark 측정
- malloc/free 최소화
- stack overflow hook

즉, 네가 우려한 RTOS 핵심 요소들은

이 주제 안에 **전부 설계 포인트로 녹일 수 있다**고 정리됐어.

---

# ~~1위. 다중 노드 분산 제어 시스템~~

> **하나의 거대한 제어기를 만드는 게 아니라, 역할별로 나뉜 여러 개의 제어 노드가 통신하면서 하나의 시스템처럼 동작하는 구조**야.
> 

쉽게 말하면 **보드 1개짜리 임베디드 프로젝트가 아니라, 소형 ECU 네트워크**를 만드는 거다.

## 이 시스템을 현실적으로 상상하면

책상 위에 STM32 보드가 3개 있다고 생각해봐.

- **Sensor Node**
- **Master Control Node**
- **Actuator Node**

그리고 이 3개가 CAN이나 UART로 연결돼 있다.

### 각 노드의 역할

## 1) Sensor Node

이 보드는 “현장의 눈과 귀” 역할이다.

예를 들면:

- 온도센서 읽기
- 전류센서 읽기
- 거리센서 읽기
- 엔코더 읽기
- 스위치 입력 읽기

즉, 얘는 **센서를 직접 붙잡고 raw data를 수집하는 보드**다.

그다음 그냥 숫자만 보내는 게 아니라,

1차적으로 간단한 전처리도 할 수 있다.

예:

- moving average
- 정상 범위 체크
- 센서 timeout 체크
- 값 이상 여부 flag 생성

그리고 이렇게 메시지를 만든다.

```
[센서 상태 메시지]
temp = 42.1
current = 1.8A
distance = 35cm
encoder_rpm = 120
fault_sensor_timeout = 0
```

이걸 Master Node로 보낸다.

---

## 2) Master Control Node

이 보드는 시스템의 “두뇌”다.

Sensor Node가 보낸 데이터를 받아서

지금 시스템이 어떤 상태인지 판단한다.

예:

- 정상 운전
- 과열 경고
- 과전류 위험
- 장애물 근접
- 긴급 정지 필요
- 제한 출력 모드 전환

즉, **상태기계(FSM)** 와 **제어 정책**이 이 노드에 들어간다.

예를 들면 이런 판단을 한다.

- 온도 60도 이상 + 전류 상승 → 과부하 위험
- 거리 10cm 이하 → 즉시 정지
- 엔코더는 도는데 목표 속도 대비 전류가 과도함 → 기구 stuck 가능성
- heartbeat 끊김 → 해당 노드 fault 처리

그리고 최종 명령을 Actuator Node에 보낸다.

```
[제어 명령 메시지]
motor_enable = 1
target_pwm = 45
fan_enable = 1
buzzer = 0
system_mode = WARNING
```

---

## 3) Actuator Node

이 보드는 “손과 발”이다.

Master Node가 보낸 명령을 받아서 실제 출력을 만든다.

예:

- DC 모터 PWM 출력
- 서보 제어
- 팬 on/off
- 릴레이 차단
- LED / buzzer 출력

즉, 실제로 전기적 출력을 담당하는 노드다.

여기서 끝이 아니라,

Actuator Node도 자기 보호 로직을 일부 가진다.

예:

- PWM 명령이 와도 전류가 너무 높으면 local stop
- 일정 시간 command 안 오면 output off
- 긴급 정지 플래그 들어오면 무조건 safe state

즉, Master가 두뇌라면, Actuator도 최소한의 **local safety**는 가진다.

---

## 시스템 전체 흐름

한 번 실제 동작처럼 써보면 이렇다.

### 상황

모터 구동 시스템이 있고, 거리센서와 온도센서가 달려 있음.

### 흐름

1. Sensor Node가
    - 온도 68도
    - 거리 8cm
    - 전류 2.5A
    를 읽음
2. Sensor Node가 Master Node로 전송
3. Master Node가 판단
    - 거리 너무 가까움 → 정지
    - 온도 높음 → 팬 ON
    - 경고 상태 진입
4. Master Node가 Actuator Node로 명령
    - motor_pwm = 0
    - fan_enable = 1
    - buzzer = 1
5. Actuator Node가 실제로 모터 정지, 팬 동작, 경고 출력

즉, **센서 읽는 놈 / 판단하는 놈 / 출력하는 놈이 분리돼 있는데 전체적으로는 하나의 시스템처럼 움직이는 구조**다.

## 이 시스템을 현실적으로 상상하면

책상 위에 이런 게 있다.

- STM32 보드 1개
- 배터리 전압 측정 회로
- 전류센서
- 거리센서
- IMU 또는 tilt sensor 일부
- LED / buzzer / relay
- 모터 드라이버 또는 서보 출력
- UART/CAN 통신 포트

이 보드는 무인체의 상위 제어기에서 명령을 받을 수도 있고,

자체 센서를 읽어 상태를 판단할 수도 있다.

## 이 보드의 핵심 역할

크게 3개다.

### 1) 상태 감시

무인체가 지금 안전한 상태인지 계속 본다.

예:

- 배터리 저전압인가?
- 전류가 급상승하는가?
- 통신 heartbeat가 살아있는가?
- 장애물이 너무 가까운가?
- 특정 자세 이상이 발생했는가?

즉, **건강 상태를 모니터링**한다.

---

### 2) 임무 보조 제어

상위 시스템이 내려준 임무 명령을 받아서 일부 실행한다.

예:

- payload release
- 짐벌 각도 제어
- 속도 제한 모드 전환
- 임무 모드 변경
- 구동 허가 / 금지

즉, 단순 모니터링만 하는 게 아니라 **실제로 일부 제어권**도 갖고 있다.

---

### 3) failsafe / safe-state 전환

이게 핵심이다.

무인체는 정상일 때보다 비정상일 때 설계가 더 중요하다.

예:

- 배터리 저전압 → 출력 제한
- heartbeat loss → 정지 모드
- 장애물 과근접 → emergency stop
- 특정 센서 불량 → degraded mode
- 과전류 → 릴레이 차단

즉, 이 보드는 **“문제 생기면 어떻게 안전하게 죽일 것인가”** 를 담당한다

## RTOS는 여기서 왜 필요하냐

이 보드 안에서 동시에 돌아가는 기능이 있다.

- Sensor Monitor Task
- Power Monitor Task
- Link Monitor Task
- Mission Command Task
- Failsafe Task
- Telemetry Task
- Logger Task

예를 들어,

- heartbeat 감시는 50~100ms
- 배터리 감시는 20ms
- telemetry는 100ms
- 로그는 낮은 우선순위

이렇게 중요도와 주기가 다 다르다.

그래서 RTOS가 딱 맞는다.
