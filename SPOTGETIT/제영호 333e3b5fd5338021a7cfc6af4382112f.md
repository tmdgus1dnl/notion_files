    # RTScope — 팀 브리핑
    
    ## 한 줄 요약
    
    > FreeRTOS에서 태스크가 언제 어떻게 돌아가는지 **실시간으로 추적**해서 PC 터미널에 보여주는 오픈소스 펌웨어 라이브러리
    > 
    
    ---
    
    ## 이게 왜 필요한가?
    
    FreeRTOS 쓰다가 태스크가 멈추거나, 어디서 병목이 생기는지 찾으려면 런타임 트레이싱 도구가 필요하다.
    현재 시장에 있는 도구(Segger SystemView, Tracealyzer)는 **전부 유료**이고, 쓸만한 오픈소스 대안이 없다.
    
    RTScope는 이 공백을 메우는 **무료 오픈소스 트레이서**다.
    
    ---
    
    ## 어떻게 동작하나?
    
    ```
    [ STM32 + FreeRTOS ]                     [ PC ]
    
     FreeRTOS가 태스크 전환할 때
     → 자동으로 이벤트 기록        ───────→   Python 스크립트가
     → DMA+UART로 PC에 전송       (UART)     받아서 로그 출력
    ```
    
    PC 터미널 출력 예시:
    
    ```
    [    0.000ms] SWITCH_IN  Idle
    [    1.023ms] SWITCH_IN  TaskA (prio=2)
    [    1.523ms] SWITCH_OUT TaskA → Blocked
    [    1.524ms] SWITCH_IN  TaskB (prio=1)
    ```
    
    ---
    
    ## 우리가 배우는 것
    
    | 만드는 것 | 배우게 되는 것 |
    | --- | --- |
    | UART 드라이버 (레지스터) | RCC, GPIO, USART 레지스터 직접 제어 |
    | DMA 전송 | DMA 채널, 인터럽트, 데이터 파이프라인 |
    | Trace Hook | FreeRTOS 스케줄러 내부 구조 |
    | IPC 추적 | Semaphore, Mutex, Queue 동작 원리 |
    | 성능 계측 | CPU 사용률, 스택/힙 관리 |
    | 오버헤드 최적화 | 실시간 시스템 성능 튜닝 |
    
    **핵심: ST HAL을 안 쓴다.** 레지스터 레벨로 직접 구현해서 MCU 하드웨어를 깊이 이해한다.
    
    ---
    
    ## 7주 일정
    
    | 주차 | 할 일 | 결과물 |
    | --- | --- | --- |
    | **W1** | UART 레지스터 드라이버 + FreeRTOS 첫 태스크 (전원 공통) | UART 드라이버 초안 |
    | **W2** | DMA 전송 + Ring Buffer (역할 분리 시작) | 전송 파이프라인 |
    | **W3** | FreeRTOS Trace Hook 연결 + Python 디코더 | **★ 첫 동작 데모** |
    | **W4** | Semaphore/Mutex/Queue/ISR 이벤트 추적 | IPC 트레이스 |
    | **W5** | CPU 사용률, Stack/Heap 모니터링 | 계측 모듈 |
    | **W6** | 오버헤드 최적화 + 설정 시스템 + 데모앱 | 벤치마크 |
    | **W7** | 문서화 + GitHub 릴리스 + 데모 영상 | v1.0 릴리스 |
    
    W3 끝에 첫 데모가 돌아가면 성공 궤도. 일정 밀리면 W5~W6 기능을 줄인다.

    ## 역할 (6명)
    
    | 역할 | 인원 | 맡는 것 |
    | --- | --- | --- |
    | **Transport** | 1명 | DMA+UART 전송 채널 (레지스터 레벨) |
    | **Ring Buffer** | 1명 | Lock-free 순환 버퍼 |
    | **Trace Engine** | 1명 | FreeRTOS trace hook, IPC 추적 |
    | **Timestamp + ISR** | 1명 | DWT 타임스탬프, 인터럽트 추적 |
    | **Metrics** | 1명 | CPU 사용률, Stack/Heap 모니터, 이상 감지 |
    | **통합 + PL** | 1명 | 빌드, 통합 테스트, Python 디코더, 문서 |
    
    W1은 전원 같은 과제 → W2부터 분리.

    ## 기술 스택
    
    - **MCU**: STM32F4 (Cortex-M4)
    - **RTOS**: FreeRTOS
    - **드라이버**: CMSIS 레지스터 직접 제어 (ST HAL 미사용)
    - **PC 도구**: Python
    - **형상 관리**: Git + GitHub

    ## 최종 사용법 (다른 사람이 우리 라이브러리를 쓸 때)
    
    ```c
    #include "rtscope.h"
    
    RTScope_Init(USART2, 921600);
    RTScope_Start();
    // 끝. 기존 FreeRTOS 코드 수정 없이 트레이싱 시작.
    ```

    # 🔥 다중 무인 소방 로봇 시스템 — 프로젝트 제안서 v3
    
    ## 한 줄 요약
    
    > 물에 취약한 장비가 밀집한 공장 환경에서, 분산 센서가 화재를 감지하고 원격 관제가 판단·지휘하며, 소방 로봇이 **화재 지점에만 정밀 분사**하여 2차 물 피해 없이 진압하는 **자율 협업 소방 시스템**
    > 
    
    ---
    
    ## 왜 이 프로젝트인가?
    
    ### 배경 — 공장 화재와 스프링클러의 한계
    
    공장, 반도체 팹, 전자부품 창고 같은 산업 시설에서 화재가 발생하면 기존 스프링클러는 **구역 전체에 물을 무차별 분사**한다. 문제는 이런 환경에 **물에 닿으면 안 되는 것들이 가득하다**는 점이다.
    
    - 반도체 웨이퍼, PCB, 전자부품 — 물 접촉 시 즉시 폐기
    - 정밀 기계 장비 — 부식, 합선으로 수억 원대 손실
    - 화학 원자재 — 물과 반응 시 오히려 위험 확대 가능
    
    **불은 한 곳에서 났는데, 물 피해는 공장 전체로 번지는 것이 현재 구조의 근본적 문제다.**
    
    ### 해결하려는 문제
    
    - 기존 스프링클러의 **무차별 분사로 인한 2차 피해** (물 피해 > 화재 피해인 경우도 존재)
    - 넓은 공장에서 사람이 직접 초동 대응하는 것은 **사각지대 + 인명 위험**
    - 화재 규모 변화에 따라 대응 자원을 **실시간으로 유동 배분**하는 체계가 없음
    
    ### 우리의 접근
    
    - 공장 곳곳에 **분산 센서 노드**를 배치하여 상시 열 감시
    - **원격 관제 시스템**이 센서 데이터를 수신하여 화재 위치/규모 판단 → 출동 명령
    - **소방 로봇 2~3대**가 출동 → 현장에서 정밀 조준 → 화재 지점에만 분사 → 복귀
    - 화재 확산 시 관제가 **실시간으로 좌표·규모 업데이트** → 로봇 경로 수정 + 추가 출동
    - 관제관이 **LCD 모니터링 + UART CLI로 수동 개입** 가능 (자동/수동 모드 전환)

    ## 시스템 아키텍처 — 3계층 구조
    
    ```
                            [공장 내부]
    
        [센서 노드 A]     [센서 노드 B]     [센서 노드 C]
         STM32 + AMG8833   STM32 + AMG8833   STM32 + AMG8833
         (구역 A 감시)      (구역 B 감시)      (구역 C 감시)
              │                  │                  │
              └──────── nRF24L01 무선 ──────────────┘
                             │
                             ▼
                    ┌─────────────────────┐
                    │   원격 관제 시스템    │  ← 안전한 관제실
                    │   STM32 + LCD        │
                    │   + UART CLI (노트북) │
                    │                      │
                    │  센서 데이터 수신     │
                    │  → 화재 판단 (TinyML) │
                    │  → 출동 명령         │
                    │  → 실시간 업데이트    │
                    │  → LCD 상황 표시     │
                    │  → CLI 수동 개입     │
                    └────────┬────────────┘
                             │
                        nRF24L01 무선
                             │
                  ┌──────────┼──────────┐
                  ▼          ▼          ▼
               [로봇 A]   [로봇 B]   [로봇 C]
                STM32      STM32      STM32
                FreeRTOS   FreeRTOS   FreeRTOS
    
                각 로봇:
                - 출동 명령 수신 → 목표 좌표로 이동
                - 이동 중 실시간 좌표/규모 업데이트 반영
                - 현장 도착 → AMG8833으로 정밀 조준 → 분사
                - 진압 완료 → 관제에 보고 → 대기 위치 복귀
    ```
    
    ### 3계층 역할 분리
    
    |  | 센서 노드 (공장 내 고정) | 관제 시스템 (원격) | 소방 로봇 (공장 내 이동) |
    | --- | --- | --- | --- |
    | 위치 | 공장 구역별 천장/벽면 | 안전한 관제실 | 공장 내 대기 → 출동 |
    | MCU | STM32 | STM32 + LCD | STM32 + FreeRTOS |
    | 핵심 역할 | 열화상 데이터 수집·전송 | 판단·지휘·모니터링 | 이동·조준·분사 |
    | AMG8833 | 광역 감시용 | 없음 (데이터만 수신) | 근접 조준용 |
    | 통신 | 관제로 데이터 송신 | 센서 수신 + 로봇에 명령 | 관제 명령 수신 + 상태 보고 |

    ## 관제 시스템 상세
    
    ### 자동 모드 (기본)
    
    ```
    센서 노드에서 열화상 데이터 수신
            │
      TinyML 모델로 화재 판단
      (64개 온도값 → 소/중/대 분류)
            │
        ┌───┴───────────────────────┐
        │                           │
     [소규모]                    [대규모]
     가장 가까운 로봇 1대 출동    로봇 2~3대 동시 출동
        │                           │
        └───────────┬───────────────┘
                    │
            주기적으로 센서 재측정
                    │
              ┌─────┴─────┐
              │           │
           변화 없음   확산 감지
              │        좌표 업데이트
              │        규모 격상 → 추가 출동
              │           │
              └─────┬─────┘
                    │
              LCD에 실시간 상황 표시
    ```
    
    ### 수동 모드 (관제관 개입)
    
    관제 STM32에 UART로 노트북을 연결하여 CLI 명령으로 직접 제어:
    
    ```
    > status                    # 전체 시스템 상태 조회
                                # 센서 노드 상태, 로봇 위치/상태, 화재 현황
    
    > sensors                   # 센서 노드 열화상 데이터 실시간 조회
    
    > dispatch A 3,5            # 로봇 A를 좌표 (3,5)로 수동 출동
    > dispatch all 3,5          # 전체 로봇을 좌표 (3,5)로 출동
    
    > recall A                  # 로봇 A 복귀 명령
    > recall all                # 전체 로봇 복귀
    
    > spray A on                # 로봇 A 수동 분사 시작
    > spray A off               # 로봇 A 분사 중지
    
    > mode auto                 # 자동 모드 전환 (AI 판단 → 자동 출동)
    > mode manual               # 수동 모드 전환 (CLI 명령으로만 제어)
    
    > emergency stop            # 전체 로봇 즉시 정지
    ```
    
    ### LCD 표시 내용 (SPI LCD — ILI9341)
    
    ```
    ┌──────────────────────────────┐
    │  [자동모드]     12:34:56      │
    │                              │
    │  구역A: 정상   구역B: ⚠ 42°C │
    │  구역C: 정상                  │
    │                              │
    │  로봇A: 대기   (1,2)         │
    │  로봇B: 출동중 → (3,5)       │
    │  로봇C: 분사중  (3,5)        │
    │                              │
    │  화재: 구역B (3,5) 규모:중    │
    └──────────────────────────────┘
    ```
    
    > LCD 드라이버도 SPI 기반 펌웨어. UART CLI 파서도 펌웨어. **관제 시스템 전체가 STM32 펌웨어로 구현된다.**

    ### FreeRTOS 태스크 설계
    
    **관제 시스템:**
    
    | 태스크 | 주기 | 우선순위 | 역할 |
    | --- | --- | --- | --- |
    | 센서 데이터 수신 | 수신 시 | 최상 | nRF24L01로 센서 노드 데이터 수신 |
    | 화재 판단 | 1Hz | 상 | TinyML 추론, 규모 분류, 확산 추적 |
    | 출동/통신 관리 | 이벤트 시 | 상 | 출동 명령 전송, 로봇 상태 수신, 업데이트 브로드캐스트 |
    | LCD 갱신 | 5Hz | 중 | 현재 상황을 LCD에 표시 |
    | UART CLI 파싱 | 수신 시 | 중 | 노트북에서 받은 명령 파싱 + 실행 |
    
    **소방 로봇:**
    
    | 태스크 | 주기 | 우선순위 | 역할 |
    | --- | --- | --- | --- |
    | 모터 PID 제어 | 1kHz | 최상 | PWM 출력, 엔코더 피드백, 속도 제어 |
    | 센서 읽기 | 100Hz | 상 | IMU + 엔코더 → 위치/자세 추정 |
    | 열화상 스캔 | 10Hz | 중 | AMG8833 읽기, 열원 방향 계산 |
    | 무선 통신 | 수신 시 | 중 | 관제 명령 수신, 좌표 업데이트 반영 |
    | 임무 판단 | 10Hz | 중 | 상태머신 전이, 분사 판단 |
    | UART 로깅 | 1Hz | 최하 | 디버깅용 상태 출력 |
    
    ### HAL 사용 전략
    
    | 계층 | 방식 | 이유 |
    | --- | --- | --- |
    | 타이머 PWM | HAL + 필요시 레지스터 | PID 루프 주기 정밀 제어 |
    | I2C / SPI | HAL | 센서·무선 모듈·LCD 통신은 HAL로 충분 |
    | DMA | HAL | 레지스터 직접 구현은 시간 대비 효율 낮음 |
    | NVIC | HAL + 직접 설정 병행 | 인터럽트 우선순위 설계는 직접 이해 필요 |
    | GPIO | HAL | 단순 ON/OFF |
    
    > "HAL로 개발 효율을 확보하되, 내부 동작 원리는 레지스터 레벨로 이해한다"

    ## 하드웨어 구성
    
    ### 센서 노드 (1개 기준, 2~3개 배치)

    ## 시연 시나리오
    
    ### 시나리오 1: 소규모 화재 — 자동 단독 진압
    
    1. 센서 노드 B가 구역 B에서 열원 감지 → 관제로 데이터 전송
    2. 관제 시스템이 TinyML로 규모 판단 → 소규모
    3. LCD에 화재 표시 + 가장 가까운 로봇 A에 자동 출동 명령
    4. 로봇 A 이동 → 현장 도착 → AMG8833으로 정밀 조준 → 분사
    5. 온도 하강 확인 → 관제에 진압 완료 보고 → 대기 위치 복귀
    
    ### 시나리오 2: 대규모 화재 — 자동 협업 진압 + 실시간 업데이트
    
    1. 센서 노드 B가 대규모 열원 감지 → 관제로 전송
    2. 관제가 규모 판단 → 대규모 → 로봇 A, B 동시 출동 명령
    3. 이동 중 센서 노드 C도 확산된 열원 감지 → 관제에 전송
    4. 관제가 좌표 업데이트 + 규모 격상 → 로봇 A에 새 좌표 전송 + 로봇 C 추가 출동
    5. 로봇 A 경로 재계산, 3대 도착 → 다방향 동시 분사
    6. 진압 완료 → 관제에 보고 → 전원 복귀
    
    ### 시나리오 3: 수동 개입
    
    1. 자동 모드에서 시나리오 진행 중
    2. 관제관이 CLI에서 `mode manual` 입력 → 수동 모드 전환
    3. `dispatch C 5,3` → 로봇 C를 특정 좌표로 수동 출동
    4. `spray C on` → 수동 분사 시작
    5. `recall all` → 전체 복귀
    6. `mode auto` → 자동 모드 복귀

    ### 1️⃣ 모터 제어 / PID

    - **산출물**: `move(Vx, Vy, omega)` — 전방향 이동 API
    
    ### 2️⃣ IR 공 추적
    
    - TSOP4838 센서 8~16개 원형 배치
    - ADC/GPIO 읽기, 노이즈 필터링
    - 공 방위각 + 상대 거리 추정 알고리즘
    - IR 센서 링 하드웨어 (만능기판 납땜 + 마운트 제작)
    - **산출물**: `getBallAngle()`, `getBallIntensity()` API
    
    ### 3️⃣ IMU / 오도메트리
    
    - MPU6050 I2C 드라이버 구현
    - Complementary 또는 Madgwick 필터로 헤딩 추정
    - 엔코더 데드레커닝 + IMU 센서 퓨전
    - 드리프트 보정, 리셋 로직
    - **산출물**: `getHeading()`, `getPosition()` API
    
    ### 4️⃣ nRF24L01 통신
    
    - SPI 드라이버 (레지스터 레벨 구현 권장)
    - 패킷 프로토콜 설계: 로봇 ID, 위치, 공 방향, 역할, 상태
    - 송수신 주기 관리 (10~50ms)
    - 채널 호핑 / 재전송으로 2.4GHz 간섭 대응
    - **산출물**: `sendState()`, `recvTeammate()` API
    
    ### 5️⃣ 자율 상태머신 / 전략
    
    - FSM: IDLE → CHASE → ALIGN → KICK → DEFEND → GOALKEEPER
    - 공 위치 + 자기 위치 + 팀 동료 정보 기반 의사결정
    - 역할 동적 전환 (가까운 로봇 → 공격, 먼 로봇 → 수비)
    - **산출물**: 매 루프마다 `move()`를 호출하는 최상위 전략 로직
    
    ### 6️⃣ 통합 / PL + 킥커·라인센서 제어
    
    - 프로젝트 아키텍처 설계, 모듈 인터페이스 헤더 정의
    - 빌드 시스템 (Makefile/CMake), Git 브랜치 전략
    - 기구부 총괄: 프레임 DXF 수정, 레이저커팅/3D프린팅 외주, 조립
    - **킥커 제어**: MT3608 충전 관리 + MOSFET 방전 타이밍
    - **라인센서**: TCRT5000 ADC 읽기 + 필드 경계 감지 로직
    - 디버깅 인프라 (UART 로깅, LED 상태 표시), 문서화, 발표자료
    - **산출물**: `kick()`, `isOnLine()` API + 빌드/통합 환경
    
    ### 모듈 간 인터페이스
    
    상태머신(5번)이 나머지 모듈의 API를 조합하여 로봇을 동작시킨다.
    
    ```
    [상태머신]
        ├── move(Vx, Vy, ω)           ← 모터 제어
        ├── getBallAngle/Intensity()   ← IR 추적
        ├── getHeading/Position()      ← IMU/오도메트리
        ├── sendState/recvTeammate()   ← nRF 통신
        ├── kick()                     ← 킥커 (PL 담당)
        └── isOnLine()                 ← 라인센서 (PL 담당)
    ```
    
    **1주차에 PL이 인터페이스 헤더를 먼저 정의 → 나머지 5명 독립 개발 가능**

    # 🚛 임베디드 팀 프로젝트 제안: 다중 AGV 물류 이송 시스템
    
    ## 프로젝트 개요
    
    3륜 옴니휠 자율 로봇(AGV) 2대가 천장 카메라 기반 위치 추적 환경에서, 중앙 서버로부터 MQTT 명령을 받아 물류(물체)를 밀어서 지정된 위치로 이송하는 자동화 시스템.
    
    오픈소스 [Robot Soccer Kit](https://github.com/robot-soccer-kit/robot-soccer-kit)의 기구부 + 비전 트래킹 코드를 활용하되, 통신을 **MQTT over Wi-Fi**로 교체하고 MCU를 **STM32(실시간 제어) + ESP32(Wi-Fi/MQTT 게이트웨이)** 이종 구조로 설계한다.
    
    > **핵심 차별점**
    > 
    > - 교육용 프로토콜(nRF, Bluetooth)이 아닌 **MQTT** — 실제 물류/IoT 현장 표준
    > - **이종 MCU 분리 설계** — 실시간 제어(STM32)와 네트워크(ESP32)의 타이밍 요구사항이 다르므로 분리 (산업 표준 아키텍처)
    > - Robot Soccer Kit 원본은 PC가 모터 속도까지 직접 지정하지만, 우리 시스템은 서버가 목표 좌표만 전달하고 **로봇이 자율적으로 이동 제어**를 수행

    ## 왜 물류 이송인가
    
    ### 로봇 축구의 우려점 해소
    
    | 로봇 축구 우려 | 물류 이송에서 해결 |
    | --- | --- |
    | 로봇 3대 이상 필요 (3:3) | **2대면 충분** |
    | 로봇 간 통신이 nRF (교육용) | **MQTT (현장 기술)** |
    | 킥커 하드웨어 필요 | **불필요, 하드웨어 단순화** |
    | 상대팀 없으면 데모 불가 | **단독으로 데모 가능** |
    
    ### 현장 기술 스택 직결
    
    | 기술 요소 | 프로젝트 내 적용 | 실제 현장 대응 |
    | --- | --- | --- |
    | MQTT pub/sub | 로봇↔서버 명령/상태 교환 | AWS IoT, 쿠팡 물류센터 AGV |
    | UART 브리지 | STM32↔ESP32 프레임 프로토콜 | 이종 MCU 간 시리얼 통신 |
    | 이종 MCU 분리 | 제어부/통신부 분리 설계 | 산업용 로봇 표준 아키텍처 |
    | OpenCV 마커 트래킹 | ArUco 기반 로봇/물류 위치 추적 | 물류 창고 비전 시스템 |
    | 타이머/PWM | 모터 PID 속도 제어 | 모터 드라이버 제어 |
    | I2C | IMU (MPU6050) | 센서 인터페이스 |
    | DMA | UART 수신 버퍼링 | 고성능 데이터 전송 |
    | A* 경로 계획 | 이송 경로 최적화 | AGV 경로 계획 |
    | Fleet Management | 다중 로봇 작업 배분/조율 | 물류 로봇 관제 시스템 |
    
    ### 포트폴리오 어필 포인트
    
    - **산업 표준 프로토콜 (MQTT)** → "교육용이 아닌 현장 기술을 사용했다"
    - **이종 MCU 시스템 설계** → "실시간 제어와 네트워크를 왜 분리했는지 설명할 수 있다"
    - **STM32 레지스터 레벨 제어** → HAL 의존 최소화로 깊이 증명
    - **협동 이송** → 2대가 동기화하여 물체를 미는 멀티에이전트 경험
    - **풀스택** → 펌웨어(STM32) ~ 게이트웨이(ESP32) ~ 서버 ~ 대시보드
    
    ### 타겟 회사
    
    현대로보틱스, 삼성SDS, 쿠팡 로보틱스, 네이버랩스, 한화시스템, LIG넥스원, 삼성 미래로봇추진단

    ### 이종 MCU 분리 설계 근거
    
    ```
    STM32 (실시간 제어 전담)        ESP32 (네트워크 전담)
    ──────────────────────        ────────────────────
    PID 루프: 1kHz 주기             Wi-Fi 스택: 수십ms 블로킹 가능
    센서 폴링: 수백µs               MQTT 패킷: 가변 지연
    타이밍 예측 가능 필수            타이밍 비결정적 허용
    
    → 같은 MCU에서 돌리면 Wi-Fi 처리 중 PID 루프가 밀려서 로봇이 떨림
    → 실제 산업 현장에서도 제어부/통신부 분리가 표준 설계
    → 면접에서 "왜 MCU를 2개 썼냐" → "실시간 제어와 네트워크 I/O의 타이밍 요구사항이 다르기 때문"
    ```

    ## 비전 트래킹 (천장 카메라)
    
    ### ArUco 마커 기반 — AI가 아닌 기하학 연산
    
    ArUco 마커는 OpenCV의 패턴 매칭으로 인식한다. 딥러닝/학습이 아닌 미리 정의된 흑백 패턴의 기하학적 위치/각도 계산이며, 일반 노트북 CPU에서 30fps 이상 처리 가능하여 실시간 제어에 적합하다.
    
    ```
    카메라가 매 프레임 인식하는 것:
    
      로봇1 마커 (ID: 1) → 위치 (x1, y1), 헤딩 θ1
      로봇2 마커 (ID: 2) → 위치 (x2, y2), 헤딩 θ2
      물류A 마커 (ID: 10) → 위치 (xa, ya)
      물류B 마커 (ID: 11) → 위치 (xb, yb)
    ```
    
    Robot Soccer Kit의 비전 트래킹 코드를 활용하되, 물류 마커 인식을 추가한다.
    
    ### 오도메트리 Fallback
    
    카메라 데이터가 끊길 경우(통신 지연, 마커 가림), IMU + 엔코더 오도메트리로 짧은 시간 동안 위치를 보조 추정한다. "비전 기반 위치 추적 + 오도메트리 fallback 이중화 설계"로 시스템 안정성을 확보한다.

    ## MQTT 토픽 설계
    
    ```
    fleet/robot{N}/command     서버→로봇   작업 명령 (목표 좌표, 밀기 방향)
    fleet/robot{N}/status      로봇→서버   상태 보고 (위치, 상태, 배터리)
    fleet/robot{N}/heartbeat   로봇→서버   생존 확인 (1초 주기)
    fleet/emergency            양방향      비상정지 (전체 로봇 구독)
    fleet/sync                 서버→전체   협동 이송 동기 신호
    ```

    ## STM32 ↔ ESP32 UART 프로토콜
    
    ```
    [STX 0x02] [LEN 1B] [CMD 1B] [PAYLOAD NB] [CRC8 1B] [ETX 0x03]
    
    CMD 0x01  MOVE_TO         서버→STM32   target_x, target_y, heading
    CMD 0x02  PUSH_START      서버→STM32   push_heading, speed
    CMD 0x03  STATUS_REPORT   STM32→서버   pos, heading, state, battery, contact
    CMD 0x04  EMERGENCY_STOP  양방향       즉시 정지
    CMD 0x05  TASK_COMPLETE   STM32→서버   task_id, 결과
    CMD 0x06  SYNC_PUSH       서버→STM32   협동 이송 시작/정지 동기 신호
    CMD 0x07  HEARTBEAT_ACK   ESP32→STM32  서버 연결 상태
    ```
    
    STM32: UART DMA 수신 → 링버퍼 → 메인 루프에서 프레임 파싱
    ESP32: MQTT 메시지 수신 → UART 프레임 변환 → STM32 전달

    ## 밀기 제어 로직
    
    ### 접근 각도 계산 (서버)
    
    ```
    물류 위치: (cx, cy)
    목표 위치: (tx, ty)
    밀기 방향: atan2(ty - cy, tx - cx)
    접근 위치: 물류 뒤쪽 (밀기 방향 반대로 일정 거리)
    
    서버가 로봇에게 전달:
      1. approach_x, approach_y (물류 뒤쪽 접근 좌표)
      2. push_heading (밀기 방향)
      3. target_x, target_y (물류 최종 목표)
    ```
    
    ### 로봇 FSM
    
    ```
    IDLE
     │ 서버에서 작업 수신
     ▼
    APPROACH ── 물류 뒤쪽 접근 좌표로 이동
     │ 접근 완료
     ▼
    ALIGN ── 밀기 방향으로 헤딩 정렬 (IMU 기반 정밀 회전)
     │ 정렬 완료
     ▼
    PUSH ── 직진으로 밀기 (저속 정밀 모드)
     │ 서버가 카메라로 물류 위치 확인 → 목표 도달
     ▼
    VERIFY ── 서버에서 완료 확인 수신
     │ 확인 완료
     ▼
    RETURN ── 대기 위치로 복귀
     │
     ▼
    IDLE
    ```
    
    ### 협동 이송 (고급 시나리오)
    
    ```
    큰 물류를 2대가 동시에 미는 경우:
    
      서버가 계산:
        로봇1 → 물류 왼쪽 뒤에서 접근
        로봇2 → 물류 오른쪽 뒤에서 접근
    
      동기화:
        1. 둘 다 APPROACH 완료 → 서버에 보고
        2. 서버가 fleet/sync로 "PUSH 시작" 동시 발행
        3. 둘 다 동일 속도로 밀기
        4. 한쪽이 접촉 해제되면 → 서버가 정지 신호
    ```

    ## 데모 시나리오
    
    ### 필드 구성 (약 1.5m x 2m)
    
    ```
    ┌──────────────────────────────────┐
    │                                  │
    │   📦A              📦B           │  ← 물류 (ArUco 마커 부착)
    │                                  │
    │       🤖1              🤖2       │  ← 로봇 2대 (ArUco 마커 부착)
    │                                  │
    │   [S1]        [S2]        [S3]   │  ← 배달 스테이션
    │                                  │
    └──────────────────────────────────┘
          📷 천장 카메라 (위에서 촬영)
    ```
    
    ### 시연 순서
    
    ```
    시나리오 1: 단독 이송
      대시보드에서 "물류A → S1" 입력
      → 로봇1이 물류A 뒤쪽으로 접근 → S1 방향으로 밀기 → 완료
    
    시나리오 2: 동시 이송
      대시보드에서 "물류A → S1, 물류B → S3" 동시 입력
      → 로봇1이 A→S1, 로봇2가 B→S3 동시 수행
      → 경로 겹치면 서버가 우선순위 조율
    
    시나리오 3: 협동 이송
      대시보드에서 "큰 물류C → S2" 입력
      → 로봇 2대가 양쪽에서 동시에 밀기 (서버 동기화)
    
    전체 과정이 웹 대시보드에서 실시간으로 시각화
    ```

    ## 역할 분배 (6명)
    
    > 전원이 임베디드 코드를 작성한다. 누구도 "관리만 하는" 역할이 아님.
    > 
    
    ### 1️⃣ 모터 제어/PID
    
    - 3륜 옴니휠 역기구학 (120° 배치, 3x3 행렬)
    - 엔코더 타이머 캡처, PID 속도 제어, PWM 출력
    - 목표 좌표 추종 제어 (현재 위치 → 목표 위치 벡터 계산 → 옴니휠 속도)
    - 밀기 시 저속 정밀 모드 (속도/힘 제어)
    - **산출물**: `moveToTarget(x, y, θ)`, `pushForward(speed)` API
    
    ### 2️⃣ IMU/오도메트리 + 접촉·장애물 감지
    
    - MPU6050 I2C 드라이버, Madgwick 필터 헤딩 추정
    - 정밀 회전 보정 (ALIGN 상태에서 목표 각도 정렬)
    - HC-SR04 초음파 센서로 물류 접촉 및 전방 장애물 감지
    - 엔코더 기반 오도메트리 (카메라 fallback용)
    - **산출물**: `getHeading()`, `correctTurnTo(θ)`, `isContactDetected()`, `isObstacleAhead()` API
    
    ### 3️⃣ UART 프로토콜 (STM32 쪽)
    
    - UART DMA 수신 + 링버퍼 구현
    - 프레임 파싱 (STX/ETX/LEN/CRC8 검증)
    - CMD별 핸들러 디스패치
    - 상태 보고 프레임 생성 + 송신
    - **산출물**: `uart_init()`, `uart_parse()`, `recvCommand()`, `sendStatus()` API
    
    ### 4️⃣ MQTT/Wi-Fi 통신 (ESP32)
    
    - ESP32 Arduino or ESP-IDF 기반 MQTT 클라이언트
    - Wi-Fi 접속, 자동 재연결 로직
    - MQTT 토픽 subscribe/publish
    - JSON 직렬화/역직렬화
    - MQTT 메시지 ↔ UART 프레임 변환 브리지
    - Heartbeat 발행 (1초 주기)
    - **산출물**: ESP32 게이트웨이 펌웨어 전체
    
    ### 5️⃣ 경로 계획/상태머신 (STM32)
    
    - 서버로부터 받은 접근 좌표/밀기 방향 기반 FSM 실행
    - IDLE → APPROACH → ALIGN → PUSH → VERIFY → RETURN
    - 긴급 정지 처리, 장애물 회피 기동
    - 협동 이송 시 SYNC 신호 대기 로직
    - **산출물**: 메인 FSM 루프, `planApproach()`, `executePush()`
    
    ### 6️⃣ PL + 서버 (비전/Fleet/대시보드)
    
    - **비전 트래킹**: 천장 카메라 + OpenCV ArUco → 로봇/물류 위치 실시간 계산
    - **Task Manager**: 작업 큐 관리, 유휴 로봇 선택, 접근 각도/밀기 방향 계산
    - **협동 조율**: 2대 동시 밀기 시 동기화 신호
    - **MQTT Broker**: Mosquitto 세팅
    - **웹 대시보드**: 필드 맵 위 로봇/물류 위치 실시간 표시, 작업 상태, 로그
    - 기구부 조립, 필드 제작, 빌드/통합, 문서화
    - **산출물**: 비전 서버 + Fleet Manager + 대시보드 (Python + 웹)
    
    ### 모듈 간 인터페이스
    
    ```
    [천장 카메라]
         │ OpenCV ArUco
         ▼
    [서버] ─── MQTT (Wi-Fi) ───▶ [ESP32] ──UART──▶ [STM32]
     │ 위치 추적                                      │
     │ 작업 배분                        ┌──────────────┼──────────────┐
     │ 경로/접근각 계산                  ▼              ▼              ▼
     │ 협동 동기화                 [FSM/경로]     [모터/PID]    [IMU/장애물]
     │ 대시보드                   executePush()  moveToTarget() getHeading()
     └─────────────────────────   IDLE→PUSH→..  pushForward()  isContact()
    ```
    
    1주차에 PL이 MQTT 토픽 + UART 프레임 포맷 + API 헤더를 정의 → 나머지 5명 독립 개발 가능.
