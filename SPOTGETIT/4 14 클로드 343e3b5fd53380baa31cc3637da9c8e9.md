# PendSV & ARM Cortex-M Exception 학습 정리

**날짜**: 2026-04-15

---

## 1. PendSV (Pendable Service Call)

- ARM Cortex-M의 시스템 exception 중 하나 (exception 번호 14)
- **Context switching을 안전하게 지연 실행**하기 위한 메커니즘
- 소프트웨어가 비트를 써서 trigger하는 방식 (하드웨어 신호가 아님)
- 우선순위를 **최하위**로 설정하여, 다른 모든 ISR이 끝난 뒤에 실행됨

### Context Switch 흐름

1. **SysTick ISR** 발생 → 스케줄러가 "task 전환 필요" 판단
2. `SCB->ICSR |= (1U << 28);` → PendSV를 **pending** 시킴 (= 실행 예약)
3. 높은 우선순위 ISR이 모두 완료됨
4. **PendSV_Handler** 실행 → 실제 context switch 수행 (레지스터 save/restore, SP 교체)

### SysTick vs PendSV 역할 분담

|  | SysTick | PendSV |
| --- | --- | --- |
| 역할 | 주기적 틱, 스케줄링 **판단** | 실제 context switch **실행** |
| 우선순위 | 높음 | 최하위 |
| Trigger | 하드웨어 타이머 | 소프트웨어 (PENDSVSET 비트) |

### "Pending"의 의미

- ARM Cortex-M에서 exception의 3가지 상태: **Inactive → Pending → Active**
- Pending = "실행해야 한다"고 등록됐지만 아직 실행 안 된 상태
- PENDSVSET = 1을 쓰면 PendSV exception이 pending 상태가 되고, 우선순위 조건이 맞으면 active로 전환되어 handler 실행

## 2. `1U` 리터럴

- `1U` = **unsigned integer** 리터럴 1
- `1`은 signed int, `1U`는 unsigned int
- 레지스터 비트 조작 시 signed int를 left shift하면 MSB 쪽에서 **undefined behavior** 발생 가능
- `(1 << 28)` → 엄밀히 UB 영역 / `(1U << 28)` → 안전
- 임베디드 비트 조작에서는 습관적으로 `1U` 사용 권장

---

## 3. SCB (System Control Block)

- ARM Cortex-M 시스템 제어 레지스터들의 집합
- 메모리 주소 `0xE000ED00`부터 시작
- 주요 레지스터:
    - **ICSR** (`0xE000ED04`): Interrupt Control and State Register — PENDSVSET 비트 포함
    - **VTOR**: Vector Table Offset Register
    - **AIRCR**: 인터럽트 우선순위 그룹 설정
    - **SCR**: Sleep 모드 제어
    - **SHPR**: System Handler Priority Register

---

## 4. SCB->ICSR 레지스터 비트 구성

| 비트 | 이름 | 설명 |
| --- | --- | --- |
| 31 | NMIPENDSET | NMI pending 시킴 |
| 30-29 | Reserved | - |
| 28 | **PENDSVSET** | PendSV pending 시킴 (RTOS context switch 핵심) |
| 27 | PENDSVCLR | PendSV pending 취소 |
| 26 | PENDSTSET | SysTick pending 시킴 |
| 25 | PENDSTCLR | SysTick pending 취소 |
| 24 | Reserved | - |
| 23 | ISRPENDING | 외부 인터럽트 중 pending된 것 존재 여부 (읽기 전용) |
| 22 | ISRPREEMPT | 디버거용, pending ISR 실행 예정 여부 (읽기 전용) |
| 21-20 | Reserved | - |
| 20-12 | VECTPENDING | pending 중 가장 높은 우선순위 exception 번호 (읽기 전용) |
| 11 | RETTOBASE | 리턴 시 Thread mode로 돌아가는지 여부 |
| 10-9 | Reserved | - |
| 8-0 | VECTACTIVE | 현재 실행 중인 exception 번호 (0=Thread mode) |

---

## 5. Exception vs Interrupt

- **Exception**: CPU의 정상 실행 흐름을 벗어나게 하는 **모든 이벤트** (상위 개념)
- **Interrupt**: exception 중 **외부 주변장치**가 trigger하는 것 (부분집합)

### Cortex-M4 시스템 Exception 전체 목록

| 번호 | 이름 | 설명 |
| --- | --- | --- |
| 1 | Reset | 리셋 |
| 2 | NMI | Non-Maskable Interrupt |
| 3 | HardFault | 처리 못한 모든 fault |
| 4 | MemManage | 메모리 보호 위반 (MPU) |
| 5 | BusFault | 버스 에러 |
| 6 | UsageFault | 잘못된 명령어, 0 나누기 등 |
| 7-10 | Reserved | - |
| 11 | SVCall | SVC 명령어, RTOS 시스템 콜 용도 |
| 12 | Debug Monitor | 디버깅용 |
| 13 | Reserved | - |
| 14 | **PendSV** | Context switch용 |
| 15 | **SysTick** | 시스템 타이머 |
| 16 이후 | 외부 IRQ | GPIO, UART, I2C 등 (STM32F446은 97개) |

## 6. Vector Table & Handler 연결

- 메모리 시작 주소(`0x00000000`)에 **함수 포인터 배열**로 존재
- Exception 발생 시 CPU가 해당 번호의 슬롯에서 handler 주소를 읽어 점프
- 하드웨어가 자동으로 R0-R3, R12, LR, PC, xPSR을 스택에 push

### `.weak` 정의

```nasm
.weak   PendSV_Handler
.thumb_set PendSV_Handler, Default_Handler
```

- **`.weak`**: 약한 심볼 정의. 다른 파일에서 같은 이름의 함수를 정의하면 링커가 대체
- **`Default_Handler`**: 무한루프 (`b Infinite_Loop`). 아무도 구현 안 하면 여기서 멈춤
- 사용자가 `void PendSV_Handler(void) { ... }`를 정의하면 weak 정의가 무시되고 연결됨
- FreeRTOS의 port.c가 이 방식으로 PendSV_Handler, SysTick_Handler, SVC_Handler를 구현

### Empty Project (HAL 없음)에서 PendSV 테스트 코드

```c
#include <stdint.h>

#define RCC_AHB1ENR   (*(volatile uint32_t*)0x40023830)
#define GPIOA_MODER   (*(volatile uint32_t*)0x40020000)
#define GPIOA_ODR     (*(volatile uint32_t*)0x40020014)
#define SCB_ICSR      (*(volatile uint32_t*)0xE000ED04)

void PendSV_Handler(void) {
    GPIOA_ODR ^= (1U << 5);  // PA5 LED 토글
}

int main(void) {
    RCC_AHB1ENR |= (1U << 0);   // GPIOA 클럭 활성화
    GPIOA_MODER |= (1U << 10);  // PA5 출력 모드
    GPIOA_MODER &= ~(1U << 11);

    while (1) {
        SCB_ICSR |= (1U << 28);  // PendSV pending
        for (volatile int i = 0; i < 500000; i++);  // 딜레이
    }
}
```

## 대화 1: Fusion 360으로 STS3215 모터 마운트 설계하기

SpotMicro 프레임을 STS3215 서보(40×30×50.5mm)에 맞게 Fusion 360에서 수정하는 방법에 대한 대화.

### STS3215 핵심 치수

- 외형: 40 × 30 × 50.5mm
- Feetech 공식 데이터시트에서 2D 도면을 DXF로 임포트하면 마운트 홀 위치를 정확하게 잡을 수 있음

### Fusion 360 필수 단축키 정리

| 단축키 | 기능 | 용도 |
| --- | --- | --- |
| S | Sketch | 스케치 모드 진입 |
| E | Extrude | 스케치를 3D로 밀어올리기/자르기 |
| Q | Push/Pull | 기존 면을 밀거나 당기기 |
| J | Joint | 컴포넌트 간 관절 연결 |
| M | Move/Copy | 바디/컴포넌트 이동/회전 |
| D | Dimension | 치수 구속 |
| 가운데 휠 드래그 | 패닝 | 화면 이동 |

### 모터 마운트 수정 워크플로우

1. 대상 면(face)에서 스케치 모드 진입
2. 모터 외곽선과 볼트 홀 위치를 Dimension(D)으로 구속
3. Extrude(E)에서 Cut 옵션으로 모터 깊이만큼 파기
4. Inspect → Interference로 서보와 프레임 간 간섭 체크
5. 스케치를 Dimension 도구로 완전 구속(Fully Constrained) 상태로 만들어야 나중에 파라메트릭 수정 가능
6. 하단 Timeline 히스토리 바에서 과거 피처를 더블클릭하면 치수 수정 가능

## 대화 2: Fusion 360에서 바디/컴포넌트 관리

SpotMicro 다리 파츠를 Fusion 360에서 조립하면서 발생한 실무적인 문제들과 해결법.

### 여러 바디를 하나로 묶어서 이동하기

**방법 1: 컴포넌트로 묶기 (가장 추천)**

- Browser 트리에서 바디들을 Ctrl+클릭으로 다중 선택 → 우클릭 → "Create Components from Bodies"
- 안 되면: 루트 우클릭 → New Component → 바디를 드래그 앤 드롭으로 넣기
- 컴포넌트 단위로 Move/Copy 하면 내부 바디가 전부 같이 움직임
- 나중에 개별 수정이 필요할 때 유연함

**방법 2: Combine → Join**

- Design → Modify → Combine → Target Body + Tool Body 선택 → Operation: Join
- 완전히 하나의 솔리드 바디가 됨
- 영구적으로 합쳐도 되는 경우에 사용
- 분리하려면 Timeline에서 Combine 피처 삭제

**방법 3: Move/Copy에서 다중 선택**

- M키 → Selection 필터를 Body로 설정 → 뷰포트에서 여러 바디 클릭 → 동시 이동
- 임시로 위치 잡을 때 빠름

### 같이 움직이는 바디 분리하기

| 원인 | 해결법 |
| --- | --- |
| 같은 컴포넌트 안에 있음 | 바디 우클릭 → Cut → 대상 컴포넌트 Activate → Paste |
| Combine(Join)으로 합쳐짐 | Timeline에서 Combine 피처 우클릭 → Delete |
| Group으로 묶여있음 | 바디 우클릭 → Ungroup |

### 바디 이동 관련 핵심

- **바디는 좌클릭 드래그로 이동 불가** — 이것이 정상 동작
- 반드시 M키(Move/Copy)로 이동해야 함
- **컴포넌트는** Grounded 해제 시 드래그 가능
    - Browser에서 컴포넌트 이름 옆 압정 아이콘 = Grounded 상태
    - 우클릭 → Unground로 해제

### 바디를 특정 컴포넌트(어셈블리)에 포함시키기

"Move to Component"가 우클릭 메뉴에 없을 때:

1. Browser에서 해당 바디 우클릭 → Cut (Ctrl+X)
2. 대상 컴포넌트(예: Left Arm:1) 우클릭 → Activate
3. Ctrl+V (Paste)
4. 바디가 해당 컴포넌트 안에 들어감

또는 Browser에서 바디를 대상 컴포넌트 위로 직접 드래그 앤 드롭 (Activate 선행 필요할 수 있음)

### Mirror (거울상 생성)

**기본 사용법:**

1. Create → Mirror
2. Pattern Type: Bodies (또는 Features/Components)
3. Objects: 미러할 바디 선택
4. Mirror Plane: 대칭 기준 평면 선택 (XY, XZ, YZ)
5. OK

**기준 평면이 없을 때:**

- Construct → Midplane: 두 면 사이 중간 평면 생성
- Construct → Offset Plane: 원하는 위치에 평면 생성

**원본-미러 종속 관계:**

- 원본 삭제 시 미러도 같이 삭제됨 (종속 피처)
- 독립시키는 방법:
    1. Timeline에서 Mirror 피처 우클릭 → Break Link (있는 경우)
    2. 미러된 바디 우클릭 → Copy → Paste로 별도 바디 생성
    3. 또는 Move/Copy (Create Copy 체크) → Modify → Scale에서 한 축 -1로 반전 → 완전 독립 바디

### 솔리드 변환이 안 되는 문제 (Open Edge)

메쉬에 빈 곳(열린 엣지)이 있으면 솔리드 바디로 변환 불가:

1. Surface 탭 → Patch 선택
2. 열린 엣지(빈 곳 경계)를 선택하면 면이 자동 생성
3. 모든 열린 엣지를 Patch로 채운 뒤 Stitch 실행
4. Open Edge가 0이 되면 솔리드로 변환 가능
5. 뷰를 돌려가며 투명한 곳(빠진 면)이 없는지 확인 필요
