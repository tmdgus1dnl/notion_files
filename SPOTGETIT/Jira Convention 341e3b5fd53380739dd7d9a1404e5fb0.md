## Jira 작성 가이드

### 1. 이슈 구조 및 상태 관리

- **계층 구조**: 에픽(Epic) → 작업(Task), 가능하다면 하위 작업(Subtask) 및 연결된 업무 항목도 적극 활용
- **상태(Status)**: `To Do` → `In Progress` → `Done` (단 3단계)
    - `In Progress` 상태에서 **플래그(Flag)**가 추가된 티켓은 '리뷰 중(`In Review`)'으로 간주

---

### 2. 백로그 작성 규칙

- 백로그 이름은 `~~제작` / `~~설계` / `~~구축` 등 **동사**로 종결
- 백로그 생성과 동시에 아래 항목 필수 작성
    - 스토리 포인트 / 담당자 / 설명 / 우선순위
- **매주 일요일 자정 전**까지 다음 주 백로그 사전 작성

---

### 3. 스토리 포인트 기준 (피보나치 수열)

| 포인트 | 기준 |
| --- | --- |
| 1점 | 아주 간단한 업무 |
| 2점 | 간단한 업무 |
| 3점 | 하루 안에 완료 가능한 업무 |
| 5점 | 며칠이 소요되는 업무 |
| 8점 | 한 스프린트 전체가 소요되는 업무 |

> 팀장 판단 하에 기준 조정 가능
> 

---

### 4. 자동화 기반 워크플로우 (GitLab 연동)

이슈 **생성 시에만** Jira에 직접 접속하며, 이후 상태 변경은 GitLab 작업을 통해 자동화합니다.

| 단계 | 행동 | 조건 | 결과 |
| --- | --- | --- | --- |
| **작업 시작** | GitLab 브랜치 생성 또는 첫 커밋 푸시 | 브랜치명 또는 커밋 메시지에 이슈 키 포함 | `To Do` → `In Progress` 자동 이동 |
| **리뷰 요청** | GitLab에서 Merge Request(MR) 생성 | MR 제목 또는 설명에 이슈 키 포함 | `In Progress` 유지 + 플래그 자동 추가 |
| **작업 완료** | 코드 리뷰 승인 후 MR 병합(Merge) | MR 제목 또는 설명에 이슈 키 포함 | `In Progress` → `Done` 자동 이동 |

---

### 5. GitLab 네이밍 규칙

- **브랜치**: `type/S14P21A206-{이슈번호}`
    - 예: `feat/S14P21A206-12`
- **커밋**: `type(scope): 작업 내용 #S14P21A206-{이슈번호}`
    - 예: `feat(be): 기상청 API 연동 주기 설정 #S14P21A206-14`
- **MR 제목**: 커밋 메시지와 동일한 포맷 사용

---

### 6. 티켓 작성 표준

- **제목**: `[도메인] 작업 내용`
    - 예: `[BE] 기상청 외부 API 연동 배치 구현`
- **본문**: 상세 내용은 PR 템플릿에 위임
