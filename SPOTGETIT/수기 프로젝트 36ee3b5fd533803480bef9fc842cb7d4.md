안녕하세요.

SSAFY 14기 자율 프로젝트 ‘다중 정찰 4족 로봇 관제 시스템’을 진행한 SPOT GET IT 팀입니다.

저희 팀은 재난 현장, 붕괴 지역, 군 작전 환경처럼 사람이 직접 진입하기 어렵거나 위험한 공간에서 구조 인력의 부담을 줄이고, 보다 안전하게 현장을 탐색할 수 있도록 **피지컬 AI 기술 기반의 다중 정찰 4족 로봇 관제 시스템**을 기획하고 개발했습니다.

이번 프로젝트의 핵심 목표는 단순히 로봇 한 대를 움직이는 것이 아니었습니다. 관제자가 여러 대의 로봇에게 탐색 경로(임무)를 할당하고, 각 로봇이 주변 환경을 인지하며, 자신의 위치를 추정하고, 장애물 상황에 따라 안전하게 경로를 조정하는 흐름을 하나의 시스템으로 연결하는 것이었습니다.

이를 위해 저희 팀은 실제 4족 보행 로봇과 가상 로봇을 함께 활용하는 시뮬레레이션 환경을 구축하고, ROS 2 기반으로 **Control,** **Perception(Camera&LiDAR), Localization, Navigation, 관제 UI**가 유기적으로 동작하는 구조를 설계했습니다.

## [프로젝트 기획 과정]

프로젝트 기획 단계에서 저희 팀이 가장 중요하게 생각한 키워드는 “위험 환경에서의 안전한 탐색”과 “다중 로봇 기반 관제”였습니다.

재난 현장이나 미지의 공간에서는 내부 구조를 정확히 알기 어렵고, 장애물이나 붕괴 위험과 같은 변수가 많습니다. 이러한 환경에서 사람이 직접 진입해 실종자를 수색하거나 현장을 확인하는 방식은 구조 인력에게 또 다른 위험이 될 수 있습니다. 그래서 저희는 사람이 직접 들어가기 전에 로봇이 먼저 투입되어 현장을 정찰하고, 관제자가 여러 로봇의 상태를 실시간으로 확인하며 제어할 수 있는 시스템이 필요하다고 판단했습니다.

처음에는 단순히 4족 보행 로봇이 정해진 경로를 따라 이동하는 기능을 생각했습니다. 하지만 기획을 구체화하면서, 실제 현장에서는 경로를 한 번 지정하는 것만으로는 충분하지 않다는 점을 확인했습니다. 로봇은 이동 중 장애물을 만날 수 있고, 위치 추정이 흔들릴 수 있으며, 관제자는 로봇이 어디까지 이동했는지와 어떤 상황을 마주하고 있는지를 지속적으로 확인해야 합니다.

따라서 저희 팀은 프로젝트의 방향을 “관제 중심의 다중 정찰 로봇 시스템”으로 구체화했습니다. 관제자는 지도 기반 화면에서 로봇별 탐색 경로를 할당하고, 각 로봇은 자신의 위치와 상태를 전달하며, LiDAR를 통해 주변 장애물과 주행 가능 영역을 판단하도록 설계했습니다. 이를 통해 사람이 직접 진입하기 어려운 공간에서 로봇이 먼저 정찰하고, 관제자는 여러 로봇의 탐색 상황을 한눈에 확인할 수 있는 구조를 목표로 삼았습니다.

저희가 기획 단계에서 정리한 핵심 기능은 다음과 같습니다.

첫째, 관제자가 로봇별 탐색 경로를 자유롭게 할당할 수 있어야 합니다.

둘째, 로봇은 자신의 현재 위치와 진행 상태를 추정하고 관제 시스템에 전달해야 합니다.

셋째, LiDAR를 활용해 주변 장애물과 주행 가능한 방향을 판단해야 합니다.

넷째, 전역 경로를 따라가면서도 동적으로 나타나는 장애물을 회피할 수 있어야 합니다.

다섯째, 실제 로봇이 명령대로 잘 보행할 수 있어야 합니다.

여섯째, 관제 화면에서 각 로봇의 현재 위치, 주행 상태, 임무 진행 상황을 실시간으로 확인할 수 있어야 합니다.

이 과정에서 저희 팀은 단순 기능 구현보다 **인지–위치추정–제어–판단–주행–관제**가 하나의 흐름으로 이어지는 시스템 구조를 만드는 데 집중했습니다.

## [프로젝트 수행 과정]

저희 프로젝트는 크게 **관제 시스템, 로봇 자율주행 SW, LiDAR 기반 환경 인지, 위치 추정, 경로 추종 및 장애물 회피, 그리고 로봇 제어 영역**으로 구성되었습니다. 각 파트는 독립적인 기능처럼 보이지만, 실제로는 관제자가 부여한 임무가 로봇의 인지·판단·제어 흐름으로 이어져야 했기 때문에 초기 단계에서 전체 시스템 구조와 파트 간 연결 방식을 명확히 정리하는 것이 중요했습니다.

### 1. 협업 및 시스템 구조 설계

프로젝트 초반에는 각자 맡은 기능을 빠르게 개발하기보다, 전체 시스템이 어떤 데이터 흐름으로 연결될지를 먼저 정리했습니다. 특히 로봇의 위치, 전역 경로, LiDAR 인지 결과, 지역 경로, 관제 화면 표시 정보가 서로 다른 기준으로 해석되면 통합 단계에서 문제가 발생할 수 있다고 판단했습니다.

그래서 ROS 2 기반으로 주요 topic, message, frame 구조를 먼저 정의했습니다. 로봇의 공통 지도 기준 좌표계는 `mission_map`으로 두고, 로봇 중심 좌표계는 `base_link`, LiDAR 좌표계는 `laser_frame`으로 구분했습니다. 이를 바탕으로 관제 시스템에서 할당한 전역 경로는 `mission_map` 기준으로 전달하고, LiDAR 기반 장애물 판단과 지역 회피는 로봇 중심인 `base_link` 기준에서 처리하도록 역할을 나누었습니다.

또한 실제 로봇과 가상 로봇이 함께 운용되는 구조였기 때문에, 특정 모듈이 아직 완성되지 않아도 다른 파트가 테스트를 진행할 수 있도록 mock publisher와 debug visualizer를 함께 구현했습니다. 예를 들어 localization이 완성되기 전에도 global path를 따라 이동하는 pose를 publish하는 mock node를 만들어 Navigation 파트를 검증할 수 있도록 했습니다. 해당 mock localization node는 global path를 입력받아 waypoint 사이를 선형 보간하고, `/localization/pose` 형태의 로봇 위치를 주기적으로 publish하도록 구현되어 있습니다.

### 2. LiDAR Perception 구현

로봇이 미지 환경에서 안전하게 이동하기 위해서는 주변 장애물과 주행 가능 영역을 인지하는 기능이 필요했습니다. 저희 팀은 LiDAR 데이터를 기반으로 로봇 전방의 장애물과 빈 공간을 판단하는 perception pipeline을 구현했습니다.

먼저 LiDAR에서 들어오는 `/scan_3D` PointCloud 데이터를 그대로 사용하지 않고, `pointcloud_preprocess_node`에서 NaN, Inf, zero point를 제거하고, `laser_frame` 기준 데이터를 `base_link` 기준으로 변환했습니다. 이후 로봇 주행에 필요한 전방 ROI만 남기고 voxel downsampling을 적용하여 계산량을 줄였습니다. 이 노드는 최종적으로 `/perception/lidar/points_filtered`를 publish하도록 구성했습니다.

다음으로 전처리된 point cloud를 기반으로 장애물 클러스터링을 수행했습니다. `obstacle_cluster_node`에서는 KD-Tree와 Euclidean clustering을 활용해 가까운 점들을 하나의 장애물 후보로 묶고, 각 cluster의 중심점, 최근접 거리, 방향각, bounding box 크기 등을 계산했습니다. 또한 cluster가 로봇 기준 front, left, right 중 어느 영역을 점유하는지 sector mask로 분류하여, 이후 판단 계층에서 장애물 방향을 쉽게 활용할 수 있도록 했습니다. 이 노드는 `/perception/lidar/obstacle_clusters`와 RViz 디버깅용 colored point cloud를 publish하도록 구현했습니다.

이후 `obstacle_model_node`에서는 여러 cluster 중에서 front, left, right 섹터별 대표 장애물을 선택했습니다. 모든 cluster 정보를 그대로 판단 계층에 넘기면 downstream node가 불필요하게 복잡해질 수 있기 때문에, 각 섹터에서 가장 가까운 대표 장애물만 요약하여 `/perception/lidar/obstacle_model`로 publish했습니다. 이를 통해 Navigation에서는 복잡한 point cloud 대신, “전방에 장애물이 있는지”, “좌우 어느 방향이 더 위험한지”를 단순한 형태로 판단할 수 있게 되었습니다.

### 3. Local Occupancy Grid와 Free Space 판단

장애물 cluster만으로는 “어느 방향으로 지나갈 수 있는가”를 판단하기 어렵습니다. 그래서 저희 팀은 LiDAR point cloud를 2D local occupancy grid로 변환하고, 이를 기반으로 주행 가능한 후보 방향을 계산하는 구조를 추가했습니다.

`local_occupancy_grid_node`에서는 전처리된 LiDAR point cloud를 `base_link` 기준의 2D grid로 변환했습니다. grid cell은 unknown, free, occupied 값으로 구분했고, LiDAR ray tracing을 통해 센서에서 장애물 endpoint까지의 경로는 free cell로 표시했습니다. 반대로 실제 point가 존재하는 endpoint는 occupied cell로 표시하고, 로봇이 장애물에 너무 가깝게 지나가지 않도록 inflation도 적용했습니다. 이 노드는 `/perception/lidar/local_occupancy_grid`를 publish합니다.

그 다음 `free_space_model_node`에서는 local occupancy grid를 angular bin 단위로 분석했습니다. 로봇 전방 LiDAR FOV를 여러 각도 구간으로 나누고, 각 bin마다 free, unknown, occupied 비율과 가장 가까운 장애물 거리를 계산했습니다. 이후 blocked 여부를 판단하고, 연속된 candidate bin들을 FreeSpaceGap으로 묶어 가장 안전한 주행 후보 방향을 선택했습니다. 최종적으로 selected gap, best heading, clearance, risk level 등을 포함한 `/perception/lidar/free_space_model`을 publish했습니다.

이 구조를 통해 로봇은 단순히 “장애물이 있다/없다”를 판단하는 수준을 넘어, 현재 시점에서 어느 방향으로 이동하는 것이 비교적 안전한지 판단할 수 있게 되었습니다.

### 4. Localization 및 로봇 위치 표현

관제 시스템에서 로봇을 지도 위에 표시하고, Navigation이 전역 경로를 추종하기 위해서는 로봇의 현재 위치가 필요했습니다. 저희 팀은 실내 환경에서 GPS를 사용할 수 없다는 점을 고려하여, 프로젝트 기준 지도 좌표계인 `mission_map`을 정의하고, 로봇의 pose를 이 좌표계 기준으로 표현하는 구조를 설계했습니다.

실제 로봇에서는 STM으로부터 보행 상태, gait cycle, IMU yaw 정보를 받아 Jetson에서 odometry를 추정하는 방향으로 설계했습니다. 이를 통해 로봇이 전진, 후진, 회전 등의 motion state를 수행할 때 cycle 단위 이동량과 yaw 변화를 누적하여 현재 위치를 계산하도록 했습니다.

또한 Navigation과 관제 UI가 localization 개발 완료 전에도 테스트될 수 있도록 mock localization pose publisher를 구현했습니다. 이 노드는 global path를 수신한 뒤, waypoint 사이를 일정 속도로 이동하는 pose를 생성하여 `/localization/pose`로 publish합니다. 이를 통해 실제 로봇이 움직이지 않아도 관제 화면과 Navigation pipeline의 동작을 검증할 수 있었습니다.

### 5. Navigation 및 Local Path Planning

Navigation 파트에서는 관제자가 할당한 global path를 그대로 따라가는 것뿐 아니라, 현재 로봇 위치 기준으로 path 진행 상태를 계산하고, LiDAR 인지 결과를 반영해 지역적으로 안전한 방향을 선택하는 구조를 구현했습니다.

먼저 `path_progress_tracker_node`에서는 global path와 localization pose를 입력으로 받아 현재 로봇이 경로상 어디까지 진행했는지 계산했습니다. 현재 pose와 가장 가까운 waypoint를 `nearest_index`로 찾고, lookahead 거리만큼 앞선 waypoint를 `target_index`로 선택했습니다. 또한 target heading, target yaw, heading error, yaw error, goal reached 여부를 계산하여 `/navigation/path_progress/spot_01`로 publish했습니다.

이 정보를 기반으로 Local Path Planner는 전역 경로를 따라가야 하는 방향과 LiDAR 기반 free space 후보 방향을 함께 고려했습니다. 전방이 비어 있는 경우에는 global path target을 따라가고, 장애물이나 위험 구간이 감지되는 경우에는 free space model에서 선택한 best heading을 활용해 회피 경로를 생성하는 방식으로 설계했습니다.

특히 이 과정에서 중요했던 점은 “장애물이 없을 때도 정상 상태를 계속 publish해야 한다”는 것이었습니다. 실제 테스트 중 장애물이 감지되지 않는 정상 상황에서 perception node가 아무 메시지도 publish하지 않으면, Navigation 입장에서는 이를 “장애물이 없음”이 아니라 “센서 데이터가 끊김”으로 해석할 수 있었습니다. 그래서 빈 장애물 상태도 명시적으로 publish하도록 구조를 개선했고, downstream node가 정상 상황과 timeout 상황을 구분할 수 있도록 했습니다.

### 6. 통합 테스트 및 시각화

로봇 시스템은 개별 node가 정상 동작하는 것만으로는 충분하지 않았습니다. LiDAR frame이 잘못되거나, pose frame이 다르거나, global path와 localization 기준 좌표계가 일치하지 않으면 전체 Navigation이 흔들릴 수 있었습니다.

그래서 저희 팀은 RViz 기반 debug visualization을 적극적으로 활용했습니다. LiDAR point cloud, obstacle cluster, local occupancy grid, free space gap, global path, localization pose, target point, local path를 함께 시각화하며 각 계층의 출력이 의도한 기준 좌표계에서 나타나는지 확인했습니다.

또한 실제 로봇 테스트가 어려운 상황에서도 mock global path와 mock localization pose를 활용해 전체 pipeline을 먼저 검증했습니다. 이후 실제 LiDAR 데이터와 localization 데이터를 연결하면서 perception, localization, navigation이 하나의 흐름으로 연결되는지 반복적으로 확인했습니다.

### 7. 고전제어와 강화학습 기반 4족 보행 제어

4족 로봇이 정찰 임무를 수행하기 위해서는 경로를 계산하는 것뿐 아니라, 계산된 이동 명령을 실제 다리 움직임으로 안정적으로 바꾸는 보행 제어가 필요했습니다. 저희 시스템에는 이를 위해 고전제어 기반 보행과 강화학습 기반 보행 정책을 모두 포함했습니다.

고전제어 기반 보행은 로봇의 기본 자세 유지와 정형적인 trot 보행을 담당하는 기능입니다. Spot Micro의 다리 구조를 기준으로 발끝 위치를 관절 각도로 변환하고, 대각선 다리 쌍이 번갈아 움직이는 trot 패턴을 통해 기본적인 전진 보행이 가능하도록 구성했습니다.

강화학습 기반 보행은 다양한 환경에 더 유연하게 대응하기 위한 기능입니다. 자율주행 모듈에서 전달하는 전진, 좌우 이동, 회전 명령을 바탕으로 로봇이 12개 관절의 목표 자세를 생성하도록 학습된 정책을 활용했습니다. 학습 과정에서는 평지 보행을 먼저 안정화한 뒤 자세 안정성, 동작 부드러움, 외란 대응 등을 점진적으로 반영해 실제 로봇 적용 가능성을 높이고자 했습니다.

## [프로젝트 후기]

이번 프로젝트를 진행하며 가장 크게 배운 점은 **로봇 시스템에서 중요한 것은 개별 기능의 완성이 아니라, 각 기능이 하나의 흐름으로 안정적으로 연결되는 것**이라는 점이었습니다.

LiDAR perception이 아무리 잘 동작해도 해당 결과가 Navigation에서 해석 가능한 message 구조로 정리되지 않으면 실제 주행 판단에 활용되기 어렵습니다. Localization이 pose를 publish하더라도 global path와 frame 기준이 다르면 경로 추종은 불안정해집니다. 또한 Navigation이 local path를 생성하더라도 관제 시스템에서 로봇의 상태를 확인할 수 없다면 다중 로봇 운용 시스템으로서의 의미가 줄어듭니다.

그래서 저희 팀은 프로젝트가 진행될수록 단순히 “기능을 하나 더 만드는 것”보다, topic, message, frame, 상태값, timeout 조건을 명확히 정의하는 것이 더 중요하다는 것을 체감했습니다. 특히 실제 로봇과 센서를 다루는 과정에서는 정상 상황, 데이터 없음, 센서 지연, frame mismatch, 장애물 없음과 같은 상태를 구분해서 처리해야 했고, 이러한 예외 상황을 설계에 반영하는 과정이 프로젝트 완성도를 높이는 데 큰 도움이 되었습니다.

협업 측면에서도 많은 것을 배웠습니다. 관제 UI, 디지털트윈, perception, localization, navigation은 서로 다른 기술 영역이었지만, 결국 하나의 시나리오 안에서 연결되어야 했습니다. 각자 맡은 파트만 이해하는 것으로는 충분하지 않았고, 내가 만든 topic이 어떤 node에서 사용되는지, 내 출력 메시지가 관제 화면이나 주행 판단에 어떤 의미로 해석되는지까지 함께 이해해야 했습니다. 예시 수기에서도 여러 도메인이 하나의 프로세스로 연결되기 위해 빠른 공유와 소통이 중요했다고 설명하는데, 저희 프로젝트 역시 동일한 부분을 가장 크게 체감했습니다.

또한 프로젝트를 통해 디지털트윈의 장점도 경험할 수 있었습니다. 실제 로봇은 테스트 환경, 배터리, 하드웨어 상태, 센서 연결 문제 등으로 반복 실험에 제약이 있습니다. 하지만 mock data와 가상 환경을 함께 활용하면 실제 로봇 투입 전에도 global path, localization, navigation, 관제 표시 흐름을 반복적으로 검증할 수 있었습니다. 이를 통해 실제 테스트에서 발생할 수 있는 문제를 사전에 줄이고, 통합 과정의 시행착오를 줄일 수 있었습니다.

후배 교육생들에게 전하고 싶은 팁이 있다면, 로봇 프로젝트에서는 초기에 **“무엇을 만들 것인가”보다 “어떤 데이터가 어떤 기준으로 흐를 것인가”를 먼저 정리하는 것**을 추천하고 싶습니다. 기능 구현을 빠르게 시작하는 것도 중요하지만, frame, topic, message, node 역할이 명확하지 않으면 개발 후반부에 다시 구조를 수정해야 하는 일이 많아집니다. 예시 수기에서도 초기 기획과 요구사항 명세의 중요성을 강조하고 있는데, 저희 팀 역시 이 부분을 프로젝트 과정에서 크게 공감했습니다.

로봇 하드웨어 및 펌웨어 설계
로봇은 Spot Micro 기반 4족 보행 구조로 설계했습니다. Jetson Orin Nano는 ROS2 Humble 기반의 인식, 측위, 경로 계획, 행동 결정을 담당하고, STM32F446RE는 실시간 관절 제어를 담당하도록 역할을 분리했습니다.

STM32는 12개의 STS3215 서보를 다리 단위로 나누어 제어하기 위해 4개의 독립 HDSEL UART 버스를 사용했습니다. 단일 버스로 모든 서보를 제어하면 500Hz 제어 주기를 맞추기 어렵기 때문에, 다리별 병렬 통신 구조를 선택했습니다.
또한 Jetson과 STM32 간에는 UART 통신을 사용하여 상위 명령과 상태 피드백을 주고받도록 구성했습니다.

이 과정에서 단순히 보행이 되는 로봇을 만드는 것보다, 재난 현장에서 안정적으로 동작할 수 있는 제어 구조를 만드는 것이 중요했습니다.
Jetson이 인식과 판단을 맡고, STM32가 실시간 제어와 IMU 기반 안정화를 맡도록 분리한 덕분에 상위 판단과 하위 제어를 명확하게 나눌 수 있었습니다.

ROS2 기반 자율 정찰 시스템
Jetson의 ROS2 워크스페이스는 perception, localization, navigation, control, network, bringup 등으로 나누어 구성했습니다.
카메라 인식 패키지는 OAK 카메라의 RGB 스트림을 구독하고 YOLOv11 기반으로 인명 탐지를 수행합니다.
LiDAR perception 패키지는 raw point cloud를 base_link 기준으로 변환한 뒤 장애물 클러스터, local occupancy grid, free-space model을 생성합니다.

특히 LiDAR 기반 free-space model은 로봇이 주변의 어느 방향으로 이동 가능한지 판단하는 데 중요한 역할을 합니다.
단순히 장애물 유무만 판단하는 것이 아니라, angular bin과 gap extraction 구조를 통해 통과 가능한 후보 공간을 계산하고, 후속 navigation 계층이 사용할 수 있는 정보를 제공합니다.

Navigation 계층에서는 로봇의 현재 위치, 장애물 정보, free-space 정보를 바탕으로 정찰 경로를 따라 이동하고, 장애물 상황에 따라 상태를 전환할 수 있도록 FSM 구조를 사용했습니다.
이를 통해 저희가 목표로 한 "임무 기반 자율 정찰"의 기반을 마련했습니다.

다중 로봇 통신 및 관제 구조
재난 현장에서는 한 대의 로봇보다 여러 대의 로봇이 구역을 나누어 움직이는 것이 훨씬 효율적입니다.
이를 위해 Raspberry Pi 5 기반 브릿지 데몬과 Qt 관제 UI를 설계했습니다.

각 Jetson 로봇은 UDP 9000 포트를 통해 이미지, LiDAR, 오도메트리, 상태, 이벤트, 경로 데이터를 브릿지로 전송합니다.
브릿지 데몬은 분할되어 들어오는 이미지와 LiDAR 패킷을 재조립하고, 로봇별 POSIX 공유메모리인 /robot_bridge_N에 저장합니다.
Qt 대시보드는 이 공유메모리를 읽어 여러 로봇의 상태, 영상, LiDAR, 지도, 경로, 이벤트 로그를 표시합니다.

반대로 관제자가 이동, 정지, 임무 시작, 일시정지, 긴급정지 명령을 내리면 Qt 앱은 공유메모리 내부 command queue에 명령을 기록하고,
브릿지 데몬이 이를 UDP 9001 포트를 통해 로봇으로 전송합니다. 이 구조를 통해 로봇 데이터 수신, UI 표시, 명령 전송이 분리되어 안정적인 통합 관제가 가능해졌습니다.

AI 학습 및 엣지 배포
인명 탐지는 YOLO 기반 비전 모델을 사용하고, Jetson 환경에서는 TensorRT FP16 엔진을 사용해 추론할 수 있도록 구성했습니다.
모델은 ONNX로 export한 뒤 Jetson 런타임에 배포할 수 있도록 설계했습니다.
탐지 결과는 단일 프레임에 즉시 반응하지 않고, N프레임 연속 탐지 조건을 만족할 때 인명 탐지 이벤트를 발행하도록 하여 오탐을 줄이고자 했습니다.

보행 제어 측면에서는 강화학습 기반 보행 정책을 학습하고,
