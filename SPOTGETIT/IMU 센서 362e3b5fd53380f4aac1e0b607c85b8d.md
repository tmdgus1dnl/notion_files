에 가깝습니다. 전원 완전 차단 후 재부팅하면 깨질 수 있습니다.

고쳐야 하는 방향은 명확합니다. 현재 CubeMX형 main.c 구조를 유지하되 USER CODE 블록
안에 #include "imu_bno055.h"를 넣고, MX_I2C1_Init() 이후 control_loop_run() 전에

bno055_init_imuplus(&hi2c1)를 명시적으로 호출해야 합니다. CubeMX가 다시 돌더라도 안
날아가게 반드시 USER CODE 블록 안에 넣어야 합니다.

**개선안**

파악한 결론부터 말하면, 현재 구조는 STM이 IMU를 직접 읽고, 단위/축 보정까지 한 값을
Jetson으로 보냅니다. Jetson/RL은 그 값을 거의 그대로 observation에 씁니다.

흐름은 이렇습니다.

1. STM32 → BNO055
    - firmware/Src/i2c.c:40: I2C1, 400 kHz.
    - firmware/Src/imu_bno055.c:96: BNO055에서 0x08부터 32바이트 burst read.
    - 읽는 값:
        - accel raw
        - gyro raw
        - Euler yaw/pitch/roll
        - quaternion
2. STM 내부 보정
    - firmware/Src/imu_bno055.c:131: yaw/pitch/roll sign/축 보정.
    - firmware/Src/imu_bno055.c:145: gyro를 rad/s로 변환, Y축 sign flip.
    - firmware/Src/imu_bno055.c:155: accel을 m/s²로 변환.
    - firmware/Src/imu_bno055.c:173: quaternion normalize.
3. STM telemetry
    - firmware/Src/telemetry.c:82: telemetry_read_imu()가 bno055_read_body(&hi2c1,
    &tmp) 호출.
    - 성공하면 firmware/Src/telemetry.c:89: g_robot_state.imu = tmp.
    - 단, 여기에는 숫자 spike 필터는 없음. I2C read 실패만 3-cycle hysteresis로 처
    리합니다.
4. STM → Jetson
    - firmware/Src/spi_protocol.c:143: feedback frame 261B 생성.
    - firmware/Src/spi_protocol.c:172: yaw rad.
    - firmware/Src/spi_protocol.c:188: gyro.
    - firmware/Src/spi_protocol.c:191: accel.
    - firmware/Src/spi_protocol.c:194: quaternion.
    - 실제 transport는 현재 SPI가 아니라 UART입니다. USART1 921600 baud: firmware/
    Src/usart.c:110.
5. Jetson ROS2
    - robot_ws/src/control/actuator_bridge/src/actuator_bridge_node.cpp:351:
    feedback을 sensor_msgs/Imu로 publish.
    - topic: /control/actuator/imu.
    - robot_ws/src/control/rl_locomotion/rl_locomotion/rl_locomotion_node.py:438:
    RL node가 이 IMU를 subscribe.
    - robot_ws/src/control/rl_locomotion/rl_locomotion/rl_locomotion_node.py:443:
    angular velocity는 그대로 base_ang_vel.
    - robot_ws/src/control/rl_locomotion/rl_locomotion/rl_locomotion_node.py:455:
    accel을 읽어서 norm으로 나눈 뒤 projected_gravity = -accel / |accel|.
    - robot_ws/src/control/rl_locomotion/rl_locomotion/obs_builder.py:10: RL
    observation의 0:3이 gyro, 3:6이 projected gravity입니다.

중요한 문제 하나 보입니다. 현재 firmware/ 기준으로 bno055_init_imuplus() 호출이 없
습니다. 함수 정의는 있는데 호출 검색 결과가 없습니다. main.c는 peripheral init 후

바로 control_loop_run()으로 들어갑니다. 그래서 BNO055가 명시적으로 IMUPLUS 모드/

axis remap으로 초기화되지 않을 가능성이 있습니다. 이건 STM 쪽에서 먼저 고쳐야 할 항  목입니다.

판단은 이렇게 하는 게 맞습니다.

- STM에서 해야 할 것
    - BNO055 init 호출 누락 수정.
    - IMU read 실패, NaN/Inf, quaternion norm 이상, accel norm 비정상 같은 hard
    validation.
    - safety에 쓰는 pitch/roll은 STM에서 계속 관리.
- Jetson에서 먼저 만져볼 것
    - RL observation용 IMU filtering.
    - 특히 지금 RL은 quaternion이 아니라 raw accel 기반 projected gravity를 씁니다. 보행 중 linear acceleration이 섞이면 gravity 방향이 튀어서 policy가 우왕좌왕
    할 수 있습니다.
    - 빠른 실험으로는 Jetson imu_callback()에서 accel 기반 projected gravity에 low-
    pass/outlier reject를 넣거나, quaternion 기반 projected gravity로 바꿔 비교하 는 게 좋습니다.

내 의견은: 초기화 누락은 STM에서 바로 수정해야 하고, 보행 중 튐 억제/정책 안정화는
Jetson RL 입력단에서 먼저 실험하는 게 안전합니다. STM에서 무겁게 필터를 걸면

safety/telemetry 전체 의미가 바뀌는데, Jetson 쪽 필터는 RL observation만 바꿔서 A/B
테스트하기 쉽습니다.
