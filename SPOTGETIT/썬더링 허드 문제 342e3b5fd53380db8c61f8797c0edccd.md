# 썬더링 허드 문제

**하나의 이벤트가 발생했을 때, 잠들어 있던 여러 프로세스나 스레드가 한꺼번에 깨어나서 동시에 같은 자원에 달려드는 문제**

예를 들어 설명하면 이렇습니다.

서버에 여러 워커 스레드가 있고, 모두 `accept()` 같은 호출로 새 클라이언트 연결을 기다리고 있다고 하겠습니다.

이때 클라이언트가 딱 1개만 접속했는데, 운영체제가 기다리던 스레드들을 여러 개 동시에 깨워버리면:

- 깨어난 스레드들은 모두 “내가 처리할 수 있나?” 하고 경쟁합니다.
- 하지만 실제로 처리할 수 있는 연결은 1개뿐입니다.
- 결국 1개만 성공하고, 나머지는 다시 실패하고 잠듭니다.

즉, **일은 하나인데 여러 놈이 같이 뛰어드는 바람에 CPU만 헛되이 쓰는 상황**입니다.

---

## 왜 문제인가

핵심은 **불필요한 깨어남**입니다.

원래는 한 스레드만 깨우면 되는데 여러 개를 깨우면:

- 문맥 전환(context switch)이 많이 발생하고
- 락 경쟁(lock contention)이 심해지고
- CPU 캐시 효율이 떨어지고
- 전체 처리량은 오히려 줄어들 수 있습니다
- 응답 지연도 커질 수 있습니다

즉, 겉보기에는 “여러 스레드가 있으니 더 빠를 것 같지만”, 실제로는 **서로 발목을 잡는 상태**가 됩니다.

---

## 이름이 왜 Thundering Herd인가

직역하면 “천둥처럼 몰려드는 무리” 정도입니다.

조용히 기다리던 여러 스레드나 프로세스가

이벤트 하나를 듣자마자 **우르르 몰려드는 모습**을 비유한 표현입니다.

---

## 어디서 자주 생기나

대표적인 경우는 다음과 같습니다.

### 1. 서버 소켓의 `accept()`

여러 워커가 같은 리스닝 소켓에서 연결을 기다릴 때 발생할 수 있습니다.

### 2. `epoll`, `select`, `poll` 같은 이벤트 대기

하나의 FD 이벤트에 대해 여러 스레드가 동시에 깨어날 수 있습니다.

### 3. 락이나 조건 변수

어떤 자원이 준비되었을 때 실제로는 한 스레드만 처리하면 되는데 여러 스레드를 한꺼번에 깨워버리는 경우입니다.

### 4. 캐시 만료

웹 서버나 DB에서 특정 인기 데이터가 캐시에서 동시에 만료되면, 많은 요청이 한꺼번에 원본 DB로 몰릴 수 있습니다.

이것도 넓은 의미에서 비슷한 종류의 문제로 봅니다.

---

## 아주 간단한 그림으로 보면

이벤트 1개 발생:

- 스레드 A 깨어남
- 스레드 B 깨어남
- 스레드 C 깨어남
- 스레드 D 깨어남

그런데 실제 처리 가능한 작업은 1개뿐:

- A만 성공
- B, C, D는 헛수고

이 헛수고가 반복되면 성능이 크게 나빠집니다.

---

## 해결 방법

### 1. 하나만 깨우기

가능하면 이벤트 하나에 대해 **정말 필요한 스레드만 하나 깨우는 방식**이 좋습니다.

예를 들어 조건 변수에서 `broadcast` 대신 `signal`을 쓰는 식입니다.

- `signal`: 하나만 깨움
- `broadcast`: 전부 깨움

일이 하나뿐이면 보통 `signal`이 더 적절합니다.

---

### 2. 커널/이벤트 시스템의 독점 웨이크업 사용

리눅스에서는 이런 문제를 줄이기 위한 기능들이 있습니다.

예를 들어 `epoll`에서는 상황에 따라 **exclusive wakeup** 계열 방식으로 여러 대기자 중 일부만 깨우는 방법을 사용합니다.

핵심은

“이벤트 하나당 대기자 하나만 깨워라” 입니다.

---

### 3. 워커 구조를 바꾸기

여러 스레드가 같은 자원을 직접 기다리지 말고,

- 하나의 스레드가 이벤트를 받음
- 그 스레드가 작업 큐에 넣음
- 워커들이 큐에서 가져감

이런 구조로 바꾸면 경쟁이 훨씬 줄어듭니다.

즉, **이벤트 감지와 작업 처리를 분리**하는 방식입니다.

---

### 4. 락 범위를 줄이기

여러 스레드가 한 자원을 두고 경쟁하지 않도록

- 공유 자원 최소화
- 락 잡는 시간 최소화
- 락을 더 잘게 분리

같은 방식도 도움이 됩니다.

다만 이건 썬더링 허드 자체를 없애기보다는, 발생했을 때 피해를 줄이는 쪽에 가깝습니다.

---

### 5. 캐시 만료 분산

캐시 계층에서 생기는 썬더링 허드 비슷한 문제는

- TTL을 랜덤하게 주기
- 요청 병합(request coalescing)
- 한 요청만 원본 조회하고 나머지는 기다리게 하기

같은 방법으로 줄입니다.

---

## 한 줄로 핵심만 말하면

**일은 적은데, 기다리던 여러 실행 주체가 한꺼번에 깨어나서 동시에 달려들며 자원을 낭비하는 문제**입니다.

---

## 면접이나 문서용으로 짧게 쓰면

“썬더링 허드 문제는 하나의 이벤트에 대해 여러 프로세스나 스레드가 동시에 깨어나 동일한 자원에 경쟁적으로 접근하면서, 실제 처리량보다 불필요한 컨텍스트 스위칭과 락 경쟁이 증가해 성능이 저하되는 현상입니다.”
