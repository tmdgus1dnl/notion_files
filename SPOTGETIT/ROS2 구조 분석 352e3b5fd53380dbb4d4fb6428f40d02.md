# ROS2 구조 분석

상태: 기타

## 1. ROS2 기본 DDS discovery 환경

bashrc:

```bash
source /opt/ros/humble/setup.bash
source ~/robot_ws/install/setup.bash

export ROS_DOMAIN_ID=0
export ROS_LOCALHOST_ONLY=0

unset RMW_IMPLEMENTATION
unset FASTRTPS_DEFAULT_PROFILES_FILE
unset CYCLONEDDS_URI
```

- `ROS_DOMAIN_ID=0` → ROS2 통신 그룹을 0번으로 통일
- `ROS_LOCALHOST_ONLY=0` → Jetson 내부뿐 아니라 외부 네트워크 ROS2 통신도 허용
- `unset RMW_IMPLEMENTATION` → 특정 DDS 구현체를 강제하지 않음 → Humble 기본 RMW 사용
- `unset FASTRTPS_DEFAULT_PROFILES_FILE` → 사용자 정의 FastDDS XML을 강제로 적용하지 않음
- `unset CYCLONEDDS_URI` → CycloneDDS 사용자 설정도 강제로 적용하지 않음

즉, 기본 DDS discovery 환경은 Jetson **내부에서 실행되는 노드들이 자연스럽게 서로를 찾을 수 있다**.

## 2. Digital Twin의 목적

![image.png](image%2032.png)

단순히 Isaac Sim에 로봇 모델을 띄우는 게 아니라, **실제 로봇 상태가 Isaac Sim 안의 Spot 엔티티에 반영되는 구조**

예를 들면 실제 로봇이 북쪽을 바라보면:

```
실제 Spot Micro / 로봇
  heading yaw = north 방향
        ↓
Jetson ROS2 state publish
        ↓
Isaac Sim ROS2 Bridge subscribe
        ↓
Isaac Sim 안의 Spot prim/entity orientation 갱신
```

이 구조에서 Jetson은 **“실제 로봇 상태의 source”**가 돼야 하고, 

Jetson에서 Isaac Sim으로 보내야 할 대표 데이터는 아래와 같다.

```
1. 로봇 위치/자세
   /tf
   /odom
   /robot_pose
   geometry_msgs/PoseStamped
   nav_msgs/Odometry

2. 로봇 관절 상태
   /joint_states
   sensor_msgs/JointState

3. 센서 데이터
   /scan
   /scan_3D
   /perception/lidar/points_filtered
   /camera/image_raw

4. 상태 정보
   battery
   mode
   velocity
   STM 상태
   motor 상태
```

Isaac Sim에서는 이 데이터를 받아서 **Spot 모델의 위치, 방향, 관절, 센서 시각화를 업데이트**하는 것으로 아래의 흐름으로 연동되어야 함

```bash
Isaac Sim ROS2 Bridge도 하나의 ROS2/DDS participant가 됨
Jetson ROS2 노드들과 같은 ROS_DOMAIN_ID 안에 들어옴
DDS discovery를 통해 토픽을 찾음
토픽을 subscribe/publish함
```

즉, Jetson과 Isaac Sim PC가 **같은 ROS2 DDS graph** 안에 들어와야 함

## 3. Digital Twin을 위한 ROS2 Framwork 기본 배경 지식

```bash
ROS2 graph
= ROS2 사용자 관점
= node/topic/service/action 중심

DDS graph
= 통신 미들웨어 관점
= participant/writer/reader/discovery/QoS 중심
```

비유하면:

```
ROS2 graph = 지하철 노선도
DDS graph = 실제 선로, 신호 시스템, 열차 제어 시스템
```

### 3-1. ROS graph란?

> ROS2 노드, 토픽, 서비스, 액션들이 서로 어떻게 연결되어 있는지 보는 논리적 연결망
> 

ROS graph는 현재 ROS2 시스템 안에서 노드 간의 정보들을 나타내는 논리적 연결망 :

```
어떤 노드가 있고
어떤 토픽이 있고
누가 publish하고
누가 subscribe하는지
```

우리가 치는 이 명령어들은 **ROS graph를 보기 위한 명령어**이다.

```
ros2 node list
ros2 topic list
ros2 topic info /perception/lidar/points_filtered--verbose
```

예를 들어 이런 출력이 있었다면:

```
/perception/lidar/points_filtered

Publisher:
  pointcloud_preprocess_node

Subscriber:
  obstacle_cluster_node
  rviz
```

이건 ROS graph상으로 이런 구조가 잡혔다는 뜻이다.

```
pointcloud_preprocess_node
        ↓
/perception/lidar/points_filtered
        ↓
obstacle_cluster_node
        ↓
			rviz
```

### 3-2. DDS graph란?

> ROS2 내부 노드들이 서로 통신하기 위해 사용되는 통신 미들웨어(DDS)
> 

DDS :

```bash
ROS2 노드들이 네트워크에서 서로를 찾고,
Publisher와 Subscriber를 매칭하고,
실제 데이터를 주고받게 해주는 통신 계층
```

즉, ROS2에서 너는 이렇게 생각하지만:

```
obstacle_cluster_node가 /obstacle_clusters를 publish한다
obstacle_model_node가 /obstacle_clusters를 subscribe한다
```

실제로 밑에서는 DDS가 이렇게 처리해.

```
DDS Participant A:
  "나는 /obstacle_clusters를 publish하는 writer가 있어"

DDS Participant B:
  "나는 /obstacle_clusters를 subscribe하는 reader가 있어"

**DDS Discovery:
  "둘이 topic 이름, type, QoS가 맞네. 연결해줄게."**

DDS Data Transport:
  "이제 실제 메시지를 UDP로 전달할게."
```

여기서 말한 DDS graph는 DDS participant, writer, reader들이 서로를 **1) discovery**하고 **2) matching**한 네트워크-level 연결 구조이다.

### 3-3. DDS Discovery란?

> Discovery는 ROS2 노드들이 서로를 자동으로 찾는 과정
> 

예를 들어 Jetson에서:

```
obstacle_cluster_node 실행
obstacle_model_node 실행
```

하면, 둘이 처음부터 서로의 존재를 아는 것이 아님

DDS가 네트워크에서 아래와 같이 찾는다.

```
"나 obstacle_cluster_node야. /obstacle_clusters publish 가능해."
"나 obstacle_model_node야. /obstacle_clusters subscribe하고 싶어."
"둘이 같은 ROS_DOMAIN_ID이고, topic/type/QoS가 맞네. 연결."
```

이 과정이 discovery이다.

### 3-4. ROS2 통신 구조 변수(domain, localhostonly, RMW)

`ROS_DOMAIN_ID` : ROS2 통신 그룹 번호

```
exportROS_DOMAIN_ID=0
```

**같은 Domain ID에 있는 노드들끼리만 서로를 발견**해.

```
Jetson ROS_DOMAIN_ID=0
Isaac Sim PC ROS_DOMAIN_ID=0
→ 서로 discovery 가능

Jetson ROS_DOMAIN_ID=0
Isaac Sim PC ROS_DOMAIN_ID=7
→ 서로 안 보임
```

그래서 Jetson과 Isaac Sim을 연결하려면 양쪽이 같은 값을 써야 한다다.

---

`ROS_LOCALHOST_ONLY`:  ROS2 통신을 같은 장비 내부로만 제한할지 정하는 변수

```
exportROS_LOCALHOST_ONLY=0
```

의미는:

```
0 → 외부 네트워크 통신 허용
1 → localhost 내부 통신만 허용
```

Jetson과 Windows Isaac Sim PC를 연결하려면:

```
ROS_LOCALHOST_ONLY=0
```

만약 1이면 Jetson 내부 노드끼리는 되지만, 외부 Isaac Sim PC와는 안 붙어.

---

`RMW` : ROS2는 DDS를 직접 쓰는 게 아니라, 중간에 RMW라는 추상 계층을 사용함.

```
ROS2 node
  ↓
rclcpp / rclpy
  ↓
 RMW
  ↓
DDS implementation
  ↓
Network
```

RMW는 “**ROS Middleware Interface**”라고 보면 돼.

DDS 구현체는 여러 개가 있을 수 있어.

```
FastDDS
CycloneDDS
ConnextDDS
```

`RMW_IMPLEMENTATION`을 지정하면 어떤 DDS 구현체를 쓸지 강제할 수 있어.

```
exportRMW_IMPLEMENTATION=rmw_fastrtps_cpp
```

현재 네가 정상화한 bashrc에서는 이걸 unset 했지.

```
unset RMW_IMPLEMENTATION
```

의미 : 

- 특정 DDS 구현체를 강제로 지정하지 않는다.
- 현재 **ROS2 Humble 기본 설정(eProsima Fast DDS)**을 사용하겠다.
