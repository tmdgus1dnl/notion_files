# SpotMicro 펌웨어 아키텍처 — 큰 그림

## 1. 코드의 본질

펌웨어가 하는 일을 한 문장으로:

> STM32가 4개의 UART 버스를 통해 12개의 STS3215 서보와 통신하면서, 다리들을 의도한 자세로 부드럽게 움직이는 것
> 

모든 코드가 이 한 가지를 위한 것.

---

## 2. 구조의 4계층

코드는 자연스럽게 4개 층으로 나뉨:

```
┌─────────────────────────────────────────┐
│  Layer 4: 시나리오 (main 함수)            │
│  "PING → Torque ON → 자세 전환 → OFF"    │
├─────────────────────────────────────────┤
│  Layer 3: 모션 (자세 + 점진 전환)         │
│  "현재 → 영점, 부드럽게 2초 동안"          │
├─────────────────────────────────────────┤
│  Layer 2: 프로토콜 (SCS 통신)             │
│  "PING / READ / WRITE / SYNC_WRITE"      │
├─────────────────────────────────────────┤
│  Layer 1: HAL (펌웨어 인프라)             │
│  "UART 초기화, 클럭 설정, GPIO"           │
└─────────────────────────────────────────┘
```

각 층은 아래 층의 함수만 호출하고, 위 층은 모름.

### Layer 1 — 인프라

하드웨어 초기화. STM32CubeMX 같은 도구가 자동 생성하는 부분과 비슷한 성격. 한 번 만들고 거의 안 건드리는 boilerplate.

```c
SystemClock_Config()           // 180 MHz PLL 설정
MX_USART2_Init()               // 디버그 UART (115200)
MX_UART4_HDSEL_Init()          // FR 다리 통신 (1 Mbps half-duplex)
MX_USART6_HDSEL_Init()         // FL 다리 통신
MX_USART3_HDSEL_Init()         // RR 다리 통신
MX_UART5_HDSEL_Init()          // RL 다리 통신
```

### Layer 2 — 프로토콜

바이트 묶음을 만들고, 보내고, 답을 받아 해석하는 함수들.

```c
sts_ping(huart, id)              // "거기 있어?"
sts_read_state(huart, id)        // "지금 위치/속도/부하 어때?"
sts_write_byte(huart, id, ...)   // "이 레지스터에 1바이트 써"
sts_sync_write_goal(...)         // "여러 서보에 한꺼번에 명령"
```

핵심 책임: SCS 패킷 포맷을 정확히 만들고 응답을 정확히 파싱.

### Layer 3 — 모션

저수준 명령("서보 ID 1번에 raw 1877 보내")을 의미적 명령("FL 다리를 default 자세로")으로 바꿔주는 층.

```c
compute_default_for_leg(leg_idx, pose)   // 다리 한 개의 default 자세 raw 값 계산
apply_pose_leg(leg_idx, pose)            // 그 자세를 한 번에 명령 (SYNC_WRITE)
transition_leg(leg_idx, from, to)        // 두 자세 사이 부드러운 보간 (smoothstep)
```

여기서 sign table, HIP/THIGH/KNEE_OFFSET, smoothstep 같은 개념들이 등장. 다리의 의미적 자세를 표현.

### Layer 4 — 시나리오

`main()` 함수 한 군데. 실제 실행 흐름 정의.

```
1. HAL/UART 초기화
2. PING 12개 (다 살아있는지 확인)
3. 모든 서보 Torque OFF (안전 baseline)
4. 5초 카운트다운 (ESC abort 가능)
5. 다리 1개의 현재 위치 읽기
6. Goal = 현재 위치로 설정 (jump 방지)
7. 그 다리 Torque ON
8. 점진 전환 (현재 → 영점, 부드럽게)
9. 잠시 유지 (모니터링)
10. Torque OFF
11. 정지
```

여러 검증 코드들의 차이는 사실 이 시나리오만 다른 것. Layer 1~3은 거의 같음.

## 3. 직교하는 두 가지 관심사

메인 흐름과 수직으로 교차하는 두 가지 부가 시스템.

```
                    main 흐름
                    ──────────►

ESC 정지 시스템     ━━━━━━━━━━━━━━ (어디서든 키 체크)

디버그 출력 (printf)  ━━━━━━━━━━━━━━ (모든 단계 로그)
```

### ESC 정지

어떤 단계에서든 사용자가 ESC 누르면 즉시 모든 서보 토크 OFF + 무한 정지.

구현: 모든 `HAL_Delay()` 호출을 `delay_with_estop()`으로 교체.

```c
static void delay_with_estop(uint32_t ms) {
    uint32_t start = HAL_GetTick();
    while (HAL_GetTick() - start < ms) {
        if (check_esc()) emergency_stop();
        HAL_Delay(10);
    }
}
```

일종의 "언제든지 인터럽트 가능한 sleep". 메인 시나리오 코드를 깔끔하게 유지하면서도 안전성 확보.

### 디버그 출력

`printf`가 USART2로 redirect돼 있어서 PuTTY에서 모든 진행 상황 시각화. 디버깅의 핵심 도구.

```c
int _write(int file, char *ptr, int len) {
    HAL_UART_Transmit(&huart2, (uint8_t *)ptr, len, HAL_MAX_DELAY);
    return len;
}
```

## 4. 데이터의 흐름

코드 안에서 데이터가 어떻게 흐르는지:

```
┌────────────┐
│   상수     │   #define HIP_OFFSET, THIGH_OFFSET, KNEE_OFFSET
│            │   leg_sign[] = {+1, -1, +1, -1}
│ (의미적    │   ZERO_POS = 2048
│  정의)     │
└─────┬──────┘
      │
      ▼ compute_default_for_leg()
┌────────────┐
│   Pose     │   uint16_t pose[3] = { hip_raw, thigh_raw, knee_raw }
│   배열     │   예: FL default = { 2048, 1877, 2389 }
└─────┬──────┘
      │
      ▼ apply_pose_leg() / transition_leg()
┌────────────┐
│  SCS 패킷  │   { 0xFF 0xFF 0xFE 0x0D 0x83 0x2A 0x02 ... }
│   바이트   │   (SYNC_WRITE 패킷)
└─────┬──────┘
      │
      ▼ HAL_UART_Transmit()
┌────────────┐
│   UART     │   1 Mbps 시리얼 → 서보들에게
│  하드웨어  │
└────────────┘
```

추상화: 위쪽은 사람의 의도("다리를 굽혀"), 아래쪽은 하드웨어의 언어(바이트). 그 사이 변환을 Layer 2, 3가 담당.

## 5. 시간 차원의 흐름

공간적 구조 외에 시간적 흐름도 중요.

```
시간 →
─────────────────────────────────────────►

전원 ON
   │
   ├─ HAL/UART init                  (수십 ms)
   │
   ├─ printf "Boot"                  (~ms)
   │
   ├─ PING 12개                      (~수십 ms)
   │
   ├─ Torque OFF all                 (~수십 ms)
   │
   ├─ 5초 카운트다운                  (5000 ms)
   │
   ├─ Read current                   (~ms)
   │
   ├─ Torque ON                      (~ms)
   │
   ├─ Transition (80단계 × 25 ms)     ← 메인 작업
   │  ├─ step 1: SYNC_WRITE → 25 ms
   │  ├─ step 2: SYNC_WRITE → 25 ms
   │  ├─ ...
   │  └─ step 80: SYNC_WRITE → 25 ms  (총 2 s)
   │
   ├─ Hold 3 s (모니터링)             (3000 ms)
   │
   ├─ Torque OFF
   │
   └─ while(1) {}                    (영원히)
```

대부분 시간은 delay에서 보냄. CPU는 거의 idle. 현재 코드의 한계 — 인터럽트/DMA 안 쓰고 polling으로만 동작. 500 Hz 제어 루프 단계에서 깔끔히 갈아엎어질 부분.

## 6. 큰 그림 요약

```
┌─────────────────────────────────────────────┐
│                                             │
│   상수 (의미)  →  Pose (raw)  →  바이트     │
│                                             │
│   ↑                              ↓          │
│                                             │
│   사람의 의도 ←──────────── 서보의 PID      │
│                                             │
│   "다리 굽혀"                                │
│                                             │
└─────────────────────────────────────────────┘

      [printf]                   [ESC]
      디버그 시각화          긴급 정지
       어디서든                어디서든
```

핵심:

- **추상화 계층**: 의미 ↔ raw ↔ 바이트
- **메인 흐름**: 시나리오를 단순한 함수 호출 시퀀스로
- **부가 시스템**: 디버그 + 안전, 메인을 오염시키지 않고 직교
