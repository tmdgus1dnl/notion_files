## 1장 Hardware 꼴라쥬

**꼭 봐야 함:**

- ⓚ Register 넌 누구냐 — 디바이스 드라이버의 본질이 레지스터 읽기/쓰기인데, 레지스터가 뭔지 모르면 시작이 안 됩니다
- ⓛ Clock 이란 — 로봇의 모터 제어 주기, 통신 클록(CAN, EtherCAT)의 동작 원리를 이해하는 기초
- ⓜ Bus Transfer Mechanism — CAN이나 EtherCAT 같은 버스 기반 통신 프로토콜을 이해하려면 버스가 어떻게 데이터를 전달하는지 알아야 합니다
- ⓝ Timing 그리고 Spec 읽기 — 하드웨어 데이터시트를 읽는 능력. 센서/모터 IC의 스펙을 보고 드라이버를 작성하려면 필수
- ⓞ Memory의 선정과 XIP — 메모리 구조와 동작 원리의 기초
- ⓠ 확장 to the CPU - How CPU works — CPU가 명령어를 어떻게 실행하는지, 실시간 제어에서 명령어 실행 지연이 왜 생기는지 이해하는 근본
- ⓡ CPU의 동작 예와 Pipe line — 파이프라인 이해 없이는 실시간 성능 최적화를 논할 수 없습니다

**보면 좋음:**

- ⓑ 신호와 주파수 영역 — 센서 신호 처리의 배경 지식
- ⓒ Analog/Digital 신호, Ground — HW 엔지니어와 대화할 때 필요
- ⓖ Pull up, Pull down 그리고 Open Collector — GPIO 설정할 때 pull-up/pull-down이 뭔지 모르면 막힙니다
- ⓙ IC 기본 — 로봇에 들어가는 각종 IC를 이해하는 기초
- ⓟ RAM Memory의 물리적 동작 — DMA나 메모리 맵을 다룰 때 배경 지식

**나중에 봐도 됨:**

- ⓐ Hardware 꼴라쥬 (개론) — 가볍게 훑는 정도
- ⓓ 초간단 회로이론 R, L, C — SW 엔지니어가 직접 회로를 설계할 일은 없으므로 감만 잡으면 됨
- ⓔ 필터 — 같은 이유
- ⓕ 트랜지스터 1% — 같은 이유
- ⓗ RLC와 Transistor 感 — 같은 이유
- ⓘ 논리회로로의 확장 — CS 전공이면 이미 알고 있을 내용

## 2장 Microprocessor 아뜰리에

**꼭 봐야 함:**

- ⓓ ARM 동작 Modes — User/System/IRQ/FIQ/Supervisor 등 모드 전환이 인터럽트, 시스템콜, 예외 처리의 근간
- ⓔ ARM Register와 Context — Context Switching을 이해하려면 어떤 레지스터가 저장/복원되는지 알아야 합니다
- ⓕ ARM Exceptions and Modes — 실시간 제어에서 예외 발생 시 시스템이 어떻게 반응하는지의 원리
- ⓖ ARM/Thumb PCS - 레지스터 사용법 — 함수 호출 규약. 디버깅할 때 스택을 추적하려면 필수
- ⓗ ARM은 Interrupt냄새를 어떻게 맡는가 — 인터럽트 처리의 하드웨어 레벨 메커니즘. 실시간 제어 프레임워크의 가장 밑바닥
- ⓘ ARM SoC — 로봇 보드에 들어가는 SoC의 구조를 이해
- ⓙ AMBA — SoC 내부 버스 규격. 문c 블로그에서도 보조 항목으로 AMBA AXI를 다루고 있는데, 여기서 기초를 잡으면 나중에 이어집니다

**보면 좋음:**

- ⓑ ARM Inside — ARM 아키텍처 전반 개요
- ⓒ ARM mode와 THUMB mode — 바이너리 크기와 성능 트레이드오프를 이해

**나중에 봐도 됨:**

- ⓐ 마이크로 프로세서 아뜰리에 (개론) — 가볍게 훑기

## 3장 Software 데꾸바쮸

**꼭 봐야 함:**

- ⓘ 변수의 scope와 그 생애 (Memory Map) — 전역변수, 지역변수, static이 메모리 어디에 놓이는지. 시스템 프로그래밍의 기본 중 기본
- ⓚ Memory Map과 Symbol 이야기 — 커널 디버깅할 때 심볼 테이블을 읽고 주소를 추적하는 능력의 토대
- ⓛ ELF format Object File에 관한 진실 — 리눅스 커널과 디바이스 드라이버가 전부 ELF 포맷. 문c 블로그의 커널 이미지 분석과도 직결
- ⓜ Linker를 마무리 짓자 - ELF와 fromelf까지 — 링킹이 어떻게 되는지 모르면 빌드 에러를 해결할 수 없습니다
- ⓝ Scatter Loading/Linker Description Script와 메모리 다루기 — BSP 작업에서 링커 스크립트를 직접 수정해야 하는 경우가 많습니다. 커널의 vmlinux.lds도 이 원리
- ⓞ Map file 분석 — 메모리 사용량 분석, 심볼 위치 확인 등 디버깅과 최적화에 실전적으로 쓰임
- ⓟ Memory Map과 Linker의 만남 - Locator — 코드가 메모리 어디에 배치되는지 이해. 부트로더/BSP 작업의 핵심

**보면 좋음:**

- ⓒ 컴파일 공장 이야기 — 전처리→컴파일→어셈블→링크 전체 흐름 이해
- ⓓ 컴파일 순서에 따라 원하는 컴파일을 해보자 — 빌드 과정 제어
- ⓖ Library - 남한테 보여주기 싫어 / ⓗ Lib을 까보자 — 공유 라이브러리와 정적 라이브러리의 차이. ROS2 패키지가 라이브러리 형태로 배포되므로 알아두면 좋음
- ⓘ 컴파일러 Option들 — 최적화 옵션(-O2, -Os)이 실시간 성능에 미치는 영향을 이해
- ⓠ Makefile은 뭘 하는 녀석일까 — 커널도 ROS2도 결국 빌드 시스템이 있고, 그 근본이 Make
- ⓡ MACRO와 SUFFIX / ⓢ 조금 더 Make 테크닉들 — Makefile 읽기/수정 능력

**나중에 봐도 됨:**

- ⓐ Little Endian과 Big Endian — 한 번 읽고 개념만 잡으면 충분
- ⓑ 컴파일에 대한 단상 — 에세이 성격
- ⓔ Preprocess와 #Include — C 기본 문법 수준이라 이미 알고 있을 가능성 높음
- ⓕ Assembly로 만드는 방법 — 4장에서 더 깊게 다루므로 여기선 가볍게
- ⓣ Make Option들 — 레퍼런스 성격

## 4장 ARM 미장센

**꼭 봐야 함:**

- ⓖ Exception Vector Table과 각 Handler의 구현 — 리눅스 커널도 EVT를 설정하는 것에서 시작. 문c 블로그의 head.S 분석으로 이어지는 직접적인 선행 지식
- ⓙ Bootloader와 Memory Budget — BSP 작업의 핵심. U-Boot 같은 부트로더가 하는 일을 이해
- ⓚ Reset Handler에서 main까지 (Entry Point) — 전원이 켜지고 OS가 올라오기까지의 과정. 공고의 "운영체제 환경 구성"의 가장 밑바닥
- ⓛ Scatter Loading과 Bootup — 부팅 시 메모리 초기화 과정

**보면 좋음:**

- ⓑ 대충의 초간단 Assembly와 Reverse Engineering — 커널 크래시 로그를 읽으려면 어셈블리를 최소한 읽을 줄은 알아야 합니다
- ⓔ Inline Assembly와 INTLOCK의 구현 — 리눅스 커널 소스에 인라인 어셈블리가 자주 등장
- ⓕ Pipe line과 Exception 관계 그리고 ^접미사 — 실시간 시스템에서 예외 발생 시 파이프라인 상태 이해
- ⓗ SWI의 진실 — 시스템 콜의 원형. 리눅스 syscall 메커니즘의 기초
- ⓘ Coprocessor Assembly — ARM의 CP15 등 시스템 레지스터 접근. MMU/Cache 설정에 쓰임

**나중에 봐도 됨:**

- ⓐ ARM Assembly ADS vs GNU — 툴체인 차이 설명으로, 개념만 알면 됨
- ⓒ ARM, Thumb mode와 S 접미사 — 세부적인 어셈블리 문법
- ⓓ ARM과 Thumb 사이의 Veneer — interworking 세부 사항

## 5장 Software 비네팅

**꼭 봐야 함:**

- ⓒ struct와 typedef 그리고 PACKED — 리눅스 커널과 디바이스 드라이버 코드가 struct 덩어리. PACKED는 하드웨어 레지스터 매핑이나 통신 프로토콜 패킷 정의에 필수
- ⓓ STACK, HEAP에 관한 소고 — 커널 스택과 유저 스택의 차이를 나중에 이해하려면 기초가 탄탄해야 합니다
- ⓔ Stack의 정체와 자세히 보기 — 스택 오버플로우가 로봇 시스템에서 치명적 크래시를 일으키는 원인 중 하나
- ⓕ 함수가 불렸을 때 일어나는 일 - Stack 뒤지기 신공 — 크래시 덤프에서 콜스택을 추적하는 능력. 실무 디버깅의 핵심
- ⓖ Stack 동작의 비밀과 실제 메모리 덤프 — 위와 연결
- ⓗ Stack Size는 어떻게 잡는가 — RTOS나 커널 태스크의 스택 크기 결정은 실시간 시스템 설계의 중요한 문제
- ⓘ 함수 포인터와 실행주소 변경 — 리눅스 커널의 ops 구조체(file_operations, platform_driver 등)가 전부 함수 포인터 테이블. 디바이스 드라이버 작성의 기본 패턴
- ⓙ Linked List와 Queue — 커널의 list_head, 태스크 간 메시지 큐, 인터럽트 핸들러의 워크 큐 등 도처에서 사용

**보면 좋음:**

- ⓐ Context와 AAPCS — 함수 호출 규약과 컨텍스트 개념의 정리
- ⓑ Pointer와 Array — 이미 알고 있을 수 있지만, 이중 포인터까지 확실히 잡아야 커널 코드를 읽을 수 있음

## 6장 RTOS 팩토리 (전부 꼭 봐야 함)

이 장은 공고의 핵심 수행업무와 가장 직접적으로 대응하므로 **전체를 빠짐없이** 보는 것을 권하지만, 그 안에서도 특별히 중요도가 높은 것을 표시하면:

**최우선:**

- ⓓ Task 상태, Task는 Service단위 — 로봇의 각 기능(보행, 조작, 센싱)을 태스크로 나누는 설계의 원리
- ⓔ Preemptive Multitasking — 실시간 제어의 본질. 높은 우선순위의 모터 제어 태스크가 낮은 우선순위 태스크를 선점하는 구조
- ⓕ Context Switching과 TCB — 태스크 전환 비용이 실시간 성능의 핵심 지표
- ⓗ Scheduler의 구현 — 어떤 태스크를 언제 실행할지 결정하는 로직. 나중에 문c 블로그의 리눅스 스케줄러와 직접 연결
- ⓙ 선점형 Kernel에서 wait, send, clr signal의 구현과 IRQ Handler — 실시간 제어 루프에서 이벤트 기반 동작의 원리
- ⓚ Clock tick ISR - Timer Service — 제어 주기(1ms, 5ms 등)를 만드는 타이머 인터럽트의 메커니즘
- ⓛ ATOMIC - Critical Section Mutex Semaphore — 다중 태스크 환경에서 공유 자원 보호. 모터 제어 명령과 센서 데이터가 동시에 접근될 때 반드시 필요
- ⓜ Interrupt VS. Polling — 센서 읽기를 인터럽트로 할지 폴링으로 할지는 실시간 시스템 설계의 핵심 판단
- ⓝ Queue와 Inter Task Communication — AI 모듈과 제어 모듈 간 데이터 교환(공고의 수행업무 ④)의 원형

**그 다음:**

- ⓐ~ⓒ (RTOS 개론, 무한 Loop, Task 구조와 Signal) — 위의 내용을 이해하기 위한 도입부이므로 순서대로 읽으면 됨
- ⓖ TCB - Task Control Block — ⓕ와 세트
- ⓘ ISR은 어떻게 구현해 — 선점형/비선점형 인터럽트 처리의 차이
- ⓞ DPC나 APC, 그리고 Bottom Half — 리눅스 커널의 softirq/workqueue와 직결되는 개념
- ⓟ Watch dog task — 로봇이 제어 불능 상태에 빠졌을 때 안전하게 복구하는 메커니즘
- ⓠ Bootup 중 Kernel로의 진입 — OS 부팅 과정
- ⓡ Kernel을 Porting 한다는 것 — BSP 작업의 개념적 이해

## 7장 Device Control and System Architecture

**꼭 봐야 함:**

- ⓐ Device를 control 한다는 것 — 디바이스 드라이버 개발의 철학적 출발점
- ⓑ Register를 Setting 한다는 것과 Memory Mapped IO — 모든 디바이스 드라이버가 하는 일의 본질. ioremap → readl/writel의 원형
- ⓚ GPIO의 정체와 GPIO ISR — 센서 입력, 모터 Enable 신호 등 로봇에서 가장 기본적인 하드웨어 제어. 문c 블로그의 GPIO Subsystem으로 이어짐
- ⓛ DMA - CPU몰래 영차영차 — 모터 엔코더 데이터, IMU 센서 데이터 등을 대량으로 실시간 전송할 때 필수. 문c 블로그의 DMA 6개 항목의 선행 지식
- ⓜ Cache, Cash? — 실시간 시스템에서 캐시 일관성 문제는 버그의 주요 원인. DMA와 캐시의 관계를 모르면 데이터가 깨짐
- ⓝ MMU (Memory Management Unit) — 리눅스 기반 시스템의 가상 메모리 관리의 근간. 문c 블로그의 페이지 테이블 분석과 직결

**보면 좋음:**

- ⓒ Memory Device를 Control 한다는 것 — 메모리 디바이스 제어의 확장
- ⓓ Bus Sizer Register와 Memory Controller — 메모리 컨트롤러 설정. BSP 작업에서 만날 수 있음
- ⓕ Shadow의 개념과 Bit operation — 레지스터 비트 조작은 드라이버 코드에서 매일 하는 일
- ⓖ C의 조미료 MACRO Technics — 리눅스 커널이 매크로 덩어리이므로 매크로 읽는 능력이 중요
- ⓗ Synchronous와 Asynchronous — CAN(비동기), EtherCAT(동기) 등 통신 프로토콜의 근본적 차이를 이해
- ⓙ PLL과 M/N:D — 클록 설정. BSP에서 SoC의 PLL을 설정하는 작업과 관련
- ⓞ JTAG Interface를 Control 해 보자 — 로봇 보드 디버깅에 JTAG을 쓸 수 있음

**나중에 봐도 됨:**

- ⓔ 특이한 Device Interface의 CS와 Address - LCD — LCD 특화 내용이라 로봇과 직접적 관련 낮음
- ⓘ Wait State 이야기 — 개념만 알면 충분

## 요약 — 정말 시간이 없다면 이것만

전체 강좌 중 이 공고 기준 절대 빠지면 안 되는 "코어" 강좌를 뽑으면:

6장 전체 → 7장 ⓐⓑⓚⓛⓜⓝ → 5장 ⓒⓓⓔⓕⓗⓘⓙ → 3장 ⓘⓚⓛⓜⓝⓞⓟ → 2장 ⓓⓔⓕⓗⓘⓙ → 1장 ⓚⓛⓜⓝⓠⓡ → 4장 ⓖⓙⓚ

이 순서대로 보면 공고의 수행업무 4개 항목과 자격요건을 가장 효율적으로 커버할 수 있고, 문c 블로그로 넘어갈 때도 자연스럽게 이어집니다.
