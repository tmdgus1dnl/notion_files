# 하드웨어 설계

# Spot Micro 4족보행 로봇 프로젝트 기획서

## 1. 프로젝트 개요

| 항목 | 내용 |
| --- | --- |
| 프로젝트 | Spot Micro 기반 커스텀 4족보행 로봇 — 강화학습 보행 제어 |
| 인원 | 2명 |
| 기간 | 5~6주 (보행 제어 파트) |
| 최종 목표 | 자율주행 / 비전 AI 등과 통합 가능한 보행 모듈 완성 |

### 1.1 하드웨어 구성

- **프레임:** Spot Micro 오픈소스 기반 커스텀 설계
- **서보모터:** Feetech STS3215 × 12개 (3-DOF × 4다리, 시리얼 버스 서보)
- **상위 컴퓨팅:** NVIDIA Jetson Orin Nano (비전 AI, 자율주행, RL 추론)
- **하위 제어:** STM32F446RE (실시간 안전 로직, 서보 통신, 센서 처리)
- **전원:** LiPo 배터리
- **센서:** IMU (STM32 연결), 추후 카메라 등 확장

### 1.2 개발 환경

| 항목 | 스펙 |
| --- | --- |
| **학습 PC** |  |
| CPU | Intel Core i9 |
| GPU | NVIDIA RTX 4070 Laptop (8GB VRAM) |
| RAM | 64GB |
| OS | Ubuntu 22.04.5 (conda Python 3.8 가상환경) |
| 학습 플랫폼 | Isaac Gym Preview 4 + legged_gym |
| **로봇 탑재** |  |
| 보드 | Jetson Orin Nano |

> **학습과 추론이 별개 컴퓨터에서 수행된다.** 학습은 GPU가 있는 PC에서 Isaac Gym으로 수행하고, 학습된 정책(ONNX)만 Jetson Orin Nano로 옮겨 추론한다.
> 

> **Isaac Lab이 아닌 Isaac Gym을 사용하는 이유:** Isaac Lab(Isaac Sim 기반)은 최소 16GB VRAM을 권장하며, 8GB VRAM에서는 불안정하거나 크래시 가능성이 높다. Isaac Gym(standalone)은 최소 요구 사항이 8GB VRAM이며, Isaac Sim/Omniverse 없이 순수 물리 시뮬레이션 + RL에 집중하는 경량 프레임워크라 VRAM 효율이 훨씬 좋다. ETH Zurich의 legged_gym 레포가 Isaac Gym 위에서 4족보행 로봇 학습을 위해 만들어진 검증된 프레임워크이므로, Spot Micro URDF만 넣으면 바로 활용 가능하다.
> 

### 1.3 핵심 설계 결정 사항

**제어 방식: 고전제어 + RL 하이브리드**

순수 RL(토크 직접 출력)이 아닌 하이브리드를 선택한 이유:

1. **하드웨어 보호:** RL은 관절 목표 각도만 출력하고, 실제 모터 구동은 STS3215 내장 PID 제어기(고전제어)가 담당. STM32에서 관절 한계 클램핑과 slew rate limiting(고전제어)을 추가로 적용하여 RL이 비정상적 명령을 내려도 서보/기구부를 보호함.
2. **Jetson 부하 분산:** 자율주행/비전 AI가 Jetson GPU/CPU를 점유할 때, STM32가 안전 로직과 서보 통신을 독립 실행하여 보행 안정성 유지.
3. **팀 간 분업 용이:** 보행 모듈이 독립적으로 동작하므로 자율주행 팀과 인터페이스만 맞추면 병렬 개발 가능.

> **하이브리드 구조 상세:** RL이 "전략"(어떤 자세를 취할지)을 결정하고, 고전제어가 "실행"(안전하게 모터를 구동)을 담당하는 역할 분담이다. 구체적으로 고전제어는 세 겹으로 구성된다: (1) STM32의 관절 한계 클램핑, (2) STM32의 slew rate limiting(급격한 관절 변화 방지), (3) STS3215 내장 PID 위치 제어기.
> 

**RL 출력: 관절 목표 각도 (Joint Position Target)**

legged_gym의 표준 방식으로, 관절 각도 오프셋을 출력하는 구조:

1. **오픈소스 최대 활용:** legged_gym, walk-these-ways, IsaacGymEnvs 등 검증된 4족보행 오픈소스를 설정 파일 수정만으로 바로 사용 가능. 환경 코드 커스터마이징이 최소화됨.
2. **검증된 Sim-to-Real 사례:** Go1, ANYmal, Humanoid 등 다양한 로봇에서 관절 타겟 + PD 제어 + Domain Randomization 조합으로 성공적인 sim-to-real 전이가 검증됨.
3. **디버깅 용이:** IK가 없으므로 "RL이 잘못인지 IK가 잘못인지" 구분할 필요 없이 RL 정책만 집중 디버깅 가능.
4. **시스템 단순화:** STM32에서 IK를 구현/유지할 필요가 없어 펌웨어 복잡도가 낮아지고, 시뮬레이션과 실제 로봇 간 불일치 요소가 줄어듦.

## 2. 시스템 아키텍처

### 2.1 전체 구조도

```
┌──────────────────── Jetson Orin Nano (Linux/ROS2 Humble) ───┐
│                                                              │
│  [Application Layer]                                         │
│     자율주행 / 비전 AI / 조이스틱 / 원격 제어               │
│       │                                                      │
│       ▼  cmd_vel (vx, vy, ω)                                │
│                                                              │
│  [Middleware Layer]                                           │
│     RL 정책 추론 (ONNX → TensorRT)                          │
│       - observation: cmd_vel + IMU + 관절 피드백             │
│       - output: 관절 목표 각도 × 12 = 12 float             │
│       │                                                      │
│       ▼                                                      │
│     UART 패킷 송신 (12 float, 50Hz)                         │
└─────────────────────────┬────────────────────────────────────┘
                          │  UART
                          ▼
┌──────────────────── STM32F446RE ─────────────────────────────┐
│                                                               │
│  [HAL Layer]                                                  │
│     패킷 수신 + 파싱 (DMA)                                   │
│     관절 한계 클램핑                                         │
│     slew rate limiting (급격한 변화 보간)                     │
│     watchdog (타임아웃 → 안전 자세 폴백)                     │
│       │                                                       │
│       ▼                                                       │
│  [Device Driver Layer]                                        │
│     STS3215 sync write (DMA, UART half-duplex, 12개 서보)    │
│     STS3215 sync read → 관절 피드백 → Jetson으로 전송 (DMA) │
└───────────────────────────────────────────────────────────────┘
                          │  STS3215 Serial Bus (UART)
                          ▼
                   ┌──────────────┐
                   │ STS3215 × 12 │
                   │  (내장 PID)  │
                   │  (3-DOF × 4) │
                   └──────────────┘
```

### 2.2 제어 흐름 상세

```
RL 정책 (Jetson)
  │  12개 관절 목표 각도 오프셋
  ▼
default_joint_angles + 오프셋 = 관절 목표 각도
  │
  ▼  UART 패킷
STM32 안전 레이어 (고전제어 1, 2)
  ├─ 관절 한계 클램핑 (각 관절의 물리적 회전 범위 제한)
  └─ slew rate limiting (이전 명령 대비 급격한 변화 보간)
  │
  ▼  STS3215 프로토콜
STS3215 내장 PID (고전제어 3)
  └─ 목표 위치 추종, 토크 자동 계산
  │
  ▼
모터 구동
```

### 2.3 계층별 역할 상세

### Application Layer (Jetson)

- 자율주행 스택(SLAM, Nav2 등)이 경로를 계산하여 `cmd_vel` 발행
- 표준 인터페이스: ROS2 `geometry_msgs/Twist`
    - `linear.x` = 전진/후진 속도 (m/s)
    - `linear.y` = 좌우 횡이동 속도 (m/s)
    - `angular.z` = 회전 속도 (rad/s, yaw rate)
- 조이스틱, 비전 AI 등도 동일한 Twist 메시지로 로봇 제어
- **보행 구현을 전혀 몰라도 됨** — 바퀴 로봇과 동일한 인터페이스

### Middleware Layer (Jetson)

- RL 정책 네트워크 추론 (학습은 Isaac Gym + legged_gym, 추론은 ONNX/TensorRT)
- **Observation space:** cmd_vel(3) + IMU 데이터(6) + 관절 각도(12) + 관절 각속도(12) + 이전 액션(12)
- **Action space:** 관절 목표 각도 오프셋 × 12 (default_joint_angles 기준)
- 50Hz 주기로 STM32에 UART 패킷 전송

### HAL Layer (STM32)

- Jetson으로부터 관절 목표 각도 12 float 수신 및 파싱 (DMA 기반)
- 관절 한계 클램핑: 각 관절의 물리적 회전 범위 제한
- Slew rate limiting: 이전 명령 대비 너무 큰 변화 시 보간 처리 (서보 보호 + 부드러운 동작)
- Watchdog: Jetson 통신 두절 시 안전 자세(crouch)로 자동 전환
- IMU 읽기 및 필터링 (추후 확장)

### Device Driver Layer (STM32)

- STS3215 시리얼 버스 프로토콜 처리 (DMA 기반, 반이중 UART 방향 제어)
- `servo_sync_write_dma()`: 관절 각도 → 서보 position 값(0~4095) 변환 → sync write
- `servo_sync_read_dma()`: 12개 서보의 위치/속도/부하 피드백 일괄 읽기
- 피드백 데이터를 Jetson으로 전송 (RL observation용, DMA)

### 2.4 자율주행 통합 인터페이스

자율주행 팀은 보행 내부 구현을 알 필요 없이 `cmd_vel` 토픽에 Twist 메시지만 publish하면 로봇이 해당 속도로 이동한다. Gazebo 시뮬레이션에서도 동일한 인터페이스를 사용하므로 코드 수정 없이 테스트 가능.

```
자율주행 팀 → cmd_vel (Twist) → 보행 모듈 → 로봇 이동
조이스틱     → cmd_vel (Twist) → 보행 모듈 → 로봇 이동
시뮬레이터   → cmd_vel (Twist) → Gazebo    → 가상 이동
```

## 3. 강화학습 설계

### 3.1 학습 환경

- **플랫폼:** Isaac Gym Preview 4 (standalone, Isaac Sim 불필요)
- **프레임워크:** legged_gym (ETH Zurich RSL, [github.com/leggedrobotics/legged_gym](http://github.com/leggedrobotics/legged_gym))
- **RL 라이브러리:** rsl_rl (legged_gym과 함께 사용하는 PPO 구현체)
- **알고리즘:** PPO (Proximal Policy Optimization)
- **병렬 환경:** 1024~2048개 (RTX 4070 Laptop 8GB VRAM 기준)
- **OS:** Ubuntu 22.04.5 (conda Python 3.8 가상환경)
- **Python:** 3.8 (conda 환경 권장)

> **VRAM 관련 참고:** 4096개 환경은 8GB VRAM에서 OOM 가능성이 있다. 1024개로 시작하여 VRAM 여유를 확인한 후 2048까지 올려볼 것. 병렬 환경 수가 줄면 학습 시간이 비례적으로 늘어나지만 (2~3배), 결과물의 품질은 동일하다.
> 

> **Ubuntu 22.04 호환성:** Isaac Gym Preview 4는 공식적으로 18.04/20.04 지원이지만, conda로 Python 3.8을 잡으면 22.04에서도 정상 동작한다. `LD_LIBRARY_PATH=$CONDA_PREFIX/lib/:$LD_LIBRARY_PATH` 설정이 필요할 수 있음.
> 

### 3.2 legged_gym 환경 구성

legged_gym은 `LeggedRobot` 베이스 클래스를 상속하여 커스텀 환경을 만드는 구조다. Spot Micro용 환경은 **AnymalTerrain 설정을 복사하여 수정하는 것**이 가장 빠른 방법이다. 수정할 항목:

- **설정 파일** (`spot_micro_config.py`): 두 클래스 포함
    - `SpotMicroCfg(LeggedRobotCfg)`: URDF 경로, 관절 이름, default_joint_angles, PD 게인, reward 스케일
    - `SpotMicroCfgPPO(LeggedRobotCfgPPO)`: 학습 하이퍼파라미터 (대부분 기본값 유지)
- **환경 파일** (`spot_micro.py`): 기본적으로 `LeggedRobot`을 그대로 사용. 커스텀 reward 함수 추가 시에만 오버라이드

환경 등록은 `envs/__init__.py`에서 `task_registry.register("spot_micro", LeggedRobot, SpotMicroCfg, SpotMicroCfgPPO)`로 수행.

학습 실행: `python legged_gym/scripts/train.py --task=spot_micro`
정책 확인: `python legged_gym/scripts/play.py --task=spot_micro`

### 3.3 URDF 제작 (최우선 과제)

URDF 품질이 sim-to-real 성공률의 70% 이상을 결정한다. 반드시 정확히 만들어야 할 항목:

- 각 링크의 질량 및 관성 텐서 (CAD에서 추출, Jetson + LiPo + 서보 무게 포함)
- 관절 한계 (STS3215 실제 구동 범위 반영)
- 관절 마찰 계수, 감쇠 설정
- STS3215 토크 한계 (약 15 kg·cm @6V) 및 속도 한계
- Collision mesh (단순화하되 self-collision 감지 가능하게)

> **참고:** Isaac Gym은 URDF를 직접 로드한다 (USD 변환 불필요). legged_gym 설정 파일의 `asset.file` 경로에 URDF 파일을 지정하면 된다.
> 

### 3.4 Observation Space

| 항목 | 차원 | 설명 |
| --- | --- | --- |
| 목표 속도 명령 | 3 | vx, vy, ω (cmd_vel) |
| 몸체 자세 (IMU) | 6 | roll, pitch, yaw + 각속도 3축 |
| 관절 각도 | 12 | 4다리 × 3-DOF |
| 관절 각속도 | 12 | 4다리 × 3-DOF |
| 이전 액션 | 12 | 이전 스텝의 관절 목표 각도 오프셋 |
| **합계** | **45** |  |

### 3.5 Action Space

- 관절 목표 각도 오프셋 × 12 = **12차원** (연속 공간)
- `default_joint_angles`에 오프셋을 더한 값이 최종 관절 목표
- legged_gym 내부 PD 제어기가 목표 각도와 현재 각도의 차이로 토크를 계산하여 시뮬레이션에 적용
- **이것이 legged_gym의 표준 방식**이며, AnymalTerrain, Go1 등 모든 4족보행 예제가 이 구조를 사용

```python
# legged_gym 내부 토크 계산 (고전제어 — PD 제어기)
torques = Kp * (action_scaled + default_angles - current_angles) - Kd * current_velocities
```

### 3.6 Reward Function 설계

legged_gym에서는 설정 파일의 `rewards.scales`에 각 항목의 가중치를 설정하면, 이름이 매칭되는 reward 함수가 자동으로 합산된다. 가중치를 0으로 설정하면 해당 항목이 비활성화된다.

| 항목 | 유형 | 설명 |
| --- | --- | --- |
| 속도 추종 (tracking_lin_vel) | 보상 | cmd_vel과 실제 속도 차이 최소화 (가장 높은 가중치) |
| 각속도 추종 (tracking_ang_vel) | 보상 | yaw rate 명령 추종 |
| 몸체 안정성 (orientation) | 페널티 | roll, pitch 흔들림 최소화 |
| 에너지 효율 (torques) | 페널티 | 토크 제곱합 최소화 |
| 관절 한계 접근 (dof_pos_limits) | 페널티 | 관절이 한계에 가까워지면 패널티 |
| 발 접촉 패턴 (feet_air_time) | 보상 | 적절한 swing 시간 유도 → trot gait 유도 |
| 동작 부드러움 (action_rate) | 페널티 | 연속 액션 간 급격한 변화 최소화 |
| 넘어짐 (termination) | 페널티 | 몸체가 지면에 닿으면 큰 페널티 + 에피소드 종료 |

> Reward 가중치 튜닝이 가장 시간이 많이 소요되는 작업. 초기에는 속도 추종 + 넘어짐 페널티만으로 시작한 후 점진적으로 항목 추가 권장. legged_gym의 AnymalTerrain 예제 가중치를 참고하면 좋은 출발점이 된다.
> 

### 3.7 Domain Randomization

Sim-to-real 전이를 위해 필수적으로 적용할 랜덤화 항목. legged_gym은 `domain_rand` 설정 섹션에서 대부분의 랜덤화를 지원한다.

| 항목 | 범위 | legged_gym 설정 |
| --- | --- | --- |
| 링크 질량 | ±15~20% | `domain_rand.added_mass_range` |
| 지면 마찰 계수 | 0.4~1.2 | `domain_rand.friction_range` |
| 외부 교란 (push) | 랜덤 방향/크기 | `domain_rand.push_robots`, `max_push_vel_xy` |
| 지형 높낮이 | 불규칙 지형 | `terrain.terrain_proportions` (legged_gym 내장) |
| 관측 노이즈 | 가우시안 | `noise.add_noise`, `noise_scales` |
| 서보 응답 지연 | 0~20ms | 커스텀 구현 필요 (액션 버퍼 딜레이) |

### 3.8 커리큘럼 학습

legged_gym은 지형 기반 커리큘럼을 내장하고 있다. 로봇이 현재 난이도를 잘 통과하면 더 어려운 지형으로 자동 이동한다.

1. **Stage 1:** 평지, 교란 없음 → 기본 보행 학습
2. **Stage 2:** 평지 + 약한 외부 push → 균형 잡기
3. **Stage 3:** 불규칙 지형 + 강한 push → 강건한 보행

### 3.9 학습 시간 예상 (RTX 4070 Laptop, 1024 envs 기준)

| 단계 | 예상 시간 | 비고 |
| --- | --- | --- |
| 평지 기본 보행 수렴 | 1~3시간 | 간단한 reward로 시작 시 |
| 전체 reward + DR 수렴 | 4~8시간 | reward 항목과 DR 범위에 따라 변동 |
| 지형 커리큘럼 포함 | 8~15시간 | 거친 지형에서의 안정적 보행까지 |

> 4096 envs 대비 약 2~3배 느리지만, 밤새 학습을 돌리면 충분히 커버 가능한 수준이다. reward 튜닝 시 빠른 이터레이션을 위해 처음에는 평지 + 간단한 reward로 30분~1시간 단위 실험을 반복하는 것이 효율적이다.

## 4. STS3215 관련 주의사항

### 4.1 스펙 요약

- 통신: 반이중 UART (TTL 시리얼 버스), 데이지 체인 가능
- 내장 PID 위치 제어기 (목표 위치만 전송하면 서보가 추종) — 하이브리드 제어의 고전제어 구성 요소
- 위치/속도/부하 피드백 제공
- 스톨 토크: 약 15 kg·cm (6V 기준)
- 라이브러리: Feetech SCServo SDK

### 4.2 토크 마진 검증 (필수)

Jetson + LiPo + 프레임 + 서보 자체 무게를 합산하면 총중량 2kg 초과 가능. STS3215 토크(15 kg·cm @6V)는 넉넉하지 않을 수 있으므로, 프로젝트 초기에 반드시 다음을 확인할 것:

1. 총중량 실측
2. 각 다리의 최대 부하 토크 계산 (다리 길이 × 하중 분배)
3. 안전 마진: 최대 부하 토크가 스톨 토크의 **67% 이하** (1.5배 마진)인지 확인
4. 마진 부족 시 다리 길이 단축 또는 경량화 검토

## 5. 통신 프로토콜 설계

### 5.1 Jetson → STM32 (다운링크)

| 필드 | 크기 | 설명 |
| --- | --- | --- |
| Header | 2 byte | 0xAA 0x55 (동기화) |
| Packet Type | 1 byte | 0x01 = 관절 목표 각도 명령 |
| Joint Targets | 48 byte | float32 × 12 (관절 목표 각도, rad) |
| Checksum | 1 byte | XOR 또는 CRC8 |
| **합계** | **52 byte** |  |

50Hz × 52byte = 2,600 byte/s → 115200 baud에서 여유 충분.

### 5.2 STM32 → Jetson (업링크)

| 필드 | 크기 | 설명 |
| --- | --- | --- |
| Header | 2 byte | 0xAA 0x55 |
| Packet Type | 1 byte | 0x02 = 피드백 |
| Joint Angles | 48 byte | float32 × 12 (현재 관절 각도) |
| Joint Velocities | 48 byte | float32 × 12 (현재 관절 각속도) |
| IMU Data | 24 byte | float32 × 6 (roll, pitch, yaw, gx, gy, gz) |
| Status | 1 byte | 비트 플래그 (watchdog, 에러 등) |
| Checksum | 1 byte |  |
| **합계** | **125 byte** |  |

### 5.3 DMA 기반 논블로킹 통신

모든 UART 통신은 DMA(Direct Memory Access)로 처리하여 CPU 블로킹을 방지한다. STS3215 sync write/read가 4~8ms 소요되더라도 CPU는 그 시간 동안 watchdog 체크, 패킷 파싱 등 다른 작업을 자유롭게 수행할 수 있다. STM32F446RE는 DMA 컨트롤러 2개, 채널 16개를 보유하고 있어 여러 UART를 동시에 DMA로 처리 가능하다.

| 리스크 | 영향도 | 대응 방안 |
| --- | --- | --- |
| STS3215 토크 부족 | 높음 | Week 1에서 조기 검증. 부족 시 다리 길이 단축 또는 경량화 |
| URDF 부정확 → sim-to-real 실패 | 높음 | CAD 기반 정밀 측정, 실측 무게 반영, DR로 보상 |

| 8GB VRAM 부족 (OOM) | 중간 | 병렬 환경 수 축소 (1024 → 512), headless 모드 사용, terrain mesh 단순화 |
| Isaac Gym이 legacy → 지원 중단 | 낮음 | 현재 Preview 4가 안정적으로 동작하며, legged_gym 생태계가 충분히 성숙함. 추후 필요 시 Isaac Lab으로 마이그레이션 가능 |
