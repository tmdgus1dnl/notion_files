## 0. 범위 정의

SIM 파트는 시뮬레이션 환경 구성부터 가상 로봇의 자율 탐색, 인명 탐지, 상태 데이터 생성까지를 담당한다.
실제 보행(관절 저수준 제어, RL 보행 정책)은 보행 파트가 담당하며, 관제 UI 표기는 GUI 파트가 담당한다.

| 블록 | 담당 | 비고 |
| --- | --- | --- |
| [A] Environment Setup / Pre-Mission | SIM | 맵, 좌표계, 경로 생성 |
| [B] Fleet Mission Manager | SIM (데이터 생성) / GUI (표기) | 상태 데이터 제공 |
| [C] Robot Mission Executor | SIM | 로봇 상위 FSM |
| [D] Planning / Local Decision | SIM | 경로 추종 및 장애물 회피 |
| [E] Perception / Environment Modeling | SIM | 센서 데이터 → 환경 모델 |
| [F] State Estimation / Localization | SIM | 위치 추정 |
| [G] Safety Supervisor | SIM | Fail-Safe |
| [H] Low-Level Control | SIM (목표 벡터까지) / 보행 파트 (보행 실행) |  |
| [I] Sim Adapter | SIM | Isaac Sim 가상 로봇 인터페이스 |
| 디지털트윈 동기화 | SIM | 실물-가상 동기화 |
| Isaac Sim 월드 구성 | SIM | 환경, 장애물, 구역, 실종자 후보 |

## 1. Isaac Sim 디지털트윈 월드 구성

### 1-1. 개요

Isaac Sim 위에 시연 환경을 재현한 디지털트윈 월드를 구성한다.
실물 로봇 1대와 가상 로봇 2대가 동일한 공간에서 병렬 탐색을 수행한다.

### 1-2. 구성 요소

| 요소 | 내용 |
| --- | --- |
| 환경 | 평지 + 벽 + 장애물 (시연장 기반) |
| 가상 로봇 | sim_01, sim_02 (Spot Micro URDF) |
| 구역 | Zone A / B / C (탐색 구역 분할) |
| 실종자 후보 객체 | 정적 객체로 배치 |
| 장애물 | 고정 장애물 + 선택적 동적 장애물 |

### 1-3. 내부 동작 흐름

1. Isaac Sim 실행 및 월드 로드
2. 시연장 맵 기반 환경 구성 (벽, 바닥, 장애물 배치)
3. 가상 로봇 2대 URDF 로드 및 초기 위치 설정
4. Zone A / B / C 구역 정의
5. 실종자 후보 객체 배치
6. ROS2 브릿지 활성화 → 센서/제어 토픽 연결

## 2. [A] Environment Setup / Pre-Mission

### 2-1. 개요

임무 시작 전 탐색 공간을 지도화하고, 공통좌표계와 global path를 생성한다.
사전 실측 또는 디지털트윈 정보를 기반으로 처리한다.

### 2-2. Input / Output

| 구분 | 데이터 | 설명 |
| --- | --- | --- |
| Input | 실측 데이터 또는 디지털트윈 정보 | 시연장 지형 정보 |
| Output | `mission_map` | 공통좌표계 기반 정밀 지도 |
| Output | `zone_assignment` | 구역 분할 결과 (Zone A/B/C) |
| Output | `global_path_set` | 탐색 출발점~종료점 경로 후보 집합 |

### 2-3. 내부 동작 흐름

1. 시연장 실측 데이터 또는 Isaac Sim 월드 정보 수신
2. 공통좌표계(`mission_map`) 정의 (GPS 대용, 출발점 기준)
3. 탐색 공간을 Zone A / B / C로 분할
4. 각 구역에 대한 global path 후보 생성
5. `mission_map`, `zone_assignment`, `global_path_set` 출력

## 3. [B] Fleet Mission Manager (데이터 생성 담당)

### 3-1. 개요

로봇별 임무 배정, 상태 수집, 임무 재할당을 담당한다.
데이터 생성 및 제공은 SIM 파트가, UI 표기는 GUI 파트가 담당한다.

### 3-2. Input / Output

| 구분 | 데이터 | 설명 |
| --- | --- | --- |
| Input | `mission_map` | 공통좌표계 기반 지도 |
| Input | 로봇 상태 (`robot_status`) | 각 로봇의 현재 상태 |
| Input | 실종자 발견 이벤트 (`victim_event`) | 탐지 결과 수신 |
| Output | `robot_mission` | 각 로봇에 할당된 global path |
| Output | `fleet_state` | 전체 로봇 상태 집합 |
| Output | `replan_command` | 임무 재계획 명령 (선택) |

### 3-3. 내부 동작 흐름

1. `mission_map`과 `global_path_set` 수신
2. 실물 1대 + 가상 2대에 각각 다른 `robot_mission` 배정
3. 각 로봇으로부터 주기적으로 `robot_status` 수집
4. `fleet_state` 생성 및 GUI 파트로 전달
5. `victim_event` 수신 시 임무 종료 또는 재할당 처리
6. 필요 시 `replan_command` 생성

## 4. [C] Robot Mission Executor

### 4-1. 개요

로봇 상위 FSM으로, 할당된 임무를 수행하고 상태 전이를 관리한다.

### 4-2. Input / Output

| 구분 | 데이터 | 설명 |
| --- | --- | --- |
| Input | `robot_mission` | 할당된 global path |
| Input | `robot_pose` | 현재 로봇 위치 |
| Input | `victim_event` | 실종자 탐지 이벤트 |
| Input | `planner_status` | Planning 계층 실행 상태 |
| Output | `mission_context` | 임무 수행에 필요한 내부 데이터 |
| Output | `mission_status` | 외부(관제)에 전달할 임무 진행 상태 |

### 4-3. mission_status 상태 목록

```
IDLE
ASSIGNED
EXECUTING
AVOIDING
PAUSED
VICTIM_FOUND
COMPLETED
FAILED
```

### 4-4. planner_status 상태 목록

```
FOLLOWING_GLOBAL_PATH
GENERATING_LOCAL_PATH
AVOIDING_OBSTACLE
STOPPED_BY_SAFETY
VICTIM_FOUND
MISSION_COMPLETE
REPLANNING
```

### 4-5. 내부 동작 흐름

1. `robot_mission` 수신 → IDLE → ASSIGNED 전이
2. 임무 실행 시작 → EXECUTING 전이
3. `planner_status` 모니터링하며 상태 전이 관리
4. 장애물 회피 중 → AVOIDING 전이
5. `victim_event` 수신 → VICTIM_FOUND 전이 → 상위 Fleet Manager에 이벤트 전송
6. 임무 완료 → COMPLETED 전이

## 5. [D] Planning / Local Decision

### 5-1. 개요

Global path 추종 및 local obstacle avoidance를 담당하고, 목표 이동 벡터를 생성한다.

### 5-2. Input / Output

| 구분 | 데이터 | 설명 |
| --- | --- | --- |
| Input | `mission_context` | 현재 임무 내부 데이터 |
| Input | `robot_pose` | 현재 로봇 위치 |
| Input | `obstacle_model` | 장애물 정보 |
| Input | `free_space_model` | 주행 가능 공간 정보 (선택) |
| Output | `local_path` | 생성된 로컬 경로 |
| Output | `desired_heading` | 목표 이동 방향 |
| Output | `desired_velocity` | 목표 이동 속도 |

### 5-3. free_space_model 필드

```
front_clearance
left_clearance
right_clearance
best_escape_direction
navigable_corridor_detected
corridor_width
free_space_score_left
free_space_score_right
```

### 5-4. 내부 동작 흐름

1. `mission_context`에서 목표 waypoint 추출
2. `robot_pose` 기반으로 global path 추종
3. `obstacle_model` 수신 시 local path 재생성
4. `free_space_model` 참조하여 회피 방향 결정
5. `desired_heading`, `desired_velocity` 생성 → Safety Supervisor로 전달

## 6. [E] Perception / Environment Modeling

### 6-1. 개요

카메라와 LiDAR raw 데이터를 수신하여 환경 모델로 변환한다.

### 6-2. Input / Output

| 구분 | 데이터 | 설명 |
| --- | --- | --- |
| Input | `camera_raw` | RGB 카메라 raw 영상 |
| Input | `lidar_raw` | LiDAR raw 포인트클라우드 |
| Output | `obstacle_model` | 장애물 위치 및 크기 정보 |
| Output | `free_space_model` | 주행 가능 공간 정보 |
| Output | `victim_detection` | 인명 탐지 결과 |
| Output | `localization_hint` | LiDAR 기반 위치 추정 보조 정보 |

### 6-3. localization_hint 필드

```
left_wall_distance       # 왼쪽 벽까지 거리
right_wall_distance      # 오른쪽 벽까지 거리
front_wall_distance      # 전방 벽까지 거리 또는 존재 여부
center_offset_estimate   # 경로 중심에서 치우친 정도
wall_alignment_score     # 벽 정렬 정도
corner_detected
map_match_hint
```

### 6-4. 내부 동작 흐름

1. `camera_raw` 수신 → YOLOv11 + TensorRT 기반 person 탐지
2. 탐지 결과 → `victim_detection` 생성
3. `lidar_raw` 수신 → 장애물 추출 → `obstacle_model` 생성
4. `lidar_raw` 후처리 → 벽/공간 분석 → `free_space_model` 생성
5. `lidar_raw` 후처리 → `localization_hint` 생성

## 7. [F] State Estimation / Localization

### 7-1. 개요

IMU, odom, LiDAR hint를 융합하여 현재 위치를 추정하고 공통좌표계 기준 pose를 생성한다.

### 7-2. Input / Output

| 구분 | 데이터 | 설명 |
| --- | --- | --- |
| Input | `imu` | IMU 센서 데이터 |
| Input | `odom` | 오도메트리 데이터 |
| Input | `localization_hint` | LiDAR 기반 위치 보조 정보 |
| Input | `mission_map_alignment` | 공통좌표계 정렬 정보 |
| Output | `robot_pose` | 공통좌표계 기준 현재 위치 |
| Output | `robot_heading` | 현재 방향 |
| Output | `pose_confidence` | 위치 추정 신뢰도 |

### 7-3. 내부 동작 흐름

1. `imu`, `odom` 수신 → 기본 pose 추정
2. `localization_hint` 수신 → pose 보정
3. `mission_map_alignment` 적용 → 공통좌표계 기준 pose 변환
4. `robot_pose`, `robot_heading`, `pose_confidence` 생성
5. Fleet Mission Manager 및 관제 시스템으로 주기적 전송

### 7-4. 위치 추정 방식

- LiDAR Odometry 기반 (SLAM 아님)
- 맵 생성·저장 오버헤드 없음
- 출발점 기준 상대 좌표 누적
- 한계 : 루프 클로저 없음, 장시간 운용 시 오차 누적

## 8. [G] Safety Supervisor

### 8-1. 개요

충돌 위험 및 시스템 이상을 실시간 감시하고, 필요 시 Planning 명령을 override한다.

### 8-2. Input / Output

| 구분 | 데이터 | 설명 |
| --- | --- | --- |
| Input | `desired_motion` | Planning이 생성한 목표 제어 명령 |
| Input | `collision_risk` | 충돌 위험 수준 |
| Input | `robot_health` | 로봇 시스템 상태 |
| Output | `safe_motion_cmd` | Safety 승인된 최종 제어 명령 |
| Output | `emergency_stop` | 긴급 정지 명령 |
| Output | `fault_state` | 시스템 고장/이상 상태 |

### 8-3. 내부 동작 흐름

1. `desired_motion` 수신
2. `collision_risk` 확인 → 위험 수준 초과 시 `emergency_stop` 발행
3. `robot_health` 모니터링 → 통신 끊김 / 자세 이상 감지 시 개입
4. 정상 범위 내 → `desired_motion`을 `safe_motion_cmd`로 통과
5. 이상 감지 → `fault_state` 생성 및 상위 Mission Executor로 전달

## 9. [H] Low-Level Control 인터페이스 (SIM 파트 담당 범위)

### 9-1. 개요

SIM 파트는 목표 이동 벡터(`safe_motion_cmd`) 생성까지를 담당한다.
실제 보행 실행(관절 저수준 제어, RL 보행 정책)은 보행 파트가 담당한다.

### 9-2. Input / Output (SIM 파트 범위)

| 구분 | 데이터 | 설명 |
| --- | --- | --- |
| Input | `safe_motion_cmd` | Safety 승인된 목표 이동 벡터 |
| Output | `joint_state` | 관절 상태 (보행 파트 → SIM 수신) |
| Output | `imu` | IMU 데이터 (보행 파트 → SIM 수신) |
| Output | `foot_contact` | 발 접촉 상태 (보행 파트 → SIM 수신) |
| Output | `control_status` | 제어 FSM 상태 |

### 9-3. 보행 파트와의 인터페이스 경계

```
SIM 파트                     보행 파트
────────────────────────────────────────
safe_motion_cmd ──────────►  RL Locomotion / 관절 제어
                ◄──────────  joint_state / imu / foot_contact
```

## 10. [I] Sim Adapter

### 10-1. 개요

Isaac Sim 가상 로봇을 공통 인터페이스로 연결한다.
실물 로봇의 Real Adapter와 동일한 공통 인터페이스를 사용하여 상위 계층의 코드 변경 없이 실물/가상 전환이 가능하다.

### 10-2. Input / Output

| 구분 | 데이터 | 설명 |
| --- | --- | --- |
| Input | `cmd_motion` | 상위 계층에서 전달된 이동 명령 |
| Output | `robot_state` | 가상 로봇 상태 |
| Output | `sensor_stream` | 가상 센서 데이터 스트림 |
| Output | `robot_pose_feedback` | 가상 로봇 위치 피드백 |

### 10-3. 공통 인터페이스 목록

```
cmd_motion
robot_state
robot_pose
victim_event
```

### 10-4. 내부 동작 흐름

1. `cmd_motion` 수신
2. Isaac Sim API를 통해 가상 로봇에 명령 전달
3. Isaac Sim에서 `robot_state`, `sensor_stream` 수신
4. 공통 인터페이스 형식으로 변환하여 상위 계층으로 전달
5. `robot_pose_feedback` 주기적 발행

## 11. 디지털트윈 동기화 계층

### 11-1. 개요

실물 로봇과 Isaac Sim 가상 환경 간의 상태를 동기화한다.

### 11-2. 구성 모듈

| 모듈 | 역할 |
| --- | --- |
| `real_robot_twin_sync` | 실물 로봇 상태를 Isaac Sim 엔티티에 동기화 |
| `world_model_aggregator` | 실물/가상 로봇의 환경 인식 결과를 통합 |
| `time_sync_manager` | 실물과 시뮬 간 타임스탬프 동기화 |

### 11-3. Input / Output

| 구분 | 데이터 | 설명 |
| --- | --- | --- |
| Input | `RobotStatus` | 각 로봇의 상태 |
| Input | `Odom` | 각 로봇의 오도메트리 |
| Input | `Event` | 탐지 이벤트 등 |
| Output | Isaac Sim 엔티티 상태 업데이트 | 실물 로봇 트윈 반영 |
| Output | 통합 환경 모델 | 전체 월드 상태 |

### 11-4. 내부 동작 흐름

1. 실물 로봇으로부터 `RobotStatus`, `Odom` 수신
