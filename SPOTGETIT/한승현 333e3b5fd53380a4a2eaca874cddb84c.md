# ROS2 범용 MCU 펌웨어 플랫폼

**STM32 + FreeRTOS + micro-ROS + ros2_control**
6인 팀 | 9주 | 교보재 100만원

---

## 1. 프로젝트 개요

### 핵심 문제 정의

> ROS2 로봇을 만들 때 하드웨어(모터, 엔코더)가 바뀌면 드라이버를 매번 새로 짜야 한다.
표준 인터페이스(ros2_control)가 있지만, MCU 레벨 구현체 오픈소스가 거의 없다.
→ **어떤 로봇에도 붙을 수 있는 범용 MCU 펌웨어를 오픈소스로 만든다.**
> 

---

### 전체 시스템 구조

```
[젯슨 Orin Nano — 상위]
  ROS2 + Nav2 + SLAM + LiDAR
  micro-ROS Agent (브릿지)
  ros2_control 인터페이스
          ↕ UART / USB
[STM32 — 하위 MCU]
  FreeRTOS 태스크 스케줄링
  micro-ROS 클라이언트
  PID 제어 + PWM 출력
  엔코더 드라이버 + odom 계산
  안전 정지 로직
          ↕
[모터 드라이버 + 바퀴]
```

---

### FreeRTOS 태스크 구조

| 태스크 | 주기 | 역할 | 이유 |
| --- | --- | --- | --- |
| Task1 — PID + PWM | **1ms** | 엔코더 읽기 + PID 계산 + PWM 출력 | 모터 제어는 타이밍이 제일 중요 → 최고 우선순위 |
| Task2 — odom | **10ms** | 바퀴 이동 거리 계산 → odom 메시지 생성 | PID보다 덜 급하지만 정확해야 함 |
| Task3 — 통신 | **100ms** | cmd_vel 수신 + odom 발행 (micro-ROS) | 네트워크 통신은 느려도 됨 |

## 4. 이 프로젝트의 차별점

### AUTOBOX vs 이번 프로젝트

|  | AUTOBOX (이전) | 이번 프로젝트 |
| --- | --- | --- |
| 하드웨어 레이어 | 블랙박스 — 누군가 만든 드라이버 사용 | 직접 설계 및 구현 |
| 제어 구조 이해 | cmd_vel 내리면 왜 움직이는지 몰라도 됨 | PID / 엔코더 / odom 내부 구조 완전 이해 |
| 재사용성 | 특정 로봇에 종속 | ros2_control 표준 → 어떤 로봇에도 재사용 가능 |
| 오픈소스 | 없음 | GitHub 배포 + 문서화 |

### 성능 비교 목표 (수치로 증명)

> **지터(Jitter)**: 제어 루프 실행 주기의 편차
예: 1ms 주기 태스크가 1.3ms / 0.7ms로 흔들리면 모터가 떨림
→ FreeRTOS 적용 전후 지터 수치를 측정해서 포트폴리오에 넣을 예정

# FreeRTOS 기반 CAN 분산 실시간 제어 시스템

---

### 프로젝트 개요

| 항목 | 내용 |
| --- | --- |
| 프로젝트명 | FreeRTOS 기반 CAN 분산 실시간 제어 시스템 |
| 역할 | 시스템 설계, 펌웨어 구현, 통신 프로토콜 설계 전담 |
| 플랫폼 | STM32 (Cortex-M4) × 2 Node |
| OS | FreeRTOS v10.x |
| 통신 | CAN 2.0B (500 kbps) |

---

### 개발 배경 및 목적

자동차 전장 시스템 및 방산 임베디드 시스템에서 공통적으로 요구되는 **분산 실시간 제어** 아키텍처를 직접 구현하여, RTOS 기반 멀티태스크 설계와 CAN 통신 스택 구축 역량을 확보하는 것을 목표로 함.

산업 현장에서는 다수의 ECU(Electronic Control Unit)가 CAN 버스를 통해 협조하며 실시간 제어를 수행하며, 이 구조는 자동차(AUTOSAR), 군용 차량, 드론 등에 공통 적용됨. 해당 구조를 오픈소스 스택(FreeRTOS + STM32 HAL) 위에서 처음부터 구현함으로써 시스템 SW 전반에 대한 깊은 이해를 추구함.

### 주요 구현 내용

### 1. FreeRTOS 태스크 설계 및 우선순위 체계

- Rate Monotonic Analysis(RMA) 기반으로 태스크 우선순위 산정
- 총 N개 태스크를 period 기반으로 구분하여 CPU 이용률 xx% 이하 유지
- `xTaskNotifyWait` / `xQueueSend` 기반 태스크 간 동기화로 뮤텍스 오버헤드 최소화
- FreeRTOS Stack Overflow Hook 및 Malloc Fail Hook 구현으로 런타임 안전성 확보

### 2. CAN 통신 프로토콜 직접 설계

- 11-bit CAN ID를 `[Priority: 3bit][Node ID: 4bit][Message Type: 4bit]` 구조로 설계
- 메시지 타입: `HEARTBEAT`, `SENSOR_DATA`, `ACTUATOR_CMD`, `FAULT_REPORT` 4종 정의
- CAN 수신 인터럽트 → ISR 내 Queue Put → Rx Task 처리 구조로 IRQ latency 최소화
- CAN 하드웨어 필터(Mask Mode) 설정으로 불필요한 메시지 CPU 처리 제거

### 3. Watchdog 기반 Fault Detection 구현

- STM32 Independent Watchdog(IWDG) + 소프트웨어 Watchdog 2중 구성
- 소프트웨어 Watchdog: Supervisor Task가 각 태스크의 heartbeat 타임스탬프를 주기적 검사
- 태스크 응답 없음(deadline miss) 감지 시 → `FAULT_REPORT` CAN 메시지 전송 → 시스템 안전 상태 전환
- Fault 상태 지속 시 NVIC Reset을 통한 자동 복구 시퀀스 구현

### 4. 실시간성 측정 및 검증

- Logic Analyzer를 사용하여 CAN 메시지 송수신 타이밍 파형 측정
- GPIO 토글 + LA 트리거로 태스크 스케줄링 지터(Jitter) 측정
- `DWT Cycle Counter` 기반 마이크로초 단위 실행 시간 프로파일링 구현
