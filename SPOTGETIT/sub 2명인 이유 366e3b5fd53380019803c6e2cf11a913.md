# sub 2명인 이유

가능 여부부터 정리하면:

토픽이 올라가 있고 내용이 있으면 그 내용을 주기적으로 읽어서 send를 ROS2에서 “그대로 poll”하는 방식은 기본적으로 안 됩니다.

ROS2 topic은 공유 메모리 변수처럼 “현재 값 읽기” API가 있는 구조가 아닙니다. subscriber는 publisher가 sample을 전달할 때 callback을 받습
니다.

TRANSIENT_LOCAL도 마찬가지입니다.

- publisher가 마지막 메시지를 DDS cache에 들고 있음
- subscriber가 붙을 때 그 cached sample을 한 번 전달받음
- 그 다음부터는 새 publish가 있을 때만 callback 호출
- “타이머마다 DDS cache를 다시 읽기” 같은 동작은 일반 subscription API로 안 됨

그래서 pose_path_event_sender가 처음 한 번만 보내던 이유는 이겁니다.

1. global path publisher는 TRANSIENT_LOCAL.
2. pose_path_event_sender가 켜짐.
3. DDS가 cached global path를 한 번 delivery.
4. on_global_path() callback 한 번 실행.
5. 기존 코드는 그 callback 안에서만 UDP send.
6. 이후 새 publish가 없으면 callback 없음.
7. 따라서 send도 없음.

ros2 topic echo --once를 여러 번 치면 매번 읽히는 것처럼 보일 수 있는데, 그건 매번 새로운 subscriber를 만들기 때문입니다. 새 subscriber
가 붙을 때마다 DDS cached sample을 한 번 받는 겁니다. 기존에 붙어 있는 subscriber가 같은 cache를 계속 다시 읽는 게 아닙니다.

가능한 구현은 이 둘 중 하나입니다.

1. pose_path_event_sender가 TRANSIENT_LOCAL로 한 번 받은 global path를 자기 메모리에 저장하고, 타이머로 계속 send한다.
2. 타이머마다 임시 subscriber를 새로 만들어서 transient cache를 다시 받는다.

2번은 구조가 나쁩니다. 비동기 discovery, QoS 매칭, timeout 처리가 필요하고, 매 주기마다 subscriber 생성/삭제라 불안정합니다.

그러니까 현실적으로 맞는 방식은 1번입니다.

다만 네가 말한 조건 중 토픽이 없으면 send 안 함까지 정확히 하려면 추가 조건이 필요합니다.

- 내용이 있으면: 우리가 한 번이라도 TRANSIENT_LOCAL callback으로 받은 latest path가 있으면 됨.
- 토픽이 있으면: 타이머에서 count_publishers("/planning/global_path/spot_01") > 0 같은 식으로 publisher 존재를 확인할 수 있음.

즉 최종적으로 가능한 구조는:

TRANSIENT_LOCAL 구독으로 latest path 수신
→ publisher 있고 latest path 있으면 UDP send
→ publisher 없거나 latest path 없으면 send 안 함
