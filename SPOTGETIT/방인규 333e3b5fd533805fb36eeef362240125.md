## 프로젝트 기안서: 다중 MCU 및 엣지 AI 기반 자율 추적 전차 시스템

- **프로젝트 목적:** 프론트엔드 개발 없이 ROS2 표준 툴을 활용하여 관제 환경을 구성하고, 젯슨(Jetson) 보드와 2개의 다중 MCU 간 고속 통신 및 정밀 모터 제어(포탑/궤도)에 기술적 역량을 집중합니다.
- **시스템 구조 (하드웨어 중심):**
    - **인지 판단 (SBC - Jetson Orin Nano):** 카메라 영상 처리(TensorRT 최적화 YOLO), 타겟 좌표 추출, 원격 조이스틱 신호 수신.
    - **주행 제어 (MCU 1):** 조향 및 구동 궤도 제어, 엔코더 오도메트리(Odometry) 연산, IMU 센서 퓨전.
    - **포탑 제어 (MCU 2):** 상탑 팬/틸트(Pan/Tilt) 서보 모터 초고속 PID 제어, 락온 유지.
    - **내부 통신망:** SBC와 MCU 간은 고속 UART(DMA) / 다중 MCU 간은 CAN(Controller Area Network) 통신망 구축.

### 역할 분담 (순수 하드웨어/로우레벨 중심 6인 체제)

**1. 포탑 정밀 제어 (MCU 1인):**

- 상탑 팬/틸트 모터 제어용 RTOS 펌웨어 작성.
- 타겟의 이동 좌표를 수신하여 모터 각도로 변환(역기구학 연산)하고, 짐벌(Gimbal) 안정화를 위한 PID 제어.

**2. 주행 시스템 제어 (MCU 1인):**

- 무한궤도 구동용 BLDC/DC 모터 드라이버 제어.
- 휠 엔코더와 IMU 센서 데이터를 결합하여 전차의 현재 위치와 자세를 계산하는 센서 퓨전 구현.

**3. 다중 디바이스 통신 및 미들웨어 (1인):**

- MCU 1, MCU 2 간의 CAN 통신망 구축 및 프레임(Frame) 규격 설계.
- SBC와 메인 MCU 간 통신 지연을 최소화하기 위한 DMA 직렬 통신 구조 최적화 및 micro-ROS 이식.

**4. 엣지 AI 및 비전 파이프라인 (SBC 1인):**

- Jetson 환경에서 YOLO 모델을 TensorRT로 경량화 및 가속화하여 FPS(Frame Per Second) 극대화.
- 2D 이미지 픽셀 좌표를 포탑 제어를 위한 3D 물리 좌표계로 변환.

**5. 전사적 시스템 통합 (SBC 1인):**

- 하드웨어 조이스틱(PS4/Xbox 패드 등)의 입력 신호를 ROS2 `cmd_vel` 제어 명령으로 매핑.
- 카메라 원본 영상과 타겟 락온 바운딩 박스를 병합하여 `rqt`로 송출하는 파이프라인 구축.

**6. 하드웨어/회로 및 전원 관리 (1인):**

- 고전류 모터 노이즈가 MCU와 카메라 센서에 미치는 영향을 차단하기 위한 전원 분리 설계 (Power Distribution Board).
- 전차 하드웨어 프레임 조립 및 각 모듈 간의 와이어링, 발열 제어 시스템 구축.

## 1. 전사적 HardFault 분석 및 포렌식 시스템

단순히 "어디서 에러 났니?"를 넘어, 현장에 배포된 기기들이 죽었을 때 그 원인을 원격으로 수집하고 소스코드 단위로 복기하는 **'블랙박스'**를 만드는 것입니다.

- **기술적 깊이:** ARM 코어의 예외 처리 메커니즘을 어셈블리 수준에서 핸들링해야 합니다.
- **시스템 흐름:**
    1. **MCU:** HardFault 발생 즉시 모든 인터럽트를 중지하고, 현재 레지스터와 스택 메모리(최대 수 KB)를 플래시의 특정 섹터에 긴급 저장합니다.
    2. **SBC:** 재부팅 시 MCU로부터 이 '덤프 데이터'를 수신하여 중앙 분석 서버로 전송합니다.
    3. **서버:** 수신된 바이너리 주소를 빌드 시 생성된 ELF 파일의 디버깅 심볼과 대조하여 **"main.c의 145번 줄에서 Null Pointer 참조 발생"**이라는 리포트를 생성합니다.
- **6인 체제 핵심:** 커널 핸들러 팀, 바이너리 파싱 엔진 팀, 웹 대시보드 팀으로 나누어 **풀스택 임베디드 인프라**를 구축합니다.

단순히 에러 라인을 찾는 것을 넘어, 실시간으로 배포된 여러 기기에서 발생하는 크래시를 수집하고 분석하는 **통합 관제 솔루션**으로 확장합니다.

- **팀 구성 (6인):**
    - **MCU/어셈블리 (2명):** RTOS 커널 내부의 Exception 핸들러 구현, 콜 스택 언와인딩(Unwinding) 엔진 개발.
    - **통신/보안 (1명):** 크래시 로그의 암호화 및 유무선(UART/Wi-Fi) 전송 프로토콜 설계.
    - **백엔드/분석 (2명):** `.elf` 파싱 엔진 구축, 바이너리 주소를 소스코드로 변환하는 심볼 매핑 서버 구축.
    - **프론트엔드 (1명):** 대시보드 구현 (기기별 오류 통계, 소스코드 하이라이팅 시각화).
- **볼륨 포인트:** 하드웨어 예외 발생 시점의 메모리 덤프 전체를 분석하여 **"왜 이 에러가 났는가"**를 추적하는 포렌식 기능을 포함합니다.

## 2. 보안 가속 및 롤백을 지원하는 엔터프라이즈 FOTA 시스템

"새 버전 깔기"가 아니라, 전원이 꺼지거나 해킹된 파일이 들어와도 시스템이 **절대로 벽돌이 되지 않게 관리**하는 고신뢰성 업데이트 시스템입니다.

- **기술적 깊이:** 링커 스크립트를 직접 수정하여 메모리 맵을 설계하고, 부트로더가 커널을 교체하는 로우레벨 제어가 핵심입니다.
- **시스템 흐름:**
    1. **서버:** 펌웨어에 디지털 서명(RSA/AES)을 입혀 배포합니다.
    2. **SBC:** 펌웨어를 조각내어 MCU에 전달하며, 전송 중 오류를 체크합니다.
    3. **MCU:** 1번 뱅크에서 기존 앱이 돌아가는 동안 2번 뱅크에 새 앱을 씁니다. 무결성 검증이 끝나면 부트로더가 실행 영역을 스위칭합니다. 만약 새 앱이 부팅에 실패하면 다시 1번으로 자동 복구(Rollback)합니다.
- **6인 체제 핵심:** 부트로더 설계, 암호화 알고리즘 적용, 서버 및 차분(Delta) 업데이트 로직 개발로 나뉩니다.

단순 펌웨어 교체를 넘어, 보안 전송과 버전 관리, 그리고 **실패 시 100% 복구**를 보장하는 인프라를 구축합니다.

- **팀 구성 (6인):**
    - **부트로더/커널 (2명):** 커스텀 부트로더 제작, 듀얼 뱅크 스위칭 로직 및 벡터 테이블 재배치.
    - **보안/암호화 (1명):** 펌웨어 서명 검증(RSA/AES), 전송 구간 보안(TLS) 적용.
    - **SBC/서버 (2명):** 펌웨어 배포 서버(OTA Server) 구축, 기기별 버전 관리 및 차분 업데이트(Delta Update) 로직.
    - **QA/자동화 (1명):** 가상 환경(QEMU) 기반 업데이트 무결성 테스트 자동화.
- **볼륨 포인트:** 펌웨어 업데이트 중 전원이 차단되어도 시스템이 벽돌(Brick)이 되지 않는 **무결성 보장 메커니즘**과 보안 프로토콜을 구현합니다.

## 3. 이기종 시스템 간 'Zero-Latency' 통신 프레임워크

SBC(Linux)와 MCU(RTOS) 사이의 통신 병목을 해결하여, 마치 하나의 칩처럼 움직이게 만드는 **고속 데이터 버스** 구축입니다.

- **기술적 깊이:** CPU를 거치지 않는 DMA 제어와 커스텀 프로토콜 설계를 통해 통신 지연(Latency)을 극단적으로 줄입니다.
- **시스템 흐름:**
    1. **MCU:** UART/SPI 데이터를 DMA 인터럽트로 처리하여 RTOS의 스케줄링 부하를 0에 가깝게 만듭니다.
    2. **프로토콜:** JSON 같은 무거운 방식 대신 Protobuf나 고정 길이 바이너리 패킷을 사용하여 오버헤드를 최소화합니다.
    3. **SBC:** 리눅스 커널의 시리얼 드라이버 설정을 최적화하여 수 밀리초(ms) 단위의 응답성을 확보합니다.
- **6인 체제 핵심:** RTOS 드라이버 최적화 팀, 리눅스 통신 최적화 팀, 프로토콜 설계 및 지연 시간(Jitter) 측정 도구 개발 팀으로 운영됩니다.

단순 모니터링을 넘어, SBC와 MCU 간의 통신 부하를 최소화하고 결정성을 보장하는 **미들웨어 스택**을 직접 만듭니다.

- **팀 구성 (6인):**
    - **MCU/RTOS (2명):** DMA 기반 고속 UART/SPI 드라이버 최적화, RTOS 태스크 우선순위 역전 방지 설계.
    - **SBC 드라이버 (1명):** Linux 커널 레벨의 시리얼 통신 최적화 및 레이턴시 저감 설정.
    - **미들웨어 설계 (2명):** 데이터 직렬화(e.g. Protobuf 기반) 및 micro-ROS 커스텀 트랜스포트 계층 개발.
    - **성능 분석 도구 (1명):** 지터(Jitter) 및 대역폭 점유율 실시간 시각화 툴 개발.
- **볼륨 포인트:** 기존 micro-ROS나 UART 통신보다 **30% 이상의 속도 향상과 낮은 지연**을 달성하는 커스텀 통신 스택을 제안합니다.

## 4. 로봇 제어 검증을 위한 디지털 트윈 기반 HILS 환경

실제 로봇을 만들기 전, 가상 세계의 물리 법칙과 실제 MCU의 펌웨어를 실시간으로 연동하여 **하드웨어를 소프트웨어처럼 테스트**하는 환경입니다.

- **기술적 깊이:** 가상 환경(시뮬레이터)과 실제 MCU 간의 시간 동기화(Synchronization)가 핵심입니다.
- **시스템 흐름:**
    1. **시뮬레이터(PC):** 가상 로봇이 움직이며 가상 거리 센서 값을 생성합니다.
    2. **브릿지(SBC):** 이 데이터를 실제 MCU의 센서 입력 핀이나 통신 포트로 쏴줍니다.
    3. **MCU:** 실제 센서 데이터인 줄 알고 연산하여 모터 제어 신호를 내보냅니다. 이 신호가 다시 시뮬레이터 속 로봇을 움직입니다.
- **6인 체제 핵심:** 로봇 제어 알고리즘 팀, 3D 물리 시뮬레이션 구축 팀, 데이터 동기화 브릿지 및 테스트 자동화 팀으로 구성됩니다.

단순 테스트 스크립트를 넘어, 가상 세계와 물리 세계를 실시간으로 동기화하는 **디지털 트윈 테스트 베드**를 구축합니다.

- **팀 구성 (6인):**
    - **임베디드 제어 (2명):** MCU 단의 PID/LQR 제어 로직 구현 및 RTOS 실시간성 확보.
    - **시뮬레이터 (2명):** Gazebo나 Isaac Sim을 활용한 가상 로봇 물리 환경 구축 및 센서 모사.
    - **브릿지 시스템 (1명):** 시뮬레이션-실제 하드웨어 간 데이터 동기화(Sync) 엔진 개발.
    - **테스트 자동화 (1명):** 수백 가지 시나리오를 자동으로 돌리고 결과를 리포팅하는 CI(Continuous Integration) 서버 구축.
- **볼륨 포인트:** 실제 로봇 없이도 **MCU의 펌웨어가 가상 로봇을 완벽하게 제어**하고, 그 결과가 실제와 동일한지 검증하는 하이엔드 환경을 구축합니다.

### 1. ARM Cortex-M 하드폴트 추적 및 분석 시스템

MCU에서 잘못된 메모리 접근이나 나누기 오류 등이 발생했을 때, 시스템이 멈추는 원인을 소스코드 라인 수준에서 역추적하는 디버깅 도구입니다.

- **시스템 구조:** * 하드폴트 발생 시 예외 처리 인터럽트(Exception Handler)가 호출됩니다.
    - 해당 시점의 CPU 레지스터 상태(R0~R3, R12, LR, PC, xPSR)가 저장된 스택 프레임(Stack Frame)을 추출하여 비휘발성 메모리에 저장하거나 UART로 출력합니다.
    - PC(호스트)의 Python 스크립트가 펌웨어 빌드 시 생성된 `.elf` 또는 `.map` 파일을 파싱하여, 추출된 PC(Program Counter) 레지스터의 메모리 주소를 실제 C/C++ 코드 라인으로 변환합니다.
- **핵심 구현 과제:** * ARM 어셈블리를 활용한 레지스터 덤프 추출 로직 작성.
    - 콜 스택(Call Stack) 역추적을 위한 메모리 맵 구조 이해.
- **요구 역량:** C, ARM Assembly, Python, 컴파일러 및 링커 구조 이해.

로우레벨 디버깅 및 메모리 맵 분석에 집중하는 로드맵입니다.

- **1단계 (환경 구축):** Cortex-M 타겟 보드에 FreeRTOS를 포팅하고 고의적인 하드폴트(예: Null 포인터 참조)를 발생시키는 테스트 펌웨어(C/C++) 작성.
- **2단계 (레지스터 추출):** 하드폴트 예외 처리 핸들러(Exception Handler)를 어셈블리어로 작성하여 예외 발생 시점의 스택 프레임(R0~R3, PC, LR 등) 데이터 추출.
- **3단계 (데이터 전송):** 추출된 레지스터 상태와 콜 스택(Call Stack) 데이터를 UART 인터페이스를 통해 호스트 PC로 전송하는 로직 구현.
- **4단계 (호스트 분석):** PC(Linux 환경 등)에서 Python 스크립트를 작성하여 수신한 PC(Program Counter) 레지스터 값을 `.elf` 파일과 대조하고, 실제 오류가 발생한 소스코드 라인을 매핑 및 출력.

### 2. 듀얼 뱅크 기반 FOTA 커스텀 부트로더

기존 애플리케이션의 동작을 중단하지 않고, 백그라운드에서 새로운 펌웨어를 수신하여 안전하게 업데이트하는 시스템입니다.

- **시스템 구조:** * MCU의 플래시 메모리를 Bootloader, Bank A(현재 실행 영역), Bank B(업데이트 대기 영역)로 파티셔닝합니다.
    - RTOS의 백그라운드 태스크가 UART 등을 통해 새 펌웨어 바이너리를 수신하고 Bank B에 기록합니다.
    - 수신 완료 후 CRC32로 데이터 무결성을 검증하고, 플래시 메모리의 특정 플래그를 변경한 뒤 시스템을 재부팅합니다.
    - 부트로더는 플래그를 확인하여 Bank B의 시작 주소로 인터럽트 벡터 테이블(IVT)을 재배치하고, 새 펌웨어로 점프(Jump)합니다.
- **핵심 구현 과제:** * 링커 스크립트(`.ld`) 수정을 통한 메모리 섹션 분할.
    - 플래시 메모리 읽기/쓰기 제어 및 IVT 오프셋 설정.
- **요구 역량:** C, 플래시 메모리 제어, 부팅 시퀀스 제어.

플래시 메모리 제어와 시스템 부팅 시퀀스 설계에 집중하는 로드맵입니다.

- **1단계 (메모리 파티셔닝):** 링커 스크립트(`.ld`)를 수정하여 MCU의 플래시 메모리를 Bootloader, Bank A(실행 영역), Bank B(업데이트 대기 영역)로 분할.
- **2단계 (수신 및 기록):** 현재 실행 중인 펌웨어(Bank A) 내에 새로운 바이너리 파일을 통신(UART/Wi-Fi)으로 수신하고 Bank B 영역에 기록하는 백그라운드 태스크 구현.
- **3단계 (부트로더 작성):** 시스템 부팅 시 Bank B의 펌웨어 무결성(CRC32)을 검증하고, 정상일 경우 인터럽트 벡터 테이블(IVT)을 이동시켜 Bank B로 점프하는 부트로더 구현.
- **4단계 (통합 테스트):** 펌웨어 업데이트 트리거를 발생시킨 후 재부팅, 펌웨어 교체, 실패 시 롤백(Rollback) 동작 검증.

### 3. 이기종 시스템(SBC-MCU) 통신 지연 모니터링 브릿지

젯슨(Jetson)이나 라즈베리파이(Raspberry Pi)와 같은 상위 제어기(SBC)와 RTOS 기반 하위 제어기(MCU) 간의 직렬 통신 상태를 정량적으로 측정하는 시스템입니다.

- **시스템 구조:** * 상위 제어기(Python/C++)에서 주기적으로 테스트 패킷을 생성하여 하위 제어기로 전송합니다.
    - 하위 제어기는 수신 인터럽트 발생 시 하드웨어 타이머(SysTick 등)의 타임스탬프를 패킷에 기록하고, DMA(Direct Memory Access)를 통해 상위 제어기로 반환(Echo)합니다.
    - 상위 제어기는 반환된 패킷을 분석하여 왕복 지연 시간(RTT), 패킷 유실률, 통신 주기 편차(Jitter)를 계산하고 로그로 기록합니다.
- **핵심 구현 과제:** * 가변 길이를 지원하고 체크섬(Checksum)이 포함된 자체 패킷 프로토콜 설계.
    - RTOS 커널 타이머를 활용한 마이크로초 단위의 타임스탬프 동기화.
- **요구 역량:** C/C++, Python, UART/SPI 통신 프로토콜 설계.

이기종 간의 실시간 통신 신뢰성 검증에 집중하는 로드맵입니다.

- **1단계 (프로토콜 정의):** 가변 길이 페이로드와 체크섬(Checksum)을 포함하는 자체 패킷 구조체 정의.
- **2단계 (하위 제어기 구현):** MCU(FreeRTOS) 측에 하드웨어 타이머를 설정하고, 수신 인터럽트 발생 시 타임스탬프를 기록하여 DMA로 즉각 반환(Echo)하는 펌웨어 작성.
- **3단계 (상위 제어기 구현):** SBC(젯슨 나노/라즈베리파이 등) 측에서 Python 또는 C++를 활용하여 주기적으로 패킷을 송신하고 수신 데이터를 파싱하는 스크립트 작성.
- **4단계 (지연 분석 및 시각화):** SBC에서 수집된 왕복 시간(RTT) 데이터를 바탕으로 패킷 유실률과 통신 주기 편차(Jitter)를 계산하고, 이를 터미널이나 UI 형태로 출력.

### 4. 임베디드 타겟 HILS 시스템

실제 물리적 환경을 구축하기 전에, PC에서 가상의 입력을 생성하여 MCU의 제어 로직을 검증하는 자동화 테스트 환경입니다.

- **시스템 구조:** * 호스트 PC의 Python 테스트 엔진이 사전에 정의된 시나리오에 따라 가상의 센서 데이터(예: 온도 변화, 장애물 거리 등)를 생성합니다.
    - 이 데이터를 통신 인터페이스를 통해 타겟 MCU로 주입합니다.
    - MCU의 RTOS 태스크는 수신된 데이터를 바탕으로 제어 알고리즘(예: PID 제어)을 수행하고, 결과값(모터 PWM 주기 등)을 다시 PC로 전송합니다.
    - PC는 수신된 출력값이 예상 허용 오차 범위 내에 있는지 비교하고 검증 리포트를 생성합니다.
- **핵심 구현 과제:** * PC와 MCU 간의 실시간 입출력 동기화.
    - PyTest 등의 프레임워크를 활용한 테스트 시나리오 스크립트 엔진 구축.
- **요구 역량:** Python(PyTest), C, 제어 공학 기본 지식.

제어 로직의 정량적 검증과 테스트 자동화 인프라 구축에 집중하는 로드맵입니다.

- **1단계 (제어 로직 펌웨어):** 타겟 MCU에 물리적 센서 대신 직렬 통신으로 데이터를 수신하여 PID 제어 등의 연산을 수행하고 결과를 송신하는 펌웨어 구축.
- **2단계 (시나리오 엔진 작성):** PC에서 Python(PyTest 프레임워크 등)을 활용해 특정 경계값이나 에러 상황을 모사한 가상 센서 데이터 생성 스크립트 작성.
- **3단계 (실시간 입출력 동기화):** PC와 MCU 간 통신을 연결하여, 시나리오 스크립트가 데이터를 주입하고 MCU의 제어 출력값을 실시간으로 반환받는 인터페이스 구성.
- **4단계 (자동화 파이프라인 구성):** Docker 컨테이너 등을 활용하여 호스트 PC 환경의 의존성을 분리하고, 테스트 스크립트 실행부터 결과 리포트(Pass/Fail) 생성까지의 과정 자동화.

### 이기종 분산 아키텍처 (SBC + MCU)

이기종 분산 아키텍처(SBC + MCU) 프로젝트의 구체적인 구현 방안과 단계별 핵심 요소입니다.

이 아키텍처는 연산량이 많은 고수준(High-level) 작업과 엄격한 타이밍이 요구되는 저수준(Low-level) 제어를 물리적으로 분리하는 산업계 표준 방식입니다.

### 1. 시스템 구성 (하드웨어 및 소프트웨어)

- **상위 제어기 (SBC):** 라즈베리파이 또는 젯슨 나노. Ubuntu OS와 ROS2를 구동하며, 카메라나 라이다 등 다중 센서 기반의 인지, 경로 계획, 판단 연산을 수행합니다.
- **하위 제어기 (MCU):** STM32, ESP32 등. FreeRTOS를 구동하며, SBC의 명령을 받아 실제 모터를 제어하고, 휠 엔코더 및 IMU 센서 데이터를 실시간으로 수집합니다.
- **통신 인터페이스:** UART, SPI 또는 CAN 통신을 사용하며, MCU에 micro-ROS를 이식하여 SBC의 ROS2 네트워크와 직접 Topic을 송수신합니다.

### 2. 하위 제어기(RTOS) 핵심 구현 과제

- **태스크(Task) 분리 및 우선순위 할당:**
    - `Comm_Task`: SBC와의 통신(micro-ROS 송수신)을 처리합니다.
    - `Control_Task`: 모터 PID 제어를 수행합니다. 정확한 주기가 보장되어야 하므로 높은 우선순위를 부여합니다.
    - `Sensor_Task`: 하드웨어 타이머 인터럽트를 통해 정해진 주기마다 센서 데이터를 수집합니다.
- **자원 동기화 (Mutex):**
    - SBC로부터 수신한 '목표 속도' 변수를 `Comm_Task`가 갱신하고 `Control_Task`가 읽어갈 때, 데이터의 무결성을 보장하고 충돌을 방지하기 위해 Mutex를 적용합니다.
- **비동기 통신 (DMA 활용):**
    - 데이터 송수신 과정에서 MCU의 CPU 자원 대기를 최소화하기 위해 DMA(Direct Memory Access)를 설정합니다. 이를 통해 RTOS 스케줄러의 지연(Latency)을 방지합니다.

### 3. 프로젝트 진행 로드맵

1. **환경 구축:** MCU 타겟 보드에 FreeRTOS 및 micro-ROS 라이브러리 포팅. SBC에 ROS2 및 micro-ROS 에이전트 구축.
2. **통신 검증:** SBC와 MCU 간 하드웨어 인터페이스(UART 등) 연결 및 단순 문자열 전송을 통한 통신 안정성 테스트.
3. **제어 로직 구현:** MCU 내부에 RTOS Task를 독립적으로 생성하여 제어 및 수집 로직 작성. Task 간 안전한 데이터 공유 구조 설계.
4. **통합 연동:** SBC에서 제어 명령(`cmd_vel`)을 발행(Publish)하고, MCU가 이를 수신하여 모터를 구동한 뒤 현재 주행 상태 정보(`Odometry`)를 다시 SBC로 발행하는 폐루프(Closed-loop) 시스템 완성.

### 4. 기대 역량

제한된 하드웨어 자원 환경에서 실시간성을 보장하기 위해 RTOS 커널의 스케줄링 및 동기화 기법을 설계한 경험을 활용할 수 있습니다. 또한, 이기종 디바이스 간의 분산 제어 시스템 구축 경험을 활용할 수 있습니다.
