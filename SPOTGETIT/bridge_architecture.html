Jetson ↔ Raspberry Pi ↔ PC 무선 통신 아키텍처

4족 보행 로봇 관제 시스템 · BridgeDaemon + Qt 앱 + 공유 메모리 (IPC)

1. 시스템 개요

Jetson(로봇 탑재 컴퓨터)이 생성하는 라이다·카메라·odom 데이터를 Raspberry Pi(중계 브리지)가 UDP로 수신한다. RPi 내부에서는 공유 메모리(IPC)를 통해 Qt 관제앱에 전달하고, Qt에서 입력한 커맨드는 역방향으로 Jetson에 전송된다.

핵심 설계 원칙

무선 환경 특성상 EtherCAT급 결정론은 불가능하다. 대신 데이터 특성에 맞는 프로토콜 분리(UDP/TCP)와 공유 메모리 기반 IPC로 지연 최소화를 달성한다.

병목 데이터

라이다(~5 MB/s) · 이미지 raw(~45 MB/s → 압축 필수). odom은 200 B × 50 Hz로 무시 가능. Wi-Fi 5 실질 처리량 ~40 MB/s 이내로 트래픽 설계 필요.

2. 전체 아키텍처

전체 시스템 구성: Jetson ↔ RPi 무선 UDP · RPi 내부 공유 메모리 IPC · RPi ↔ PC 유선/무선 소켓

3. BridgeDaemon 내부 데이터 흐름

BridgeDaemon은 RPi에서 단일 프로세스로 동작하며, 역할별로 분리된 6개 스레드로 구성된다. 스레드 간 데이터는 큐(Queue)를 통해 전달되어 블로킹이 전파되지 않는다.

4. 공유 메모리 (IPC) 설계

4-1. SharedData 구조

모든 앱이 단일 헤더 파일 shm_def.h 을 공유한다. 포인터 저장은 금지 — 프로세스마다 attach 주소가 달라서 역참조 시 세그폴트가 발생한다.

// shm_def.h — 모든 앱이 이 헤더 하나만 include
#pragma once
#include <stdint.h>
#include <pthread.h>
#include <stdatomic.h>

#define SHM_NAME      "/robot_bridge"
#define MAX_IMG_BYTES (640*480*2)   // JPEG 압축 기준
#define MAX_LIDAR_PTS  1024          // 2D: 360 / 3D: decimation 후
#define TRIPLE_SLOTS   3

typedef struct {
    uint8_t  state;            // 슬롯 상태: EMPTY/WRITING/READY/READING
    uint32_t size;             // 실제 데이터 바이트 수
    uint64_t timestamp_us;    // 송신 측 타임스탬프
    uint8_t  data[MAX_IMG_BYTES];
} ImgSlot;

typedef struct {
    // ── 이미지: Triple Buffer ──────────────────────────
    ImgSlot           img_slots[TRIPLE_SLOTS];
    _Atomic int       img_ready_idx;   // 최신 슬롯 인덱스

    // ── 라이다: Double Buffer ─────────────────────────
    float             lidar[2][MAX_LIDAR_PTS][2]; // [slot][point][x,y]
    uint32_t          lidar_count[2];
    _Atomic int       lidar_ready_idx;

    // ── Odom: rwlock + 직접 쓰기 ─────────────────────
    pthread_rwlock_t  odom_lock;
    float             odom[3];            // x, y, theta
    uint64_t          odom_ts_us;

    // ── 메타: 연결 상태 / 통계 ───────────────────────
    pthread_rwlock_t  meta_lock;
    uint8_t           jetson_connected;
    uint32_t          pkt_drop_count;
    float             avg_latency_us;

} SharedData;

Triple Buffer 적용 대상

이미지·라이다처럼 크고 자주 덮어쓰는 데이터. Writer와 Reader가 서로 다른 슬롯을 사용하므로 lock 없이 atomic index swap만으로 동기화 가능.

rwlock 적용 대상

odom·메타처럼 작고 단순한 데이터. 여러 Reader가 동시 읽기 가능, Writer는 배타적 점유. 크기가 작아 memcpy 오버헤드 무시 가능.

4-2. Triple Buffer 슬롯 상태 전이

WRITING 슬롯은 Reader가 절대 접근하지 않음 → lock 불필요

4-3. eventfd 알림 흐름

Qt의 data_consumer가 폴링 없이 새 프레임을 감지하는 방법. read() 호출 시 신호가 올 때까지 스레드가 완전히 슬립한다.

// BridgeDaemon (reassembly_shm) — 새 프레임 완성 시
uint64_t val = 1;
write(efd, &val, sizeof(val));   // Qt 깨우기

// Qt (data_consumer) — 블로킹 대기
uint64_t val;
read(efd, &val, sizeof(val));    // 신호 올 때까지 sleep
int slot = atomic_load(&shm->img_ready_idx);
render_frame(shm->img_slots[slot].data);

5. 스레드 우선순위

RPi에서 SCHED_FIFO 사용 시 root 권한 필요. 우선순위는 숫자가 클수록 높다.

스레드우선순위이유
jetson_tx★★★★★정지 커맨드·heartbeat 지연 불허. 가장 높게.
protocol_timer★★★★★ACK timeout 정확해야 재전송 타이밍이 맞음
jetson_rx★★★★커널 수신 버퍼 overflow 방지
reassembly_shm★★★재조립 지연 → Qt 화면 끊김
pc_link★★★PC 통신은 로봇 제어보다 낮은 우선순위
data_consumer (Qt)★★★UI 프레임 부드럽게 유지
main / watchdog★★초기화 · 재시작 정책만 담당
logger (큐 drain)★실시간 경로 절대 방해 금지
 
// 스레드 생성 시 우선순위 지정 예시
pthread_attr_t attr;
struct sched_param param;
pthread_attr_init(&attr);
pthread_attr_setschedpolicy(&attr, SCHED_FIFO);
param.sched_priority = 80;             // jetson_tx
pthread_attr_setschedparam(&attr, ¶m);
pthread_create(&tx_tid, &attr, jetson_tx_func, NULL);

6. 핵심 개념

Triple Buffer

Writer와 Reader가 서로 다른 슬롯을 쓰기 때문에 mutex 없이 동작한다. Write 완료 시 atomic_store(&ready_idx, slot) 하나로 최신 슬롯을 공개한다. Reader는 atomic_load 로 최신 슬롯을 확인하고 읽는다.

적용: SHM-IMG (이미지), SHM-MAP (라이다) — 크고 자주 갱신되는 데이터

rwlock (Reader-Writer Lock)

여러 Reader 동시 읽기 허용, Writer는 배타적 점유. pthread_rwlockattr_setpshared(PTHREAD_PROCESS_SHARED) 로 프로세스 간 공유 가능. Lock 구간은 memcpy만 포함할 정도로 최대한 짧게 유지한다.

적용: odom, SHM-META — 작고 단순한 데이터

eventfd

폴링 없이 프로세스 간 신호를 전달하는 Linux 파일 디스크립터. Bridge가 write(efd, 1) 하면 Qt의 read(efd) 블로킹이 풀린다. epoll과 결합해 여러 이벤트를 하나의 루프에서 처리할 수도 있다.

사용처: reassembly_shm → data_consumer 프레임 완성 알림

OSAL (OS Abstraction Layer)

시스템 콜을 래퍼 함수로 감싸 에러 처리·로그를 중앙화한다. sys_shm_open(), sys_create_thread() 등 내부 구현이 바뀌어도 애플리케이션 코드는 영향을 받지 않는다.

효과: 코드 품질 향상 · 디버깅 용이 · 이식성 확보

9. 구현 로드맵

1

shm_def.h 설계

SharedData 구조체 정의 · 슬롯 상태 enum · MAX 크기 상수. 모든 앱이 이 헤더 하나만 include — 크기 불일치 버그 원천 차단.

2

BridgeDaemon SHM 초기화

shm_open + mmap + rwlock/atomic 초기화 래퍼(OSAL). Bridge가 SHM을 먼저 생성, Qt는 나중에 attach (재시도 루프 필요).

3

eventfd 생성 및 알림 흐름

eventfd(0, EFD_SEMAPHORE) 생성 · reassembly_shm write 완료 시 notify. Qt data_consumer는 read(efd) 블로킹으로 대기.

4

reassembly_shm Triple Buffer write 로직

빈 슬롯 선택 → WRITING 마킹 → memcpy → atomic_store(ready_idx) → eventfd notify. 3D 라이다면 zero-copy(직접 접근) 고려.

5

Qt data_consumer SHM read + UI emit

eventfd 대기 → atomic_load(ready_idx) → memcpy → Qt signal emit. 네트워크 스레드와 UI 스레드 분리 필수 (프리징 방지).

10. 주요 설계 결정 근거

왜 EtherCAT 대신 UDP인가?

EtherCAT은 IEEE 802.3 유선 전용 프로토콜로, CSMA/CA 기반 Wi-Fi에서는 나노초 결정론 불가. UDP + 고정 주기 타이머 송신 + 우선순위 큐로 수 ms 수준 안정성을 달성한다.

왜 gRPC가 아닌 소켓인가?

gRPC는 소켓 위에 HTTP/2 + TLS + Protobuf 직렬화 레이어를 추가한 것. RPi ↔ Qt는 같은 기기 IPC이므로 소켓(또는 Unix Domain Socket)이 최적.

왜 Lock-free Queue가 아닌 POSIX MQ인가?

커맨드는 초당 10~20회 수준. Lock-free가 빛을 발하는 것은 초당 수십만 건. 지금 규모에선 POSIX Message Queue로 충분하고 구현 난이도도 낮다.

왜 중개 앱 없이 직접 공유 메모리인가?

앱이 3개(Receiver, Qt, Sender) 수준이면 중개 앱은 오버엔지니어링. 직접 공유 + pthread_rwlock/atomic으로 충분하며 프로세스를 하나 줄일 수 있다.
